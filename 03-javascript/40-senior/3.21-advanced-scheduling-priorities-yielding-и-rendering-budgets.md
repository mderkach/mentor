# 3.21 Advanced scheduling: priorities, yielding и rendering budgets

На middle-уровне достаточно понимать, что есть Event Loop, microtasks, timers и `requestAnimationFrame`. На senior-уровне этого уже мало.

Приходится отвечать на более жёсткие вопросы:

- что должно случиться прямо сейчас;
- что может подождать до кадра;
- что может подождать до idle-периода;
- что надо разбить на куски;
- что вообще нельзя оставлять на main thread.

Этот урок про то, как распоряжаться временем браузера, а не просто знать названия API.

## 🎯 Что ты освоишь

- различать `queueMicrotask`, `requestAnimationFrame`, `requestIdleCallback`, `scheduler.postTask` и `scheduler.yield`;
- выбирать между scheduling, алгоритмической оптимизацией и worker-выносом;
- разбивать длинную работу так, чтобы не убивать responsiveness;
- думать budget-ами кадра и interaction path, а не только функциями и очередями.

---

## Зачем это нужно на практике

- Чтобы UI был отзывчивым под реальной нагрузкой, а не только на пустом демо.
- Чтобы не лечить любой лаг через `debounce` или `setTimeout`.
- Чтобы фоновая работа не перетирала input и render path.
- Чтобы production-performance обсуждался как архитектура времени, а не как случайный набор микрофиксов.

---

## Главный принцип: не все проблемы решаются scheduling-ом

Когда код тормозит, у тебя обычно есть пять вариантов:

1. Вообще уменьшить объём работы.
1. Выбрать лучшую структуру данных или алгоритм.
1. Перенести работу в другое время на main thread.
1. Разбить работу на куски и уступать управление браузеру.
1. Вынести работу с main thread в worker.

Scheduling — это только пункт 3 и 4.

Если сортировка на 100000 элементов тяжёлая сама по себе, `requestAnimationFrame` не сделает её дешёвой. Он лишь поменяет момент боли.

---

## Есть три разных вопроса, которые нельзя смешивать

### 1. Нужно ли выполнить работу как можно скорее?

Тогда это не idle и не background.

### 2. Нужна ли работа к ближайшему кадру?

Тогда это уже разговор про `requestAnimationFrame`.

### 3. Можно ли уступить браузеру и продолжить позже?

Тогда в ход идут chunking, yielding и более низкие приоритеты.

Большинство performance-багов возникают, когда на эти три вопроса отвечают одним инструментом.

---

## `queueMicrotask` и `Promise.then`: для tiny sync coordination, не для chunking

Да, ими можно "отложить" работу. Но это плохая идея для длинных серий тяжёлых шагов.

Почему:

- microtasks вычищаются до конца перед следующей обычной задачей;
- рендер и другие события могут ждать;
- бесконтрольная цепочка microtasks легко делает starvation.

Нормальное применение:

- маленькая синхронизация состояния;
- завершение API-контракта в том же тике;
- согласование поведения между sync и async-ветками.

Плохое применение:

- разбивать тяжёлую работу на сотни кусков через `queueMicrotask`;
- пытаться так "оптимизировать" рендер.

---

## `requestAnimationFrame`: когда задача должна попасть в ближайший кадр

Используй `requestAnimationFrame`, когда результат работы влияет на то, что пользователь увидит в следующем кадре:

- синхронизация визуального положения;
- JS-анимация;
- координация DOM-записи перед paint.

Не используй его как "универсальный defer".

Типичная ошибка:

- засунуть в `requestAnimationFrame` тяжёлые вычисления;
- получить поздний кадр и думать, что API виноват.

`requestAnimationFrame` хорошо выбирает момент. Но стоимость работы внутри callback остаётся твоей проблемой.

---

## `requestIdleCallback`: только для low-priority main-thread housekeeping

MDN подчёркивает две важные вещи:

- API limited availability;
- для обязательной работы нужен `timeout`, иначе callback может ждать очень долго.

Это хороший инструмент для:

- фоновой аналитики;
- постепенной инициализации второстепенных частей;
- низкоприоритетного прогрева кеша;
- неблокирующей housekeeping-логики.

Это плохой инструмент для:

- input response;
- обязательного UI-state transition;
- логики, которая должна точно случиться к кадру.

---

## `scheduler.postTask`: когда нужно приоритизировать задачи, а не просто откладывать их

MDN описывает Prioritized Task Scheduling API как способ ставить задачи с грубыми приоритетами:

- `user-blocking`
- `user-visible`
- `background`

Инженерная польза здесь не в "ещё одном API", а в том, что код наконец-то может явно сказать runtime, насколько срочна работа.

Примерный смысл:

```js
if ('scheduler' in globalThis) {
  scheduler.postTask(renderSecondaryPanel, { priority: 'user-visible' });
  scheduler.postTask(sendAnalytics, { priority: 'background' });
}
```

Плюсы:

- приоритеты становятся явными;
- можно абортить или управлять задачами через сигнал;
- API хорошо ложится на фреймворк- и platform-level orchestration.

Ограничение:

- это limited availability API, поэтому нельзя делать на нём безусловную основу поведения без feature-check и fallback.

---

## `scheduler.yield`: не "отложить всё", а вежливо уступить main thread

`scheduler.yield()` полезен, когда длинная работа уже началась, но ты хочешь отдать браузеру шанс обработать более важные вещи и продолжить потом.

Смысл:

```js
async function processLargeBatch(items) {
  for (let index = 0; index < items.length; index += 1) {
    processOne(items[index]);

    if (index % 100 === 0 && 'scheduler' in globalThis) {
      await scheduler.yield();
    }
  }
}
```

Это полезнее, чем бесконечно лепить `setTimeout(..., 0)` по всей функции, потому что намерение читается явно: "уступи поток и продолжим".

Но снова: если сама работа не подходит для main thread, yielding не делает её бесплатной.

---

## Когда нужен scheduling, а когда нужен worker

### Scheduling на main thread подходит, если:

- работа всё ещё тесно связана с DOM или текущим UI-состоянием;
- её можно разбить на маленькие куски;
- полная стоимость работы не чудовищна;
- важен fine-grained контроль над моментом выполнения.

### Worker нужен, если:

- работа CPU-bound и тяжёлая;
- её можно оторвать от DOM;
- главное — убрать вычисления с interaction path;
- задержка между шагами уже не помогает.

Если ты chunk-ишь тяжёлую JSON-нормализацию, парсинг или вычисление layout-hints на main thread, иногда это просто знак, что пора в worker.

---

## Кейс: фильтрация каталога на 50 000 товаров

Симптомы:

- input начинает лагать после 3-4 символов;
- список обновляется "рывком";
- попытка обернуть фильтрацию в `requestAnimationFrame` чуть меняет ощущение, но не чинит проблему;
- `queueMicrotask` только усиливает ощущение зависания.

Нормальный разбор:

1. Уменьшить объём работы: индекс, префильтрация, debounce.
1. Если работы всё ещё много, разбить на chunks.
1. Если chunks всё равно мешают interaction path, уступать поток между частями.
1. Если задача действительно CPU-heavy и независима от DOM, переносить в worker.

Это и есть senior-мышление про scheduling: сначала понять природу боли, потом выбирать инструмент.

---

## Что нужно знать

- различие между микрокоординацией, задачами кадра, idle-работой и приоритизированными задачами;
- ограничения `requestIdleCallback` и `scheduler.*`;
- что yielding и chunking не заменяют нормальную оптимизацию;
- когда main-thread orchestration пора заканчивать и выносить работу в worker.

---

## Что нужно уметь

- выбрать инструмент под конкретный класс работы;
- разбить длинную задачу на осмысленные куски;
- формулировать priority-level языком продукта и UX, а не только API;
- объяснить, почему проблема решается scheduling-ом, алгоритмом или worker-выносом.

---

## Частые ошибки

- Пытаются любую тяжёлую работу "починить" через `requestAnimationFrame`.
- Используют `queueMicrotask` для chunking и получают starvation.
- Считают `requestIdleCallback` подходящим местом для важной логики.
- Используют `scheduler.*` без feature-check и fallback.
- Забивают main thread второстепенной работой, пока interaction path уже горит.

---

## Anti-pattern

"Если мы разложили всё по очередям, значит performance под контролем".

Нет. Под контролем только момент запуска. Цена самой работы никуда не делась.

---

## 🔧 Практика

1. Возьми реальный тяжёлый interaction flow в проекте.
1. Раздели всю работу на четыре класса: urgent, frame-bound, user-visible secondary, background.
1. Для каждого класса выбери механизм: sync, `requestAnimationFrame`, `requestIdleCallback`, `scheduler.postTask`, `scheduler.yield`, worker.
1. Для одного длинного цикла опиши, почему здесь нужен yielding, а не просто `setTimeout`.
