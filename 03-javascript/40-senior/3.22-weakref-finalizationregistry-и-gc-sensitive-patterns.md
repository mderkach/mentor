# 3.22 WeakRef, FinalizationRegistry и GC-sensitive patterns

Это один из тех уроков, где главное не "научиться применять API везде", а научиться очень редко и осознанно их применять.

`WeakRef` и `FinalizationRegistry` существуют не потому, что JavaScript не хватало ещё пары объектов. Они существуют для узких сценариев, где:

- хочется не удерживать тяжёлый объект сильной ссылкой;
- полезно получить шанс почистить хвосты после сборки мусора;
- при этом нельзя строить на этом критичную бизнес-логику.

Если после урока ты начнёшь использовать эти API в каждом кеше, урок провалился.

- память, ссылки и утечки из [3.15 Память, ссылки, утечки и профилирование](../30-middle/3.15-память-ссылки-утечки-и-профилирование.md).

---

## 🎯 Что ты освоишь

- зачем существуют `WeakRef` и `FinalizationRegistry` и почему это не everyday API;
- почему GC-sensitive код нельзя делать частью essential program logic;
- где weak-reference patterns реально окупаются;
- как не превратить управление памятью в магическое мышление.

---

## Зачем это нужно на практике

- Чтобы не держать в памяти крупные объекты только из-за кеша-обёртки.
- Чтобы понимать ограничения GC-semantics библиотек и runtime-инфраструктуры.
- Чтобы не путать weak-reference patterns с нормальным lifecycle management.
- Чтобы не писать код, который работает "пока GC ведёт себя удобно".

---

## Сначала главное предупреждение

MDN формулирует это очень прямо:

- `WeakRef` лучше избегать, если можно;
- `FinalizationRegistry` не должен использоваться для essential program logic.

Причина простая:

- garbage collector работает недетерминированно;
- ты не контролируешь, когда объект будет собран;
- ты не контролируешь, когда cleanup callback будет вызван;
- иногда он может не понадобиться вообще до конца жизненного цикла страницы.

Если логика приложения требует "после удаления объекта обязательно немедленно сделать X", это не задача для GC-aware API.

---

## `WeakRef`: слабая ссылка без гарантии жизни объекта

`WeakRef` позволяет держать ссылку на объект так, чтобы сама эта ссылка не мешала сборке мусора.

```js
const ref = new WeakRef(expensiveObject);

const value = ref.deref();
if (value) {
  useValue(value);
}
```

Ключевая идея:

- объект может быть жив;
- а может уже быть собран;
- `deref()` может вернуть объект, а может `undefined`.

Это делает `WeakRef` полезным для отдельных видов кешей, но плохим для обычной доменной логики.

---

## Когда `WeakRef` может быть оправдан

### 1. Кеш больших производных объектов

Когда:

- восстановить объект можно повторно;
- объект тяжёлый;
- держать его сильной ссылкой всё время невыгодно;
- потеря кеша не ломает поведение, а только приводит к повторному вычислению.

### 2. Инфраструктурные слои и адаптеры

Например, слой, который хочет помнить крупные вычисленные артефакты, но не обязан гарантировать их существование.

### 3. Особые сценарии с DOM-объектами или view-моделями

Только если ты очень хорошо понимаешь lifetime и умеешь жить с тем, что объект может исчезнуть в любой момент, когда нет сильных ссылок.

---

## Когда `WeakRef` обычно не нужен

- обычный продуктовый state;
- сущности домена;
- данные, отсутствие которых ломает flow;
- логика "если объект исчез, надо гарантированно выполнить обязательное действие".

Если объект действительно важен для поведения, его надо держать нормальной сильной ссылкой и управлять lifecycle явно.

---

## `FinalizationRegistry`: callback после GC, но без гарантий по времени

`FinalizationRegistry` позволяет попросить runtime вызвать cleanup callback после того, как зарегистрированное значение будет собрано garbage collector-ом.

```js
const registry = new FinalizationRegistry((key) => {
  cache.delete(key);
});
```

На бумаге выглядит прекрасно. В голове нужно сразу держать ограничения:

- callback не обязан вызываться быстро;
- callback не привязан к бизнес-такту приложения;
- его нельзя использовать как замену `dispose`, `cleanup`, `unsubscribe`, `abort`.

Он подходит только как вспомогательная уборка вокруг не-критичного инфраструктурного хвоста.

---

## Почему `FinalizationRegistry` не заменяет явный cleanup

Представь подписку на событие, сокет, таймер или DOM-listener.

Если ты надеешься, что cleanup случится через GC callback, у тебя сразу несколько проблем:

- момент cleanup недетерминирован;
- ресурсы могут жить дольше, чем ты хочешь;
- поведение будет зависеть от давления памяти и решений движка;
- тестировать это поведение крайне неприятно.

То, что требует deterministic cleanup, должно жить на явных контрактах:

- `dispose()`;
- `unsubscribe()`;
- `abort()`;
- `finally`;
- owner-controlled lifecycle.

---

## Кейс, где weak-reference pattern может быть ок

Допустим, у тебя есть кеш `url -> previewBlob`.

Требования:

- если preview уже есть и ещё жив, отлично;
- если GC его выкинул, можно пересчитать;
- отсутствие preview не ломает систему.

Тогда weak-reference cache может быть разумным:

```js
const cache = new Map();

async function getPreview(url) {
  const cached = cache.get(url)?.deref();
  if (cached) {
    return cached;
  }

  const preview = await buildPreview(url);
  cache.set(url, new WeakRef(preview));
  return preview;
}
```

Если хочется подчищать пустые хвосты в `Map`, можно добавить `FinalizationRegistry`, но только как best-effort cleanup, а не как бизнес-контракт.

---

## Кейс, где weak-reference pattern вреден

Допустим, есть объект с активной подпиской на stream, и ты думаешь:

- "не будем явно отписываться";
- "пусть GC когда-нибудь соберёт объект";
- "а `FinalizationRegistry` потом почистит хвосты".

Это уже архитектурный анти-паттерн.

Почему:

- жизненный цикл ресурса должен быть понятен бизнес-логике;
- stream, listener, timer, socket и subscription нельзя отдавать на волю GC;
- так рождаются призрачные подписки, скрытые утечки и flaky bugs.

---

## `WeakMap` / `WeakSet` vs `WeakRef`: не путай разные семейства weak-паттернов

`WeakMap` и `WeakSet`:

- держат weak relationship вокруг object-key semantics;
- хороши для metadata на объекте;
- не дают прямого "достань объект, если он ещё жив".

`WeakRef`:

- даёт прямую слабую ссылку на объект;
- позволяет попытаться получить объект через `deref()`;
- требует более осторожной работы, потому что ты буквально имеешь дело с "может быть, жив, а может, уже нет".

То есть это не взаимозаменяемые инструменты.

---

## Что нужно знать

- чем weak-reference pattern отличается от обычного lifecycle management;
- ограничения `WeakRef` и `FinalizationRegistry`;
- почему GC не является надёжным расписанием для бизнес-логики;
- где weak-reference cache допустим, а где он превращается в hazard.

---

## Что нужно уметь

- оценивать, восстановим ли объект и безопасно ли его потерять;
- не строить обязательное поведение на `deref()` и cleanup callbacks;
- выбирать explicit cleanup вместо магии GC;
- объяснять команде, почему weak-reference API здесь неуместен.

---

## Частые ошибки

- Используют `WeakRef` как "умный универсальный кеш".
- Считают `FinalizationRegistry` заменой явному cleanup.
- Строят обязательную бизнес-логику на факте срабатывания GC callback.
- Не различают `WeakMap` и `WeakRef`.
- Начинают "оптимизировать память" без измерения реальной проблемы.

---

## Anti-pattern

"Раз движок сам умеет собирать мусор, значит можно привязать к этому и cleanup ресурсов".

GC умеет освобождать память. Это не значит, что он является хорошим оркестратором жизненного цикла приложения.

---

## 🔧 Практика

1. Возьми один сценарий кеша и ответь, безопасна ли потеря значения.
1. Если нет, запрети себе `WeakRef` и опиши явный lifecycle.
1. Если да, опиши weak-reference вариант и отдельно распиши, что произойдёт при `deref() === undefined`.
1. Для одного существующего cleanup-сценария в проекте объясни, почему `FinalizationRegistry` здесь был бы плохой заменой.
