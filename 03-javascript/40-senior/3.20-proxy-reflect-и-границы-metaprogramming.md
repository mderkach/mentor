# 3.20 Proxy, Reflect и границы metaprogramming

`Proxy` почти всегда производит сильное первое впечатление. Кажется, что вот он, универсальный инструмент:

- логировать все обращения;
- валидировать все записи;
- делать "умные" объекты;
- прятать инфраструктуру за красивым API;
- реализовать реактивность, ленивую загрузку и ещё полмира.

Проблема в том, что `Proxy` легко создаёт магию быстрее, чем создаёт ценность. Этот урок не про "редкую фичу языка", а про то, где metaprogramming реально оправдано, а где лучше остаться на стороне явного API.

## 🎯 Что ты освоишь

- что именно перехватывает `Proxy` и зачем рядом нужен `Reflect`;
- где `Proxy` оправдан, а где создаёт вредную runtime-магию;
- что такое invariants и почему их нельзя безнаказанно ломать;
- почему explicit API часто инженерно лучше, чем "умный" объект.

---

## Зачем это нужно на практике

- Чтобы не тащить proxy-механику в обычный продуктовый код без реальной причины.
- Чтобы понимать, почему библиотеки реактивности и state-management иногда опираются на `Proxy`.
- Чтобы не получить "магический" объект, который трудно дебажить, тестировать и сериализовать.
- Чтобы не ломать базовые семантики языка при перехвате `get`, `set`, `has`, `ownKeys` и других операций.

---

## Термины по-человечески

- `Proxy` — объект-обёртка, который может перехватывать фундаментальные операции над target-объектом.
- `Trap` — функция в handler-е, которая перехватывает конкретную операцию.
- `Reflect` — набор методов, который позволяет выполнить стандартное поведение объекта явно и предсказуемо.
- `Invariant` — свойство семантики, которое proxy не имеет права нарушать.

---

## Что именно делает `Proxy`

`Proxy` не "добавляет пару хуков". Он умеет вмешиваться в фундаментальные операции языка:

- чтение свойства;
- запись свойства;
- проверку через `in`;
- перечисление ключей;
- вызов функции;
- создание через `new`;
- работу с descriptors и prototype.

Простой пример:

```js
const user = { name: 'Nina' };

const proxy = new Proxy(user, {
  get(target, key, receiver) {
    console.log('read', key);
    return Reflect.get(target, key, receiver);
  },
});

console.log(proxy.name);
```

Важно: proxy не заменяет target как "новый тип данных". Он виртуализирует существующий объект.

---

## Почему `Reflect` почти всегда идёт рядом с `Proxy`

Очень частая ошибка новичка в proxy-коде:

```js
const proxy = new Proxy(user, {
  get(target, key) {
    console.log(key);
    return target[key];
  },
});
```

Иногда это работает. Но инженерно правильнее чаще писать через `Reflect`:

```js
const proxy = new Proxy(user, {
  get(target, key, receiver) {
    console.log(key);
    return Reflect.get(target, key, receiver);
  },
});
```

Почему:

- `Reflect` даёт стандартную семантику для той операции, которую ты перехватываешь;
- проще не сломать `receiver`, prototype behavior и другие нюансы;
- код честно читается как "перехватили, что-то добавили, затем делегировали обычное поведение".

Практическое правило:

- `Proxy` перехватывает;
- `Reflect` помогает либо безопасно продолжить нормальное поведение, либо явно проверить результат операции.

---

## `Proxy` vs getters/setters vs explicit API

Вот где чаще всего появляется лишняя магия.

### `Getter/setter`

Подходит, когда ты контролируешь конкретное свойство и поведение локально:

```js
class User {
  set name(value) {
    if (!value) {
      throw new Error('name is required');
    }

    this._name = value;
  }
}
```

### Explicit API

Подходит, когда поведение должно быть явным и доменно понятным:

```js
settings.updateTheme(nextTheme);
```

### `Proxy`

Подходит только там, где value именно в перехвате фундаментальных операций:

- библиотека реактивности;
- виртуализация доступа;
- cross-cutting instrumentation;
- debug/devtool layer;
- адаптер, который должен выглядеть как обычный объект.

Если задача описывается как "просто хотим валидировать запись", explicit API или setter часто лучше.

---

## Где `Proxy` реально оправдан

### 1. Реактивность и dependency tracking

Когда система должна автоматически замечать чтения и записи свойств.

### 2. Инструментирование и debug wrappers

Например, временно логировать обращения к нестабильному API в dev-среде.

### 3. Виртуальные адаптеры

Когда нужно представить удалённый или лениво вычисляемый ресурс как объект с привычным интерфейсом.

### 4. Защитные adapter layer

Когда legacy-объект нужно ограничить, санитизировать или стабилизировать для нового слоя.

Во всех этих сценариях proxy решает именно runtime-задачу. Если этой runtime-задачи нет, `Proxy` почти всегда перебор.

---

## Где `Proxy` обычно вреден

- domain-модели и обычные product-объекты;
- публичные API, где важна прозрачность поведения;
- validation, которую можно сделать явной функцией;
- состояние, которое уже и так тяжело читать в коде;
- код, где критична простота дебага и предсказуемость stack trace.

Если после внедрения proxy поведение стало "удобнее писать, но тяжелее понять", это плохой обмен.

---

## Invariants: почему proxy не всесилен

MDN отдельно подчёркивает: `Proxy` обязан соблюдать invariants target-объекта. Если trap нарушает эти ограничения, движок бросит `TypeError`.

Это особенно важно, когда ты работаешь с:

- non-configurable properties;
- non-extensible objects;
- descriptor semantics;
- prototype operations;
- `has`, `ownKeys`, `getOwnPropertyDescriptor`, `defineProperty`.

Инженерный смысл простой:

- proxy может виртуализировать поведение;
- proxy не может безнаказанно переписать базовые guarantees языка.

Именно поэтому "сейчас просто скроем свойство через trap" иногда заканчивается исключением и неожиданным поведением.

---

## Кейс: "спрячем приватные поля через proxy"

Популярная идея:

```js
const proxy = new Proxy(target, {
  get(obj, key, receiver) {
    if (String(key).startsWith('_')) {
      throw new Error('private');
    }

    return Reflect.get(obj, key, receiver);
  },
});
```

На демо это выглядит круто. На масштабе быстро всплывают вопросы:

- как это сочетается с `Object.keys`, `in`, `ownKeys`, descriptors;
- что увидит debugger;
- как это сериализуется;
- как это тестировать;
- почему не использовать нормальный module boundary или `#private` поля класса.

Здесь proxy не добавляет ясности. Он маскирует проблему выбора неподходящей модели инкапсуляции.

---

## `Proxy` и поддерживаемость: цена магии

Даже хороший proxy-код дороже в поддержке, чем явный код:

- сложнее читать control flow;
- тяжелее понять, откуда приходит поведение;
- инструменты и дебаг часто становятся менее прозрачными;
- тесты должны проверять не только данные, но и семантику перехвата;
- сериализация, equality и iteration могут вести себя иначе, чем ожидает разработчик.

Senior-подход здесь не "уметь написать proxy". Senior-подход — уметь не писать его без причины.

---

## Что нужно знать

- какие операции может перехватывать `Proxy`;
- зачем нужен `Reflect` и почему он снижает риск поломки semantics;
- что такое invariants;
- где proxy оправдан как инфраструктурный инструмент, а где нет.

---

## Что нужно уметь

- отличать metaprogramming-задачу от обычной доменной логики;
- использовать `Reflect` для делегирования стандартного поведения;
- видеть, когда proxy делает API менее прозрачным;
- выбирать explicit API вместо proxy там, где важнее читаемость и поддерживаемость.

---

## Частые ошибки

- Используют `Proxy` ради "красивой магии", а не ради конкретной runtime-задачи.
- Пишут traps без `Reflect` и ломают стандартное поведение.
- Забывают про invariants и получают `TypeError` в неожиданных местах.
- Маскируют через proxy плохо спроектированный API.
- Тащат proxy в production-domain там, где явная функция или класс были бы лучше.

---

## Anti-pattern

"Раз можно перехватить всё, значит надо спрятать всю сложность за `Proxy`".

На практике это часто превращается в код, который легко писать автору и тяжело сопровождать команде.

---

## 🔧 Практика

1. Возьми один сценарий с логированием доступа к объекту и реализуй его через `Proxy` + `Reflect`.
1. Для того же сценария опиши альтернативу через explicit API.
1. Сравни по трём критериям: читаемость, дебаг, риск скрытой магии.
1. Для одного существующего "умного" объекта в проекте ответь, действительно ли ему нужен `Proxy`.
