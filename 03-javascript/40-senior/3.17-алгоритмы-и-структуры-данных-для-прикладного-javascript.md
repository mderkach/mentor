# 3.17 Алгоритмы и структуры данных для прикладного JavaScript

Это не «алгоритмы ради собеседований». На senior-уровне важно видеть, где фронтенд-код внезапно становится алгоритмически дорогим: поиск, агрегация, кеширование, tree traversal, нормализация данных и выбор структуры под паттерн доступа.

Когда этого взгляда нет, команда пишет визуально аккуратный код, который на росте данных начинает жечь CPU, блокировать main thread и ломать продуктовые сценарии.

## 🎯 Что ты освоишь

- `Big O` без академического шума и ложного культа;

- как отличать раннюю оптимизацию от ситуации, где массив уже перестал быть нормальным default.

---

## Зачем это нужно на практике

- Чтобы не строить поиск, фильтрацию и агрегацию как случайный набор `map/filter/find/reduce` в каждом рендере.
- Чтобы понимать цену больших таблиц, permission trees, каталогов, routing maps, редакторов и графов зависимостей.
- Чтобы не лечить CPU-проблемы мемоизацией там, где нужно поменять структуру данных.
- Чтобы обсуждать performance языка решений, а не только "здесь что-то тормозит".

---

## Что важно понять

- На фронтенде тоже есть реальные алгоритмические bottleneck'и: search, filtering, lookup, tree traversal, dedupe, joins по коллекциям, diff-like transformations.
- Senior-уровень здесь не в том, чтобы помнить учебник по структурам данных, а в том, чтобы видеть стоимость операций и компромисс между скоростью доступа, обновления и памяти.
- `Big O` полезен, когда помогает обсуждать рост стоимости. Он вреден, когда им пытаются объяснить всё, игнорируя константы, частоту вызова и реальный размер входных данных.

---

## Термины по-человечески

- `Big O` — как быстро растёт стоимость операции при увеличении входных данных.
- `Lookup` — доступ к данным по ключу или идентификатору.
- `Hash map` — структура для быстрого доступа по ключу.
- `Tree` — иерархическая структура данных.
- `Graph` — структура, где важны связи между сущностями, а не только их порядок.

---

## Выбирай структуру данных по паттерну доступа

Самый полезный вопрос здесь не "какая структура моднее", а "что мы делаем с данными чаще всего":

- читаем по `id`;
- фильтруем по нескольким признакам;
- часто добавляем/удаляем;
- строим иерархию;
- проверяем принадлежность к множеству;
- объединяем несколько коллекций;
- переиспользуем результат между interaction-ами.

Если операция "найти по `id`" встречается часто, массив почти всегда хуже, чем `Map`.

```js
// плохо: lookup идёт линейно каждый раз
const selected = items.find((item) => item.id === selectedId);

// лучше: один раз построили индекс, дальше читаем по ключу
const itemsById = new Map(items.map((item) => [item.id, item]));
const selectedFast = itemsById.get(selectedId);
```

Но и `Map` не универсальное добро. Если коллекция маленькая, lookup редкий, а код становится сложнее ради гипотетической победы, массив остаётся нормальным выбором.

---

## Частые hotspot-ы в реальном frontend-коде

### 1. Lookup в больших коллекциях

Каталоги, списки пользователей, permissions, feature flags, конфигурации.

### 2. Дедупликация и membership checks

Когда нужно быстро понять "видели ли мы это значение раньше", `Set` обычно честнее бесконечных `includes`.

### 3. Tree traversal

Навигация, комментарии, permission trees, файловые структуры, layout builders.

### 4. Join-подобные операции

Связка данных из нескольких массивов по ключу. Если делаешь вложенные `find`/`filter`, вероятно, уже нужен индекс.

### 5. Repeated derived data

Один и тот же expensive transformation пересчитывается в нескольких местах без нормального ownership.

---

## Tree и graph нужны не только "сложным приложениям"

Фронтенд часто работает с неявными графами:

- роуты и переходы;
- зависимости полей формы;
- граф разрешений;
- редакторы с вложенными блоками;
- dependency maps в tooling UI.

Если команда упорно пытается держать всё линейным массивом, потом появляются:

- странные recursive хелперы в пяти местах;
- повторные обходы одной и той же структуры;
- дорогие "поисковые" проходы на каждый interaction.

Иногда правильный шаг не в том, чтобы ещё раз `filter`-нуть, а в том, чтобы один раз построить adjacency map, индекс или нормализованное дерево.

---

## Кейс: таблица ролей и прав на 5000 сущностей

Наивный вариант:

- права каждой строки вычисляются через несколько вложенных проходов по массивам;
- фильтрация и сортировка пересчитывают derived state заново;
- UI лагает, а команда лечит это memoization-ом.

Более зрелый подход:

- один раз строим `Map` по `roleId` и `permissionId`;
- membership checks переводим в `Set`;
- тяжёлые derived структуры имеют owner и lifecycle;
- повторные проходы по связанным коллекциям превращаем в индексированные lookup-операции.

Здесь важно не то, что `Map` "быстрее". Важно, что структура начинает соответствовать вопросу, который реально задаёт код.

---

## Что нужно знать

- базовые классы структур данных и их прикладной смысл;
- как частота вызова и размер данных меняют цену линейного прохода;
- где `Map`/`Set` оправданы, а где массивы остаются нормальным default;
- почему алгоритмическая модель и memory cost должны обсуждаться вместе.

---

## Что нужно уметь

- читать кусок кода и видеть паттерн доступа к данным;
- замечать повторные линейные проходы и вложенные lookup-ы;
- предлагать структуру данных, которая упрощает конкретную задачу;
- объяснять trade-off: что выигрывается по CPU, а что дорожает по памяти и сложности кода.

---

## Частые ошибки

- Игнорируют асимптотику, пока всё работает на демо-данных.
- Усложняют структуру данных без реального выигрыша.
- Считают, что `Big O` автоматически объясняет всю производительность.
- Выбирают массив по привычке там, где уже нужен словарь, индекс или дерево.
- Путают "реальный алгоритмический bottleneck" и "просто хочется сделать умнее".

---

## Anti-pattern

"Это же фронтенд, тут алгоритмы не важны".

Они становятся важны ровно в тот момент, когда данных, фильтров, связей и переобходов становится больше, чем в демо. И обычно это обнаруживается не на созвоне про архитектуру, а в виде лага в проде.

---

## 🔧 Практика

1. Возьми реальный кусок кода с несколькими проходами по коллекции.
1. Оцени, какие операции там происходят чаще всего: lookup, dedupe, traversal, sort, join.
1. Предложи другую структуру данных и объясни, что выигрывается и что усложняется.
1. Отдельно ответь, нужен ли здесь новый тип структуры или достаточно один раз построить индекс рядом с owner-модулем.
