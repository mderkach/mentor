# 3.13 Iterators, generators и async iteration

Большинство разработчиков пользуются `for...of`, не думая, что под ним живет iterator protocol. Это нормально до определенного уровня. Дальше важно понимать, как устроена сама модель обхода данных и потоков.

---

- циклы и `for...of` из [3.3 Условия, циклы и control flow](../10-intern/3.3-условия-циклы-и-control-flow.md);
- асинхронность и `await` из [3.11 Promise, async/await и управление асинхронностью](../20-junior/3.11-promise-async-await-и-управление-асинхронностью.md).

## 🎯 Что ты освоишь

- iterator protocol;
- generators и `yield`;
- `for...in` vs `for...of` vs `Object.entries()`;
- async iterators и `for await...of`;
- где эти модели реально полезны в прикладном коде;
- почему генераторы — это не «хитрый синтаксис», а способ выражать ленивое вычисление и пошаговую выдачу данных.

---

## Что важно понять

- Iterator — это контракт получения следующего значения, а не только удобный цикл.
- Не все обходы в JavaScript устроены одинаково: обход свойств объекта, обход значений iterable и обход асинхронного потока — это три разные модели.
- Генератор удобен там, где данные или шаги лучше выдавать постепенно.
- Async iteration важен, когда поток значений приходит не сразу и не одним куском.

---

## Термины по-человечески

- `Iterator` — объект, который умеет отдавать следующее значение по запросу.
- `Generator` — функция, которая может приостанавливать выполнение и продолжать его позже.
- `Async iterator` — тот же контракт, но следующий шаг сам асинхронен.
- `Iterable` — значение, по которому можно пройтись через `for...of`.

---

## Базовый пример

```js
async function* streamPages() {
  yield await fetchPage(1);
  yield await fetchPage(2);
}

for await (const page of streamPages()) {
  render(page);
}
```

---

## Сначала разложи в голове три разных вопроса

Когда разработчик говорит "мне надо пройтись циклом", на самом деле там обычно один из трёх вопросов:

1. Мне нужны свойства объекта?
2. Мне нужны значения iterable-структуры?
3. Мне нужен поток значений, который приходит асинхронно?

Если эти вопросы не разделены, появляются типичные ошибки:

- `for...in` пихают туда, где нужен `for...of`;
- plain object принимают за iterable;
- `for await...of` используют как "какой-то умный цикл", не понимая, что каждая итерация может ждать.

---

## `for...in` vs `for...of`: это не конкуренты

Они похожи синтаксисом, но работают по разным моделям.

### `for...in`

Нужен для обхода имён свойств объекта.

```js
const profile = { name: 'Ada', city: 'London' };

for (const key in profile) {
  console.log(key);
}
```

Он отвечает на вопрос:

"Какие свойства можно перечислить у этого объекта?"

### `for...of`

Нужен для обхода значений iterable-структуры.

```js
const tags = ['js', 'runtime', 'async'];

for (const tag of tags) {
  console.log(tag);
}
```

Он отвечает на вопрос:

"Какие значения выдаёт эта последовательность?"

Практическое правило:

- объектовые ключи и поля — это история про `Object.keys/values/entries` и иногда `for...in`;
- массивы, строки, `Map`, `Set`, generators — это история про `for...of`.

---

## Почему `Object.entries()` часто лучше, чем `for...in`

Потому что в прикладном коде нам чаще нужен предсказуемый обход собственных полей объекта, а не более низкоуровневая enumeration-модель.

```js
const featureFlags = {
  checkoutV2: true,
  compactSidebar: false,
};

for (const [name, isEnabled] of Object.entries(featureFlags)) {
  console.log(name, isEnabled);
}
```

Здесь ты сразу получаешь:

- собственные поля объекта;
- массив пар `[key, value]`;
- нормальную совместимость с `for...of`, `map`, `filter` и прочими массивными операциями после преобразования.

Это особенно важно для обучения: человек должен сначала понять ясную прикладную модель, а уже потом лезть в низкоуровневые нюансы enumeration.

---

## Async iteration — это уже третий слой, а не "for...of с await"

Очень частая ошибка мышления:

"`for await...of` — это просто ещё один цикл".

Нет. Здесь модель уже другая:

- значение приходит не сразу;
- каждая итерация может ждать;
- сам источник данных может быть потоком, пагинацией, очередью сообщений или ленивой асинхронной последовательностью.

```js
async function* streamPages() {
  yield await fetchPage(1);
  yield await fetchPage(2);
}

for await (const page of streamPages()) {
  render(page);
}
```

Смысл тут не в "красивом синтаксисе". Смысл в том, что источник значений живёт во времени.

---

## Полезная карта выбора

Если у тебя:

- plain object с полями -> чаще всего `Object.keys`, `Object.values`, `Object.entries`;
- массив, строка, `Map`, `Set`, generator -> `for...of`;
- асинхронный поток значений -> `for await...of`;
- низкоуровневый обход свойств объекта и особая reason это делать -> `for...in`.

Это не абсолютный закон, но как учебный baseline работает отлично.

---

## Кейс: настройки, список задач и поток страниц

Представь три разных данных:

### 1. Настройки пользователя

```js
const settings = { theme: 'dark', locale: 'ru' };
```

Здесь нужен обход полей объекта. Нормальный прикладной выбор:

```js
for (const [key, value] of Object.entries(settings)) {
  console.log(key, value);
}
```

### 2. Список задач

```js
const tasks = ['build', 'test', 'deploy'];
```

Здесь уже нужна последовательность значений:

```js
for (const task of tasks) {
  console.log(task);
}
```

### 3. Постраничная загрузка API

Здесь данные приходят постепенно:

```js
for await (const page of streamPages()) {
  render(page);
}
```

Если показать новичку сразу все эти формы без объяснения различий, JavaScript действительно выглядит как набор почти одинаковых циклов "для чего-то". А по-хорошему это три разные модели данных и времени.

---

## Частые ошибки

- Путают `for...in` и `for...of`, потому что визуально они похожи.
- Пытаются использовать `for...of` для plain object без преобразования через `Object.entries`.
- Изучают генераторы как экзотику без прикладного сценария.
- Путают массив данных и ленивую последовательность значений.
- Используют `for await...of`, не понимая, что каждая итерация может ждать.
- Пишут iterator abstraction там, где простого массива и функции было бы достаточно.

---

## Anti-pattern

"Раз это middle-тема, надо везде искать применение генераторам".

Нет. Сначала должна быть проблема: ленивость, потоковость, итерация по шагам, backpressure-like сценарий.

---

## 🔧 Практика

1. Возьми три случая: plain object, массив и async generator.
1. Для каждого выбери правильную модель обхода и объясни почему.
1. Напиши простой generator, который выдает последовательность значений по шагам.
1. Перепиши сценарий постраничной загрузки через async generator.
1. Объясни, где массив удобнее iterator, а где уже нет.
