# 3.14 Встроенные структуры и границы данных: Map, Set, JSON, Symbol и structuredClone

На этом уровне уже недостаточно знать только объекты и массивы. Важно уметь выбирать встроенную структуру данных под задачу и понимать, что происходит на границах копирования, сериализации и идентичности.

---

## Что уже должно быть знакомо

- объекты и массивы из [3.5 Объекты и массивы: базовый уровень](../10-intern/3.5-объекты-и-массивы-базовый-уровень.md);
- объектную модель из [3.8 Объектная модель: prototype, class и descriptors](../20-junior/3.8-объектная-модель-prototype-class-и-descriptors.md).

---

## 🎯 Что ты освоишь

- `Object` vs `Map`, `Array` vs `Set`, `Map` vs `WeakMap`, `Set` vs `WeakSet`;
- `JSON.stringify/parse` vs `structuredClone`;
- `Symbol`, `BigInt`, `Date`, `Intl`, `RegExp` как встроенные инструменты;
- как не терять смысл данных на границах сериализации, копирования и идентичности;
- почему выбор структуры данных — это инженерное решение, а не вопрос вкуса.

---

## Зачем это нужно на практике

- Чтобы не использовать plain object там, где нужен настоящий keyed collection.
- Чтобы не клонировать сложный state через `JSON.stringify/parse` и не терять полданных.
- Чтобы не решать проблему уникальности через массивы, `includes` и костыли.
- Чтобы не держать временные метаданные на DOM-узлах или объектах так, что потом это тяжело чистить.

---

## Что важно понять

- Объект не является универсальной заменой для всех коллекций и словарей.
- `JSON.stringify/parse` и `structuredClone` решают разные задачи.
- Встроенные структуры нужны не для «знания стандартной библиотеки», а чтобы не изобретать свои костыли вокруг ключей, уникальности, форматирования и копирования.
- Важно не только "что можно использовать", но и "что этот выбор говорит о природе данных".

---

## Термины по-человечески

- `Map` — коллекция пар ключ-значение, где ключом может быть не только строка.
- `Set` — коллекция уникальных значений.
- `WeakMap/WeakSet` — слабые коллекции для ссылочных ключей/значений, не удерживающие объекты от GC так же, как обычные коллекции.
- `structuredClone` — стандартный способ глубокого клонирования поддерживаемых структур данных.

---

## Сначала главный принцип: похожая структура не всегда означает одинаковую семантику

Очень часто разработчик видит, что две вещи "вроде могут хранить данные", и на этом выбор заканчивается:

- объект и `Map` оба хранят пары;
- массив и `Set` оба хранят набор значений;
- `JSON.stringify/parse` и `structuredClone` оба "копируют";
- `Map` и `WeakMap` оба "умеют кеш".

Но на инженерном уровне это разные контракты. Дальше разбираем их не списком API, а через различия.

---

## `Object` vs `Map`: обе структуры хранят пары, но отвечают на разные вопросы

### `Object`

Нормальный выбор, когда данные по смыслу похожи на запись полей:

```js
const profile = {
  id: 'u-42',
  name: 'Nina',
  role: 'editor',
};
```

Здесь важно не "обходить коллекцию", а хранить осмысленные именованные свойства.

### `Map`

Нормальный выбор, когда данные по смыслу похожи на коллекцию пар:

```js
const sessions = new Map();
sessions.set(userId, session);
```

Плюсы `Map` в прикладном коде:

- ключом может быть не только строка;
- API честно говорит "это коллекция";
- порядок вставки определён;
- удобны `size`, `has`, `get`, `set`, `delete`, итерация.

Практическое правило:

- plain object для модели "поля сущности" или JSON-подобной записи;
- `Map` для динамической коллекции ключ-значение.

---

## Почему `Object.keys()` не делает объект заменой `Map`

Да, объект тоже можно обходить:

```js
for (const [key, value] of Object.entries(settings)) {
  console.log(key, value);
}
```

Но это не делает plain object полноценной коллекцией.

У объекта другая природа:

- он завязан на собственные и унаследованные свойства;
- часть поведения связана с prototype chain;
- строковые ключи и `Symbol`-ключи живут по своим правилам;
- обычно он лучше подходит для данных вида "сущность с полями", а не "изменяемое хранилище".

Если у тебя код звучит как "добавить элемент", "удалить элемент", "проверить наличие", "получить размер", "итерация по коллекции", это уже сильный сигнал смотреть в сторону `Map`.

---

## `Array` vs `Set`: не путай список и уникальное множество

### `Array`

Это упорядоченный список, где:

- важна позиция;
- возможны дубликаты;
- доступны методы преобразования, фильтрации, сортировки.

### `Set`

Это коллекция уникальных значений, где:

- главное не индекс, а факт присутствия;
- дубликаты автоматически схлопываются;
- операция membership обычно читается честнее.

```js
const selectedIds = new Set();
selectedIds.add('a');
selectedIds.add('a');
selectedIds.add('b');

console.log(selectedIds); // Set {'a', 'b'}
```

Практическое правило:

- если тебе нужен список элементов и важен порядок обработки, чаще нужен массив;
- если тебе нужен набор уникальных значений и быстрая проверка "уже есть или нет", чаще нужен `Set`.

---

## `Map` vs `WeakMap`: оба умеют хранить пары, но только один подходит для GC-friendly metadata

### `Map`

Нужен, когда:

- коллекция должна быть наблюдаемой;
- нужны размер, перебор и явный lifecycle;
- ключи живут независимо от конкретного объекта-потребителя.

### `WeakMap`

Нужен, когда:

- ключами выступают именно объекты;
- данные являются временными метаданными, привязанными к жизни этих объектов;
- тебе не нужен перебор всей коллекции;
- важно не удерживать объект в памяти только ради кэша или метаданных.

Пример с DOM-элементами:

```js
const clickedState = new WeakMap();

function attach(button) {
  clickedState.set(button, false);

  button.addEventListener('click', () => {
    clickedState.set(button, true);
  });
}
```

Здесь `WeakMap` хороша тем, что метаданные живут ровно столько, сколько живёт сам объект-ключ.

---

## Почему у `WeakMap` и `WeakSet` нет `size` и нормального перебора

Это не "странное ограничение API", а следствие weak-семантики.

Если бы движок позволял тебе надёжно перечислить содержимое `WeakMap`, ты бы мог наблюдать, какие объекты уже собраны garbage collector-ом, а какие ещё нет. Это ломало бы саму модель.

Отсюда практический вывод:

- `WeakMap` и `WeakSet` нужны не как "чуть более хитрые Map/Set";
- они нужны только для узких сценариев вокруг lifetime объекта и памяти.

Если тебе нужен перебор, размер, логирование содержимого и контроль над жизненным циклом, почти наверняка нужен обычный `Map` или `Set`.

---

## `Set` vs `WeakSet`: та же история, но для membership по объектам

`WeakSet` уместен, когда нужно помнить факт про объект, не удерживая его в памяти:

```js
const visited = new WeakSet();

function mark(node) {
  visited.add(node);
}
```

Ограничения такие же:

- только объекты;
- нет полноценного перебора;
- не подходит для доменных данных и обычных коллекций приложения.

---

## `JSON.stringify/parse` vs `structuredClone`: оба "копируют", но только один вообще про копирование

### `JSON.stringify/parse`

Это история про сериализацию:

- подготовить данные к передаче по сети;
- положить в storage;
- сохранить как строку;
- получить JSON-compatible representation.

### `structuredClone`

Это история про глубокое клонирование поддерживаемых структур:

- сохранить shape данных;
- корректно обработать циклические ссылки;
- клонировать `Map`, `Set`, `Date`, часть других встроенных типов;
- использовать тот же базовый механизм, который участвует в `postMessage()` и некоторых других API.

```js
const state = {
  filters: new Set(['new', 'popular']),
  createdAt: new Date(),
};

const nextState = structuredClone(state);
```

---

## Что теряет `JSON.stringify/parse`

Вот почему этот приём опасно называть "deep clone по-быстрому":

- теряются `Map`, `Set`, `Date`, `RegExp`-семантика;
- ломаются циклические ссылки;
- отваливаются значения, не представимые в JSON;
- пропадает смысл "это была структура данных", остаётся только JSON-совместимая строка.

Если ты пишешь `JSON.stringify/parse` именно для клонирования runtime-state, это уже повод остановиться и спросить: зачем я сериализую то, что никуда не отправляю?

---

## Что важно знать про `structuredClone`

`structuredClone(...)` гораздо честнее для копирования, но и у него есть границы.

По данным MDN:

- функции не клонируются;
- DOM-узлы не клонируются;
- prototype chain не копируется;
- property descriptors, getters и setters не сохраняются как есть.

Практический вывод:

- `structuredClone` отлично подходит для plain data;
- `structuredClone` плохо подходит для "объектов с поведением", сложных class-instance и runtime-magic.

То есть это инструмент для данных, а не для переноски всей объектной модели один в один.

---

## `Symbol`: когда строкового ключа уже недостаточно

`Symbol` нужен не "ради редкой фичи языка", а когда важно:

- не столкнуться по имени свойства;
- добавить внутренний ключ без риска конфликтов;
- участвовать в protocol-level механике языка вроде итерации.

```js
const internalId = Symbol('internalId');

const entity = {
  [internalId]: 'hidden-42',
  title: 'Draft',
};
```

Для everyday данных `Symbol` нужен редко. Но полезно понимать, почему это отдельный тип, а не просто "ещё одна строка".

---

## Built-in API, которые стоит знать как инженерные инструменты

### `Intl`

Нужен, когда речь о локализации форматов:

- дата;
- число;
- валюта;
- plural rules.

Это почти всегда лучше ручного склеивания строк.

### `Date`

Нужен для работы со временем, но не надо ждать от него "полной удобной модели времени".

Практический вывод:

- храни абсолютное время явно;
- различай machine timestamp и user-facing formatting;
- для отображения почти всегда смотри в сторону `Intl`.

### `RegExp`

Нужен, когда задача действительно про pattern matching, а не про "попробуем одной регуляркой распарсить весь мир".

### `BigInt`

Нужен там, где важна точная работа с большими целыми числами, а не плавающая арифметика `Number`.

---

## Базовый пример

```js
const cache = new Map();
cache.set(userId, profile);

const tags = new Set(['js', 'runtime', 'js']);
// Set {'js', 'runtime'}

const cloned = structuredClone(originalState);
```

---

## Частые ошибки

- Используют обычный объект как словарь, не думая про семантику ключей.
- Используют массив как множество уникальных значений и потом постоянно пишут `includes`.
- Клонируют сложные данные через `JSON.stringify/parse`, не понимая потерь.
- Применяют `WeakMap/WeakSet`, не осознавая, зачем вообще нужна weak-семантика.
- Пытаются хранить в `WeakMap` всё подряд, а потом удивляются отсутствию перебора и `size`.
- Путают сериализацию данных с их копированием.
- Используют `structuredClone` для объектов с поведением, ожидая сохранить всю объектную модель.

---

## Anti-pattern

"Если что-то хранит пары ключ-значение, значит обычного объекта достаточно".

Иногда да. Но на масштабе это часто рождает лишние ограничения и неясную семантику.

---

## Кейс: local cache для DOM-элементов

Задача: нужно хранить временные вычисленные метаданные для DOM-элементов.

Плохой вариант:

- писать служебные поля прямо на DOM-элемент;
- держать глобальный `Map`, не задумываясь о cleanup;
- использовать plain object с ключами-строками от `id`, хотя объект может жить и меняться отдельно.

Нормальный вариант:

- если метаданные должны жить ровно столько, сколько живёт DOM-элемент, смотреть в сторону `WeakMap`;
- если нужен явный ownership, размер и перебор, использовать `Map` и явно чистить коллекцию.

То есть вопрос не в "какая структура круче", а в том, какой у данных lifecycle.

---

## Что нужно знать

- чем plain object отличается от `Map`;
- чем список отличается от множества;
- зачем weak-коллекции существуют как отдельный класс инструментов;
- почему сериализация и копирование нельзя смешивать в одну ментальную модель.

---

## Что нужно уметь

- выбирать между `Object`, `Map`, `Set`, `WeakMap`, `WeakSet` под реальный сценарий;
- объяснять, где данные являются plain record, а где коллекцией;
- не использовать `JSON.stringify/parse` как суррогат глубокого клонирования;
- видеть границу между данными и поведением при копировании.
