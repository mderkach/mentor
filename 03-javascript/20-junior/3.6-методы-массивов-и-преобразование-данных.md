# 3.6 Методы массивов и преобразование данных

Именно на массивах чаще всего видно, кто пишет JavaScript как последовательность случайных операций, а кто умеет строить читаемую цепочку преобразований данных.

Проблема в том, что многие методы внешне похожи:

- `map` и `forEach` оба "куда-то ходят по массиву";
- `find` и `filter` оба "что-то выбирают";
- `sort` и `toSorted` оба "сортируют";
- `slice`, `splice` и `toSpliced` звучат почти как одна семья.

Если не понимать различия, код быстро становится хрупким: случайные мутации, лишние проходы, странная асинхронность и ревью в стиле "вроде работает, но почему-то неприятно".

- `map`, `filter`, `find`, `some`, `every`, `forEach`;
- `reduce` без религиозных крайностей;
- `flat`, `flatMap`, `sort`, `toSorted`, `toSpliced`, `with`;

- почему `forEach(async ...)` почти всегда ловушка;
- как писать преобразование данных так, чтобы его было легко читать и ревьюить.

---

## Что важно понять

- Методы массивов полезны не сами по себе, а как язык преобразований над коллекцией.
- Самая частая production-ошибка здесь — случайная мутация данных там, где ожидалась новая версия.
- Хороший код с массивами читается как последовательность шагов, а не как борьба за самый короткий chain.
- У каждого метода есть своя роль. Когда один метод начинают использовать вместо другого "потому что тоже работает", теряется смысл кода.

---

## Термины по-человечески

- `Transformation` — преобразование одной коллекции в другую форму.
- `Predicate` — функция, которая отвечает `true/false` для элемента.
- `Accumulator` — значение, которое накапливается в `reduce`.
- `Mutation` — изменение исходного массива.
- `Immutable update` — создание новой версии массива без изменения исходной.

---

## Базовый пример

```js
const visibleTitles = posts
  .filter((post) => post.isPublished)
  .map((post) => post.title.trim())
  .toSorted((a, b) => a.localeCompare(b));
```

---

## Сначала ответь на вопрос: что ты хочешь получить

Когда у тебя есть массив, чаще всего задача одна из этих:

- пройтись по элементам и выполнить side effect;
- получить новый массив;
- найти один элемент;
- проверить условие;
- собрать одно итоговое значение;
- изменить порядок;
- обновить массив без мутации.

Если этот вопрос не задан, разработчик начинает выбирать метод по привычке. Именно отсюда и растут `forEach` вместо `map`, `filter()[0]` вместо `find`, `sort` с тихой мутацией и монструозный `reduce` на пол-экрана.

---

## `forEach` vs `map`: одинаковый обход, разный смысл

### `forEach`

Нужен, когда ты не хочешь получить новый массив, а делаешь side effect.

```js
items.forEach((item) => {
  console.log(item.id);
});
```

Подходит для:

- логирования;
- метрик;
- вызова побочных действий;
- простого прохода без результата.

### `map`

Нужен, когда из каждого элемента ты строишь новую версию.

```js
const ids = items.map((item) => item.id);
```

Подходит для:

- получения нового массива;
- приведения данных к другой форме;
- подготовки данных для рендера или отправки дальше.

Практическое правило:

- если результат нового массива тебе нужен, это `map`;
- если нового массива не нужно, а нужен только проход, это `forEach`.

Плохой вариант:

```js
const ids = [];

items.forEach((item) => {
  ids.push(item.id);
});
```

Такой код работает, но он хуже читается: ты используешь side-effect метод там, где язык уже дал прямой инструмент преобразования.

---

## `find` vs `filter`: один элемент против набора

`find` нужен, когда ты ищешь первый подходящий элемент.

```js
const admin = users.find((user) => user.role === 'admin');
```

`filter` нужен, когда тебе нужны все подходящие элементы.

```js
const admins = users.filter((user) => user.role === 'admin');
```

Почему это важно:

- `find` останавливается раньше;
- `filter` всегда строит новый массив;
- по названию метода сразу видно намерение автора.

Плохой smell:

```js
const admin = users.filter((user) => user.role === 'admin')[0];
```

Это сигнал, что автор хотел `find`, но взял более тяжёлый и менее точный инструмент.

---

## `some` / `every` vs `filter().length`

Если вопрос звучит так:

- "есть ли хотя бы один?"
- "все ли подходят?"

то чаще всего нужны `some` или `every`.

```js
const hasExpired = tokens.some((token) => token.isExpired);
const allValid = tokens.every((token) => token.isValid);
```

Почему не `filter().length`:

- это хуже выражает смысл;
- делает лишнюю работу;
- создаёт промежуточный массив, который тебе не нужен.

---

## `reduce`: полезный, но не универсальный молоток

`reduce` хорош, когда действительно нужно собрать одно итоговое значение:

- сумму;
- словарь по id;
- агрегат;
- grouping;
- сложный accumulator с ясным owner.

```js
const usersById = users.reduce((acc, user) => {
  acc[user.id] = user;
  return acc;
}, {});
```

Но `reduce` плохо подходит как "универсальный способ сделать всё в одном месте".

Если внутри `reduce`:

- фильтрация;
- сортировка;
- несколько веток;
- несколько разных смыслов;
- mutation-магия, которую трудно читать,

то почти всегда лучше разбить решение на несколько шагов или взять более прямой метод.

Хорошее правило:

- если без комментария непонятно, что делает `reduce`, вероятно, его уже слишком много.

---

## `sort` vs `toSorted`: вот здесь чаще всего ломают данные

`sort` мутирует исходный массив.

```js
const sorted = items.sort(compareFn);
```

Это значит:

- исходный массив изменился;
- все, кто держат на него ссылку, увидят новый порядок;
- bug может всплыть далеко не там, где сортировка произошла.

`toSorted` возвращает новый массив и оставляет исходный нетронутым.

```js
const sorted = items.toSorted(compareFn);
```

Практическое правило:

- если тебе нужен новый массив, почти всегда начинай с `toSorted`;
- `sort` используй только тогда, когда мутация действительно допустима и понятна.

То же мышление полезно держать для:

- `splice` vs `toSpliced`;
- прямой записи по индексу vs `with`.

---

## `slice`, `splice`, `toSpliced`: похожие названия, разная цена

### `slice`

Берёт кусок массива и не мутирует исходник.

```js
const firstThree = items.slice(0, 3);
```

### `splice`

Меняет исходный массив: удаляет, вставляет, перестраивает.

```js
items.splice(1, 2);
```

### `toSpliced`

Делает почти ту же операцию, что `splice`, но возвращает новую версию массива.

```js
const next = items.toSpliced(1, 2);
```

Если человек путает `slice` и `splice`, это почти всегда заканчивается багом "почему у меня внезапно исчезли элементы".

---

## Методы массивов тоже имеют ограничения

Вот что важно знать, чтобы не воспринимать их как магию:

### 1. Не все методы умеют остановиться раньше

- `find`, `some`, `every` могут закончить обход раньше;
- `map`, `filter`, `forEach` проходят коллекцию до конца.

### 2. Не все методы подходят для async-логики

`forEach(async ...)` почти всегда выглядит проще, чем работает.

```js
items.forEach(async (item) => {
  await save(item);
});
```

Проблема:

- `forEach` не ждёт `await`;
- порядок и завершение такого кода часто понимают неправильно;
- внешняя функция может закончиться раньше, чем завершатся все операции.

Если нужен последовательный async-проход, чаще честнее:

```js
for (const item of items) {
  await save(item);
}
```

Если нужен параллельный запуск:

```js
await Promise.all(items.map((item) => save(item)));
```

### 3. Sparse arrays ведут себя не всегда так, как ожидают

Массив с дырками и "массив с `undefined`" — это не одно и то же.

Это особенно важно при:

- `map`;
- `forEach`;
- `entries`;
- `length`;
- ручной работе по индексам.

На junior-уровне не обязательно знать все edge cases наизусть, но важно хотя бы помнить: массивы в JS не всегда являются плотным списком значений.

---

## Когда обычный `for` или `for...of` лучше методов массива

Методы массивов не обязаны побеждать всегда.

Обычный цикл или `for...of` честнее, когда:

- нужна ранняя остановка в сложной логике;
- есть `break` / `continue`;
- нужен последовательный `await`;
- слишком длинный chain уже перестал читаться;
- операция не про "красивое преобразование", а про пошаговый контроль.

Зрелый код не пытается доказать любовь к array methods. Он выбирает форму, в которой лучше видно смысл.

---

## Кейс: рендер каталога и локальное сохранение

Представь массив товаров. Тебе нужно:

- взять только доступные;
- построить список карточек;
- отсортировать по цене;
- потом сохранить видимые `id`.

Нормальное решение может выглядеть так:

```js
const visibleProducts = products
  .filter((product) => product.isAvailable)
  .toSorted((a, b) => a.price - b.price);

const cardModels = visibleProducts.map((product) => ({
  id: product.id,
  title: product.title,
}));

const ids = visibleProducts.map((product) => product.id);
```

А side effect уже вынести отдельно:

```js
ids.forEach((id) => saveRecentId(id));
```

Здесь видно важное разделение:

- сначала преобразование данных;
- потом побочное действие.

Это и делает код более ревьюемым.

---

## Частые ошибки

- Используют `forEach`, когда нужен новый массив.
- Пихают всю бизнес-логику в один `reduce`.
- Мутируют исходный массив через `sort` или `splice`, не ожидая этого.
- Делают длинный chain без промежуточных имен там, где логика уже потеряла прозрачность.
- Используют `filter()[0]`, когда нужен `find`.
- Проверяют наличие через `filter().length`, когда нужен `some`.
- Пишут `forEach(async ...)`, ожидая, что внешний код будет ждать завершения итераций.

---

## Anti-pattern

"Если можно собрать всё в один chain, значит это автоматически хорошо".

Нет. У преобразований данных есть порог читаемости, после которого лучше назвать промежуточный шаг.
