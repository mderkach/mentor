# 3.11 Promise, async/await и управление асинхронностью

Асинхронность в JavaScript ломает не синтаксисом, а потерей причинно-следственной связи. Этот урок про то, как сохранять контроль над потоком: кто ждет, кто отменяет, кто агрегирует результаты и где реально происходит ошибка.

- базовая отладка через DevTools из [0.0.1 JavaScript в DevTools: console, breakpoints, scope и async stacks](../00-zero/0.0.1-JavaScript-в-DevTools-console-breakpoints-scope-async-stacks.md).

- `Promise`, `then/catch/finally`;
- `async function` и `await`;
- `Promise.all`, `allSettled`, `race`, `any`;
- `AbortController` как базу отмены;
- как не превратить асинхронный код в неуправляемую цепочку зависимостей.

---

## Что важно понять

- `async/await` делает запись линейнее, но не отменяет саму асинхронную природу выполнения.
- Promise combinators нужны не для «знания API», а для выбора правильной стратегии ожидания.
- Асинхронный код должен явно отвечать на вопросы: что делаем параллельно, что делаем последовательно и что происходит при ошибке или отмене.
- Основные баги асинхронности в продукте редко выглядят как "не знаю синтаксис". Обычно это stale result, гонка состояний, out-of-order response и тихо проглоченная ошибка.

---

## Термины по-человечески

- `Promise` — объект, представляющий результат, который станет известен позже.
- `Await` — пауза в рамках текущей async-функции до получения результата.
- `AbortController` — стандартный способ сигнализировать отмену для поддерживающих API.
- `Stale update` — обновление UI или состояния результатом, который уже не соответствует текущему запросу или текущему экрану.
- `Out-of-order response` — ситуация, где более старый запрос приходит позже нового и перетирает более актуальный результат.

---

## Базовый пример

```js
const controller = new AbortController();

try {
  const [profile, settings] = await Promise.all([
    loadProfile({ signal: controller.signal }),
    loadSettings({ signal: controller.signal }),
  ]);

  render(profile, settings);
} catch (error) {
  handleError(error);
}
```

---

## `AbortController` vs "просто игнорируем результат"

Обе стратегии встречаются в коде, но это не одно и то же.

### Игнорируем результат

Подходит, когда:

- операция уже ушла в мир;
- остановить её нельзя или не нужно;
- нам важно только не применять устаревший результат.

```js
let requestId = 0;

async function loadSearch(query) {
  const currentRequestId = ++requestId;
  const result = await search(query);

  if (currentRequestId !== requestId) {
    return;
  }

  render(result);
}
```

### `AbortController`

Подходит, когда API реально поддерживает отмену и тебе важно:

- не тратить лишнюю сеть;
- не читать ненужный response body;
- не держать лишний in-flight запрос;
- явно завершить старую операцию.

MDN отдельно подчёркивает: при `abort()` `fetch()` отвергается с `AbortError`, и даже чтение body после аборта тоже упадёт с `AbortError`.

```js
let controller;

async function loadSearch(query) {
  controller?.abort();
  controller = new AbortController();

  try {
    const response = await fetch(`/api/search?q=${query}`, {
      signal: controller.signal,
    });

    const payload = await response.json();
    render(payload);
  } catch (error) {
    if (error.name === 'AbortError') {
      return;
    }

    throw error;
  }
}
```

Практическое правило:

- если можно отменить старую операцию, `AbortController` лучше;
- если отмена технически не даёт выигрыша, но stale result опасен, достаточно игнорировать устаревший результат;
- часто нужны обе вещи одновременно: и отменить старый запрос, и не применять старый ответ.

## Failure modes асинхронного кода: где реально ломается продукт

### 1. Гонки состояний

Два async-потока меняют один и тот же state, не понимая, кто из них "главнее".

### 2. Stale updates

Пользователь уже ушёл на другой экран или ввёл новый запрос, а старый результат всё ещё прилетает и перерисовывает UI.

### 3. Out-of-order responses

Более новый запрос уходит позже, но старый ответ приходит после него и перетирает актуальные данные.

### 4. Потерянные ошибки

Ошибка произошла, но:

- её проглотили в `catch`;
- её не `await`-нули;
- её превратили в `null` без контракта;
- `Promise.all` завалил весь сценарий, а команде был нужен частичный успех.

### 5. Тихая отмена

Запрос отменили или просто не применили результат, но UI не знает, что произошло, и остаётся в подвешенном состоянии.

### 6. Каскадные side effects

Один async-result приводит к toast, analytics, router transition, state update и кешированию сразу, хотя результат уже устарел.

## Кейс: поиск по каталогу и out-of-order response

Пользователь быстро вводит:

- `ca`
- потом `car`
- потом `card`

Если код просто делает три `fetch()` подряд и всегда рендерит последний пришедший ответ, UI может показать не `card`, а старый ответ для `ca` или `car`.

Это и есть зрелая async-проблема:

- синтаксис корректен;
- код компилируется;
- но contract между UI и сетью сломан.

Лечат это обычно так:

- отменяют предыдущий запрос;
- или маркируют запросы и игнорируют stale result;
- или делают оба шага вместе.

---

## Combinators надо выбирать по failure mode, а не по привычке

`Promise.all` говорит:

- все должны успешно завершиться;
- одна ошибка валит весь результат.

`Promise.allSettled` говорит:

- мне нужен итог по каждому запросу, даже если часть упала.

`Promise.race` говорит:

- меня интересует первый завершившийся исход, не обязательно успешный.

`Promise.any` говорит:

- меня интересует первый успешный результат.

То есть combinator — это не синтаксическая прихоть, а выбор модели отказа.

---

## Частые ошибки

- Ждут последовательность там, где можно было запускать параллельно.
- Используют `Promise.all`, когда частичный успех тоже допустим и нужен `allSettled`.
- Теряют ошибку внутри async-flow, потому что не понимают, где именно она будет поймана.
- Не закладывают сценарий отмены, хотя UI уже может уйти в другое состояние.
- Игнорируют race conditions и stale results, потому что "локально же всё работает".
- Молча swallow-ят `AbortError` и любые другие ошибки одним и тем же `catch`.

---

## Anti-pattern

"Переписали на `async/await`, значит асинхронность стала простой".

Нет. Она стала читабельнее, но ответственность за порядок, отмену и обработку ошибок никуда не исчезла.

---

## 🔧 Практика

1. Возьми код с `then`-цепочкой и перепиши на `async/await`.
1. Отдельно определи, что должно идти параллельно, а что последовательно.
1. Замени один неудачный combinator на более уместный и объясни выбор.
1. Возьми UI-сценарий с поиском или автокомплитом и покажи, как защититься от stale result.
1. Сравни решение через `AbortController` и решение через игнорирование устаревшего ответа.
