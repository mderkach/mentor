# 3.8 Объектная модель: prototype, class и descriptors

Классы в JavaScript удобны, но под ними по-прежнему живет prototype-based модель. Этот урок нужен, чтобы не путать синтаксический сахар с реальной моделью наследования и поведения объектов.

Без этого разработчик обычно запоминает только внешнюю форму:

- есть `class`;
- есть `extends`;
- есть методы;
- где-то существует `prototype`, но вроде не очень понятно зачем.

Проблема в том, что без этой модели потом становятся непонятны:

- откуда объект берёт метод;
- чем собственное свойство отличается от унаследованного;
- почему `for...in` ведёт себя не так, как `Object.keys`;
- почему `class` не делает JavaScript "как Java".

## Что важно понять

- `class` в JavaScript не отменяет prototype model, а лишь упаковывает ее в более читаемый синтаксис.
- Наследование полезно не потому, что «ООП так учит», а потому что иногда реально выражает связь между сущностями.
- На junior-уровне важнее не строить хитрые иерархии, а понимать, что произойдет при чтении метода, поля и наследовании поведения.
- Prototype model в JavaScript нужна не ради теории, а чтобы объекты могли переиспользовать общее поведение без копирования одних и тех же методов в каждый экземпляр.

---

## Термины по-человечески

- `Prototype` — объект, у которого другой объект может искать недостающие свойства.
- `Descriptor` — метаданные свойства: можно ли его менять, перечислять, переопределять.
- `Getter/Setter` — специальные функции для чтения или записи свойства.
- `Own property` — свойство, которое лежит прямо на самом объекте.
- `Inherited property` — свойство, которое объект нашёл выше по prototype chain.

---

## Базовый пример

```js
class User {
  #token;

  constructor(name, token) {
    this.name = name;
    this.#token = token;
  }

  get isAuthenticated() {
    return Boolean(this.#token);
  }
}
```

---

## Зачем JavaScript вообще нужны прототипы

Самый полезный ответ без академической пыли такой:

прототипы нужны, чтобы объекты могли разделять общее поведение.

Представь, что у тебя есть тысяча пользователей. Если каждый объект будет хранить собственную копию одного и того же метода `getFullName`, это лишняя работа и лишняя память.

Prototype model решает это так:

- данные конкретного объекта живут на самом объекте;
- общее поведение хранится выше;
- если объект не нашёл свойство у себя, он смотрит дальше по цепочке.

Это и есть prototype chain в практическом смысле.

---

## Как реально происходит поиск свойства

Когда ты пишешь:

```js
user.isAuthenticated
```

JavaScript делает не магию, а довольно понятный поиск:

1. Есть ли `isAuthenticated` прямо у `user`?
2. Если нет, смотри в его prototype.
3. Если там нет, иди ещё выше.
4. Продолжай, пока цепочка не закончится.

Это важно, потому что объект может "выглядеть так, будто у него есть свойство", хотя физически оно лежит не на нём, а выше по цепочке.

## Own properties vs inherited properties

```js
class User {
  constructor(name) {
    this.name = name;
  }

  greet() {
    return `Hi, ${this.name}`;
  }
}

const user = new User('Ada');
```

Здесь:

- `name` — собственное свойство экземпляра;
- `greet` — не лежит прямо на `user`, а приходит через prototype.

Это легко проверить:

```js
console.log(Object.hasOwn(user, 'name')); // true
console.log(Object.hasOwn(user, 'greet')); // false
console.log('greet' in user); // true
```

И вот тут появляется очень важное различие:

- `Object.hasOwn()` отвечает на вопрос "это свойство лежит прямо на объекте?";
- оператор `in` отвечает на вопрос "это свойство вообще доступно объекту, в том числе через prototype?".

---

## `in` vs `Object.hasOwn()`: это не одно и то же

Если тебе важно понять, что действительно принадлежит объекту как данным, чаще нужен `Object.hasOwn()`.

Если тебе важно понять, доступно ли свойство в принципе, включая наследование, полезен `in`.

```js
console.log('toString' in user); // true
console.log(Object.hasOwn(user, 'toString')); // false
```

Это хороший пример того, почему нельзя путать "свойство видно" и "свойство лежит прямо здесь".

---

## Почему `for...in` связан с прототипами

Вот где prototype model внезапно начинает влиять на прикладной код.

`for...in` обходит перечислимые свойства объекта и может пройтись не только по собственным свойствам, но и по унаследованным enumerable-свойствам.

Именно поэтому `for...in`:

- требует больше осторожности;
- не является лучшим default для plain object в прикладном коде;
- плохо подходит для массивов.

В everyday-коде чаще безопаснее:

- `Object.keys(obj)`
- `Object.values(obj)`
- `Object.entries(obj)`

Потому что они дают предсказуемый обход собственных полей объекта.

---

## `class` не отменяет prototype model

Одна из самых вредных иллюзий у начинающих:

"Раз я пишу `class`, значит дальше работает другая, более привычная объектная система".

Нет. `class` делает код удобнее, но модель остаётся той же:

- методы экземпляра живут на prototype;
- `extends` строит цепочку прототипов;
- `super` тоже работает поверх этой модели;
- поведение объекта всё ещё определяется поиском по chain.

Поэтому хороший JavaScript-разработчик понимает `class`, но не забывает, что за ним стоит.

---

## Когда про прототипы нужно помнить каждый день, а когда нет

Не нужно жить с мыслью о `[[Prototype]]` каждую минуту. Но про него важно помнить в таких местах:

- чтение чужого кода с `class` и наследованием;
- `for...in`;
- проверка свойств через `in`;
- debugging странного поведения метода или поля;
- расширение встроенных объектов и прочая опасная магия;
- метапрограммирование и advanced runtime patterns.

В обычной feature-разработке тебе чаще нужен не "культ прототипов", а понимание базовой модели: откуда берётся поведение и почему свойства бывают собственные и унаследованные.

---

## Почему composition часто лучше наследования

Наследование может быть полезно, но в JS им очень легко перегрузить систему.

Плохой путь:

- сделать глубокую иерархию;
- переопределять методы в нескольких местах;
- спрятать бизнес-правила в prototype magic.

Более здоровый путь для прикладного фронтенда:

- маленькие объекты с понятной ответственностью;
- общая логика через композицию и явные функции;
- inheritance только там, где связь реально стабильна и оправдана.

То есть смысл урока не в том, чтобы запретить наследование. Смысл в том, чтобы оно не стало default просто потому, что есть `class`.

---

## Частые ошибки

- Думают, что `class` делает JavaScript «как Java».
- Не понимают, где метод берется из prototype, а где лежит на самом объекте.
- Используют наследование там, где проще и безопаснее composition.
- Пишут getters/setters с побочными эффектами, из-за чего чтение свойства перестает быть очевидным.
- Путают `in` и `Object.hasOwn()`.
- Используют `for...in`, не понимая, что prototype chain тоже может влиять на обход.

---

## Anti-pattern

"Раз есть `class`, значит сложная иерархия наследования — хороший default".

В JavaScript это быстро превращается в хрупкую объектную модель и больную поддержку.
