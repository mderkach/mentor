# 3.10 ESM-модули, import/export и code splitting

Модули — это не просто синтаксис `import/export`. Через них мы задаем границы ответственности, переиспользование кода и момент, когда логика вообще попадает в рантайм.

---

## Что уже должно быть знакомо

- функции и объекты из intern-уровня;
- ошибки и контракты из [3.9 Ошибки, throw и fail-fast patterns](./3.9-ошибки-throw-и-fail-fast-patterns.md).

---

## 🎯 Что ты освоишь

- ESM как основной модульный формат;
- `import`, `export`, default и named exports;
- `dynamic import()` и code splitting;
- `top-level await` и где он уместен;
- почему модульная граница — это архитектурное решение, а не только способ разбить файлы.

---

## Что важно понять

- Плохая модульность делает код тяжелым для чтения даже при хорошем синтаксисе внутри файла.
- Граница модуля должна помогать навигации и зависимостям, а не просто дробить код на мелкие куски.
- Dynamic import нужен по задаче: lazy loading, feature boundary, optional dependency — а не «потому что можно».
- `import` сам по себе не создаёт хороший boundary. Хороший boundary появляется только там, где модуль не тащит наружу случайные side effects и скрытое состояние.

---

## Термины по-человечески

- `Named export` — модуль отдает несколько именованных сущностей.
- `Default export` — модуль отдает одну основную сущность как default.
- `Code splitting` — загрузка кода частями, а не всем сразу.
- `Side effect module` — модуль, который при импорте уже что-то делает: пишет в глобальное состояние, подписывается, стартует таймер, меняет окружение.
- `Boundary contract` — ответ на вопрос, что модуль обещает наружу и что он не должен делать за спиной потребителя.

---

## Базовый пример

```js
export function formatDate(value) {
  return new Intl.DateTimeFormat('ru-RU').format(value);
}

const modalModule = await import('./modal.js');
modalModule.openModal();
```

---

## `import` и настоящая модульная граница — не одно и то же

Очень частая ошибка выглядит так:

- код разложили по файлам;
- между файлами стоят `import/export`;
- значит архитектура уже "модульная".

На практике модуль может оставаться плохой зависимостью, если он:

- экспортирует слишком много случайных вещей;
- хранит скрытое глобальное состояние;
- делает side effects при импорте;
- заставляет consumer-а знать его внутреннюю кухню.

То есть хороший модуль — это не "файл поменьше", а зависимость с понятным контрактом.

---

## Side effects: почему "удобный модуль" быстро становится хрупким

Плохой пример:

```js
import './analytics.js';
```

На вид всё просто. Но если модуль `analytics.js` при импорте:

- читает environment;
- вешает глобальные listeners;
- патчит `fetch`;
- стартует таймер;
- пишет в singleton,

то у тебя появляется зависимость, которая не выглядит опасной по коду вызова.

Проблемы отсюда типовые:

- порядок импортов внезапно начинает что-то значить;
- тесты зависят от того, что уже было импортировано;
- tree shaking и lazy loading дают неожиданные эффекты;
- один "удобный shared module" начинает жить как скрытый runtime-owner.

Практическое правило:

- pure module экспортирует значения и функции;
- effectful module должен явно читаться как effectful и иметь понятную точку инициализации.

---

## Глобальное состояние под видом модуля

Иногда модуль выглядит как невинная утилита:

```js
// sessionStore.js
let session = null;

export function setSession(nextSession) {
  session = nextSession;
}

export function getSession() {
  return session;
}
```

Технически всё "нормально". Но инженерно это уже shared mutable state.

Что это меняет:

- consumer зависит не только от API, но и от порядка вызовов;
- тесты инициализации становятся хрупкими;
- параллельные сценарии и SSR-like execution начинают вести себя неочевидно;
- модуль уже не просто helper, а state owner.

Проблема не в том, что global state всегда запрещён. Проблема в том, что его часто прячут в модуле, который выглядит как обычная функция-утилита.

---

## `dynamic import()` полезен, но меняет не только размер бандла

Когда ты делаешь:

```js
const modalModule = await import('./modal.js');
```

ты меняешь не только момент загрузки. Ты меняешь ещё и:

- момент инициализации модуля;
- жизненный цикл side effects, если они есть;
- порядок, в котором код впервые попадает в runtime;
- возможные failure modes: сетевой сбой, timeout, частичная загрузка.

Поэтому `dynamic import()` стоит выбирать не только по принципу "лень тянуть код сразу", а по вопросу:

- эта зависимость действительно опциональна;
- пользователь получает ценность от более поздней загрузки;
- модуль не ломает систему, если инициализируется позже.

---

## Частые ошибки

- Смешивают default и named exports без понятной причины.
- Делают модуль слишком большим и тащат в него unrelated logic.
- Используют dynamic import без понимания, что это меняет момент загрузки и инициализации.
- Прячут циклические зависимости вместо того, чтобы чинить границы.
- Прячут global state и side effects в модуле, который выглядит как обычный helper.

---

## Anti-pattern

"Разбили код по файлам — значит модульная архитектура уже есть".

Без хороших границ модульность становится просто косметической нарезкой.

---

## 🔧 Практика

1. Возьми один большой файл и предложи нормальную модульную нарезку.
1. Отдели pure-утилиту, runtime-логику и optional feature import.
1. Объясни, где dynamic import реально нужен, а где нет.
1. Для одного shared-модуля ответь, нет ли в нём скрытых side effects или global state.
