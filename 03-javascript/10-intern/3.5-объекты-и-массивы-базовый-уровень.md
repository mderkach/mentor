# 3.5 Объекты и массивы: базовый уровень

Большая часть прикладного JavaScript работает с объектами и массивами. Поэтому важно не просто «уметь обратиться по ключу», а понимать, как устроены данные, как они копируются и как не ломать их случайными изменениями.

## 🎯 Что ты освоишь

- object literal и базовую работу со свойствами;
- массив как ordered-structure, а не «объект с индексами»;
- `keys`, `values`, `entries` на базовом уровне;
- `for...in` vs `Object.keys/Object.values/Object.entries` на базовом уровне;
- `push/pop`, `shift/unshift`, `slice/splice`;
- optional chaining и nullish coalescing для безопасного доступа к данным.

---

## Что важно понять

- Объект и массив — это не просто контейнеры, а разные модели данных под разные задачи.
- Самая частая проблема на этом уровне — не «не знаю метод», а случайно меняю структуру там, где ожидалась новая копия.
- В JavaScript очень важно не только "уметь итерироваться", а понимать, по чему именно ты сейчас идёшь: по индексам, по собственным ключам объекта, по значениям iterable-структуры или по унаследованным свойствам.
- Без этой базы массивные методы, классы, прототипы и async-код будут ощущаться как хаос поверх хаоса.

---

## Термины по-человечески

- `Object literal` — способ создать объект прямо в коде через `{}`.
- `Mutation` — изменение существующего объекта или массива.
- `Copy` — новая структура данных, которая не делит текущее значение по ссылке так же, как исходник.
- `Enumeration` — обход свойств объекта.

---

## Базовый пример

```js
const user = {
  id: 1,
  profile: {
    city: 'Moscow',
  },
};

const city = user.profile?.city ?? 'Unknown';

const list = ['a', 'b'];
const next = [...list, 'c'];
```

---

## Объект и массив итерируются по-разному

Вот здесь у новичков часто и начинается путаница.

Массив обычно нужен, когда:

- важен порядок элементов;
- есть индексы;
- мы хотим пройтись по значениям;
- к данным хорошо подходят `map`, `filter`, `find`, `for...of`.

Объект обычно нужен, когда:

- есть именованные поля;
- важен доступ по ключу;
- это запись с полями, а не список однотипных элементов;
- нам нужно получить `keys`, `values` или `entries`.

Полезная грубая формула:

- массив отвечает на вопрос "какие элементы и в каком порядке";
- объект отвечает на вопрос "какие поля есть у сущности".

Если начать обходить объект как массив или массив как произвольный словарь, код быстро становится неочевидным.

---

## `for...in` и `Object.keys()` делают похожие вещи, но это не одно и то же

На вид оба варианта помогают получить ключи. Но смысл у них разный.

```js
const profile = {
  name: 'Ada',
  city: 'London',
};

for (const key in profile) {
  console.log('for...in', key);
}

for (const key of Object.keys(profile)) {
  console.log('Object.keys', key);
}
```

Почему это не одно и то же:

- `for...in` обходит перечислимые свойства объекта и может затронуть свойства из prototype chain.
- `Object.keys()` возвращает массив собственных ключей объекта, и дальше ты уже работаешь с обычным массивом.

Практическое правило для прикладного кода:

- если нужен предсказуемый обход plain object, почти всегда безопаснее `Object.keys`, `Object.values` или `Object.entries`;
- `for...in` держи в голове как более низкоуровневый и более опасный инструмент, особенно если объект не полностью под твоим контролем.

То есть:

- `for...in` отвечает на вопрос "какие перечислимые свойства можно обойти у этого объекта";
- `Object.keys()` отвечает на вопрос "какие собственные ключи есть у этого plain object прямо сейчас".

---

## `Object.keys`, `Object.values`, `Object.entries`: что выбирать

```js
const settings = {
  theme: 'dark',
  locale: 'ru',
};
```

Если нужны только ключи:

```js
Object.keys(settings); // ['theme', 'locale']
```

Если нужны только значения:

```js
Object.values(settings); // ['dark', 'ru']
```

Если нужны и ключ, и значение:

```js
Object.entries(settings);
// [['theme', 'dark'], ['locale', 'ru']]
```

Почему `entries` часто удобнее:

- оно сразу даёт пару `[key, value]`;
- результатом можно пользоваться как обычным массивом;
- легче писать читаемые преобразования.

```js
for (const [key, value] of Object.entries(settings)) {
  console.log(key, value);
}
```

---

## `for...of` и объект по умолчанию не дружат

Это ещё одна частая ловушка.

`for...of` работает по iterable-значениям:

- массивам;
- строкам;
- `Map`;
- `Set`;
- другим структурам с iterator protocol.

Plain object по умолчанию не iterable.

То есть вот так нельзя:

```js
const user = { name: 'Ada', city: 'London' };

for (const item of user) {
  console.log(item);
}
```

Зато можно так:

```js
for (const [key, value] of Object.entries(user)) {
  console.log(key, value);
}
```

Полезная мысль:

- `for...of` идёт по значениям iterable;
- `for...in` идёт по ключам/именам свойств объекта;
- `Object.entries()` часто оказывается мостом между plain object и удобной итерацией через `for...of`.

---

## Частые ошибки

- Путают изменение массива с созданием новой версии.
- Используют `splice`, когда нужен неразрушающий вариант.
- Пишут сложную логику чтения данных без `?.` и `??`.
- Выбирают объект там, где реально нужен массив, и наоборот.
- Используют `for...in` там, где нужен предсказуемый обход собственных полей объекта.
- Пытаются пройтись `for...of` по plain object, не понимая, что объект сам по себе не iterable.

---

## Anti-pattern

"Если данные уже в объекте или массиве, значит дальше разберемся по ходу".

Неясная модель данных быстро ломает и API функции, и читаемость условий, и преобразования.

---

## 🔧 Практика

1. Возьми объект с вложенными данными и безопасно прочитай 2-3 уровня через `?.` и `??`.
1. На одном массиве покажи разницу между `slice` и `splice`.
1. Для одного plain object сравни `for...in`, `Object.keys`, `Object.values` и `Object.entries`.
1. Напиши функцию, которая не мутирует входной массив, а возвращает новую версию.
