## Термины урока

- `UI framework` — это система, которая помогает описывать интерфейс как модель данных, состояний и реакций на события.
- `Declarative UI` — это подход, где разработчик описывает желаемое состояние интерфейса, а не вручную переставляет DOM-узлы по одному.
- `Component tree` — это дерево компонентов или template-узлов, из которого собирается экран.
- `State-driven rendering` — это модель, где изменение состояния приводит к пересчету UI.
- `Props` / `inputs` — это внешние входы компонента.
- `Local state` — это данные, которые принадлежат конкретному UI-сценарию или экземпляру компонента.
- `Virtual DOM` — это внутренняя структура, с которой React сравнивает следующий снимок UI перед применением изменений.
- `Fiber` — это внутренняя модель единиц работы и планирования в React, которая позволяет разбивать обновление на шаги и управлять приоритетами.
- `Reconciliation` — это процесс сравнения старого и нового дерева элементов, чтобы понять, что реально нужно обновить.

## Реализация в React

В первой итерации раздела React остается основной реализацией, поэтому именно на нем проще собрать базу.

### JSX как форма component tree

`JSX` полезен не потому, что "похож на HTML", а потому что позволяет описывать дерево интерфейса прямо рядом с логикой компонента.

### `props`, `state`, `children`

- `props` отвечают за входы извне;
- `state` отвечает за локальное состояние конкретного экземпляра;
- `children` отвечают за вложенную композицию.

```tsx
type ProductCardProps = {
  title: string;
  price: number;
  children?: React.ReactNode;
};

function ProductCard({ title, price, children }: ProductCardProps) {
  return (
    <article className="card">
      <h2>{title}</h2>
      <p>{price} ₽</p>
      {children}
    </article>
  );
}
```

### `key`

`key` нужен React, чтобы понимать идентичность элементов в списке. Хороший `key` стабилен и привязан к сущности. Плохой `key` дает потерю фокуса, странные перерисовки и "живущие своей жизнью" строки списка.

### Virtual DOM, Fiber и reconciliation

На уровне mental model React полезно понимать три вещи:

- component function возвращает описание следующего дерева UI;
- React сравнивает это дерево с предыдущим через reconciliation;
- применять обновления и планировать их помогает Fiber-модель.

`Virtual DOM` не нужно мистифицировать. Это не "магический быстрый DOM", а промежуточное представление UI, с которым Reactу удобнее сравнивать старое и новое состояние дерева.

`Fiber` нужен не для того, чтобы разработчик думал внутренностями рантайма каждый день. Но без базового понимания Fiber сложно объяснить:

- почему React может прерывать и переупорядочивать часть работы;
- почему update имеет стоимость не только в DOM, но и в самом React runtime;
- откуда берутся темы вроде приоритетов, transitions и deferred updates.

Практический вывод простой: структура дерева, стабильные `key` и разумные boundaries влияют не только на "читаемость", но и на то, сколько работы React вообще должен сделать.

### Что еще важно в React-механике

На этой ступени достаточно видеть карту тем:

- `React.lazy` и `Suspense` задают границы отложенной загрузки и fallback UI;
- `memo`, `useMemo` и `useCallback` — это частные инструменты оптимизации, а не базовый способ проектировать компонент;
- transitions и deferred updates нужны, когда React должен различать срочные и несрочные обновления.

Дальше эти темы уже разбираются отдельно через routing, server state и оптимизацию, а не сваливаются в один вводный урок.

### Controlled и uncontrolled

Это уже не "два способа написать input", а два разных контракта:

- controlled-поле управляется через React state;
- uncontrolled-поле хранит текущее значение в DOM, а React считывает его по мере необходимости.

Практическое правило простое: если значение нужно синхронизировать с UI и логикой вокруг, controlled-подход обычно честнее. Если форма простая и библиотека умеет работать через `ref`, uncontrolled может быть дешевле.

### Reconciliation в React-терминах

React не "перерисовывает все целиком", а пересчитывает дерево и решает, какие изменения реально применить. Поэтому стабильная структура дерева и корректные `key` — не мелочь, а часть контракта компонента.

## Частые ошибки

- думают, что UI framework нужен только для "красивого JSX";
- смешивают входы компонента, локальное состояние и derived values в одну кучу;
- заводят `state` там, где достаточно `props`;
- используют индекс массива как `key` в динамическом списке;
- учат React как набор API и не видят общей компонентной модели.

---

## Anti-pattern

"UI framework = библиотека с компонентами и хуками".

Нет. Framework или framework-like UI system нужен для управления изменяющимся интерфейсом: структурой дерева, состоянием, обновлениями и границами ответственности.

## 🔧 Практика

1. Возьми любой экран и разложи его на component tree: shell, секции, карточки, формы, вложенные части.
1. Для трех компонентов на этом экране отдельно назови `props`, local state и то, что вообще не должно быть state.
1. Найди на экране одно место, где в React подошел бы `children`, а во Vue это было бы ближе к `slot`.
1. Объясни своими словами разницу между `Virtual DOM`, reconciliation и DOM update.
1. Объясни, почему стабильный `key` важен не только для производительности, но и для корректности поведения.
