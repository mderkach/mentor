# 6.1 Компонентная модель UI framework-ов

Этот урок не про то, "как писать React". Он про то, зачем вообще UI framework нужен современному интерфейсу, почему он появился как ответ на боль imperative DOM-кода и почему почти все такие системы в итоге приходят к компонентам, дереву интерфейса и управлению состоянием.

Если эту базу не собрать, дальше React, Vue или любой другой framework будут ощущаться как набор случайных API.

---

## Что уже должно быть знакомо

- базовый HTML, CSS и JavaScript для чтения UI-компонентов и их состояния.

---

## 🎯 Что ты освоишь

- что такое UI framework как модель управления интерфейсом;
- зачем нужны декларативность, component tree и state-driven rendering;

---

## Термины урока

- `UI framework` — это система, которая помогает описывать интерфейс как модель данных, состояний и реакций на события.
- `Declarative UI` — это подход, где разработчик описывает желаемое состояние интерфейса, а не вручную переставляет DOM-узлы по одному.
- `Component tree` — это дерево компонентов или template-узлов, из которого собирается экран.
- `State-driven rendering` — это модель, где изменение состояния приводит к пересчету UI.
- `Props` / `inputs` — это внешние входы компонента.
- `Local state` — это данные, которые принадлежат конкретному UI-сценарию или экземпляру компонента.

---

## Core / Fundamentals

### Почему UI framework вообще появился

UI framework решает одну и ту же базовую проблему: интерфейс постоянно меняется под действием данных, событий, навигации и асинхронности. Если делать это руками через imperative DOM API, код быстро превращается в кашу из "найди элемент", "обнови текст", "скрой блок", "не забудь синхронизировать это еще в трех местах".

Исторически это и есть причина появления framework-ов:

- интерфейс перестал быть статичной HTML-страницей;
- состояние и сеть начали постоянно менять экран;
- ручная синхронизация DOM, данных и событий перестала масштабироваться;
- команде понадобились явные границы между частями интерфейса.

UI framework в этом контексте нужен не ради "удобного синтаксиса", а ради четырех вещей:

- описывать экран как следствие данных и состояния;
- собирать интерфейс из управляемых границ;
- повторно использовать не только разметку, но и поведение;
- централизовать модель обновления UI, а не держать ее размазанной по обработчикам и DOM-манипуляциям.

Поэтому современные framework-ы почти всегда приходят к общим идеям:

- экран собирается из повторно используемых частей;
- у каждой части есть входы и границы ответственности;
- интерфейс рендерится из состояния, а не редактируется вручную по кускам;
- дерево UI описывается структурно, а не через россыпь `querySelector` и `appendChild`.

Компонент нужен не ради моды. Он одновременно решает четыре задачи:

- композиция: сложный экран собирается из небольших частей;
- изоляция: локальная логика не растекается по всему приложению;
- повторное использование: можно переиспользовать не только разметку, но и поведение;
- локализация ответственности: понятно, где живут входы, локальное состояние и обработчики.

## Сравнение моделей

На базовом уровне у React, Vue, Angular и Svelte гораздо больше общего, чем кажется:

- у всех есть компонентная единица или template boundary;
- у всех есть входы компонента;
- у всех есть способ хранить локальное состояние;
- у всех есть механизм обновления UI после изменения состояния.

Полезные пары для сравнения:

- `component tree` в JSX и `template tree` в template-first framework-ах — это разные формы одной идеи: UI строится как дерево;
- `props` в React/Vue и `inputs` в Angular — это входы компонента;
- `children` в React и `slots` во Vue/Svelte — это способы управлять вложенной композицией;
- local state в React и reactive state в Vue/Solid — это ответы на один и тот же вопрос: где живет изменяемое значение и как UI узнает о его изменении.

Главная разница дальше не в том, что "у одного props, а у другого не props", а в mental model:

- React чаще заставляет думать через render function и явные зависимости;
- Vue часто ощущается как template-first и reactive-first;
- Angular сильнее задает структурную модель приложения через DI, templates и app architecture;
- Svelte многое решает на этапе компиляции, а не только в runtime.

---

## Другие подходы кратко

- `Angular` сильнее навязывает структурную модель приложения через `inputs`, `outputs`, DI и templates. Это тяжелее на входе, но полезно для больших enterprise-систем.
- `Svelte` переносит часть реактивности в compile step, поэтому код ощущается проще, но mental model вокруг компиляции и runtime другая.
- `Solid` делает ставку на fine-grained reactivity: обновляется не все поддерево, а конкретные зависимые узлы.

---

## Частые ошибки

- думают, что UI framework нужен только для "красивого JSX";
- смешивают входы компонента, локальное состояние и derived values в одну кучу;
- заводят `state` там, где достаточно `props`;
- используют индекс массива как `key` в динамическом списке;
- учат React как набор API и не видят общей компонентной модели.

---

## Anti-pattern

"UI framework = библиотека с компонентами и хуками".

Нет. Framework или framework-like UI system нужен для управления изменяющимся интерфейсом: структурой дерева, состоянием, обновлениями и границами ответственности.

---

## 🔧 Практика

1. Возьми любой экран и разложи его на component tree: shell, секции, карточки, формы, вложенные части.
1. Для трех компонентов на этом экране отдельно назови `props`, local state и то, что вообще не должно быть state.
1. Найди на экране одно место, где в React подошел бы `children`, а во Vue это было бы ближе к `slot`.
1. Объясни, почему стабильный `key` важен не только для производительности, но и для корректности поведения.
