# 6.8 Cost model UI framework-ов

Этот урок остается React-heavy, но важно не путать две вещи: оптимизация UI framework-а как общей системы и оптимизация конкретного React-кода. `memo` и `useCallback` — это не стратегия сами по себе, а частные рычаги внутри более общей cost model.

Senior-уровень здесь в том, чтобы сначала измерить, потом выбрать самый дешевый и устойчивый рычаг.

## 🎯 Что ты освоишь

- чем profiling важнее догадок на глаз;

---

## Термины урока

- `Profiling` — это измерение причин, а не догадки.
- `Render cost` — это цена вычисления UI-дерева.
- `Commit cost` — это цена применения изменений к DOM.
- `Perceived responsiveness` — это то, как быстро и стабильно интерфейс ощущается пользователю.
- `Deferred UI` — это стратегия, где менее срочные обновления могут догонять более срочные.
- `Virtualization` — это рендер только видимой части большого списка.

---

## Core / Fundamentals

У UI framework-а есть несколько основных зон стоимости:

- частота обновлений;
- объем работы на update;
- размер DOM/UI tree;
- network/data pressure;
- browser work: layout, paint, compositing;
- perceived responsiveness, даже если "все технически работает".

Поэтому "UI тормозит" может означать разное:

- плохо лежит состояние;
- слишком велико поддерево обновления;
- тяжелый список без virtualization;
- плохой network/data flow;
- неудачный code split или hydration;
- layout/paint bottleneck, а не React bottleneck.

Измерение прежде оптимизации — не банальность, а единственный способ не лечить не ту причину.

---

## Сравнение моделей

Сравнивать полезно так:

- coarse-grained component updates vs fine-grained reactivity;
- runtime reactivity vs compile-time reactivity;
- framework cost vs browser cost.

React чаще платит ценой компонентных rerender boundaries и explicit memoization. Vue и Solid могут выигрывать на другой модели реактивности. Svelte может выигрывать на compile-time решениях. Но никакая модель не отменяет:

- плохой list rendering;
- тяжёлый DOM;
- network waterfall;
- плохие state boundaries.

---

## Другие подходы кратко

- `Solid` показывает, как fine-grained reactivity меняет цену update и подписок.
- `Svelte` показывает, что compile-time reactivity может упростить часть runtime costs.
- `SSR/hydration/islands/streaming` в разных framework-ах — это разные стратегии управления initial render cost и perceived responsiveness, а не только "React server components story".

---

## Частые ошибки

- начинают "оптимизировать" без profiler и воспроизводимого сценария;
- лечат плохие state boundaries через blanket memoization;
- оборачивают дешевые вычисления в `useMemo` просто ради солидности;
- рендерят огромные списки без virtualization;
- не различают медленный render, медленный network и медленный browser paint.

---

## Anti-pattern

"Оптимизация UI = везде поставить `memo` и `useCallback`".

Это не стратегия. Это попытка замаскировать архитектурную проблему локальным трюком.

---

## 🔧 Практика

1. Возьми один действительно медленный экран и зафиксируй сценарий профилирования.
1. Разложи проблему по слоям: state boundary, list size, network/data flow, route transition, browser paint.
1. Выбери один самый дешевый рычаг и измерь эффект до и после.
1. Сравни, как бы этот сценарий выглядел в framework-е с fine-grained reactivity.
