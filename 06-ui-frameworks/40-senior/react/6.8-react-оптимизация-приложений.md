## Термины урока

- `Profiling` — это измерение причин, а не догадки.
- `Render cost` — это цена вычисления UI-дерева.
- `Commit cost` — это цена применения изменений к DOM.
- `Perceived responsiveness` — это то, как быстро и стабильно интерфейс ощущается пользователю.
- `Deferred UI` — это стратегия, где менее срочные обновления могут догонять более срочные.
- `Virtualization` — это рендер только видимой части большого списка.
- `React.lazy` — это способ отложить загрузку части кода до момента, когда она реально понадобится.
- `Suspense` — это boundary, через который React показывает fallback, пока код или данные еще не готовы.

---

## Реализация в React

Самые дешевые и полезные React-рычаги обычно такие:

- state colocation и честные boundaries;
- стабильные `key` и нормальная структура списков;
- server-state стратегия без лишних запросов и спиннеров;
- `React.lazy` и `Suspense` там, где code split действительно разгружает initial path;
- точечная memoization там, где есть измеримый выигрыш;
- virtualization для больших списков;
- `useDeferredValue` или `startTransition`, когда важно разделить срочные и несрочные обновления.

```tsx
function ProductCatalog({ products }: { products: Product[] }) {
  const [query, setQuery] = useState("");
  const deferredQuery = useDeferredValue(query);

  const filteredProducts = products.filter((product) =>
    product.title.toLowerCase().includes(deferredQuery.toLowerCase())
  );

  return (
    <>
      <input value={query} onChange={(event) => setQuery(event.target.value)} />
      <ul>
        {filteredProducts.map((product) => (
          <li key={product.id}>{product.title}</li>
        ))}
      </ul>
    </>
  );
}
```

Практический порядок почти всегда такой:

1. Профилируй.
1. Чини state boundaries и data flow.
1. Чини list rendering.
1. Проверь, нужен ли route-level или component-level code split через `React.lazy` и `Suspense`.
1. Только потом думай про `memo`, `useMemo`, `useCallback`.

---

## Частые ошибки

- начинают "оптимизировать" без profiler и воспроизводимого сценария;
- лечат плохие state boundaries через blanket memoization;
- оборачивают дешевые вычисления в `useMemo` просто ради солидности;
- ставят `Suspense` и `lazy` без понимания, где это реально разгружает critical path;
- рендерят огромные списки без virtualization;
- не различают медленный render, медленный network и медленный browser paint.

---

## Anti-pattern

"Оптимизация UI = везде поставить `memo` и `useCallback`".

Это не стратегия. Это попытка замаскировать архитектурную проблему локальным трюком.

---

## 🔧 Практика

1. Возьми один действительно медленный экран и зафиксируй сценарий профилирования.
1. Разложи проблему по слоям: state boundary, list size, network/data flow, route transition, browser paint.
1. Выбери один самый дешевый рычаг и измерь эффект до и после.
1. Сравни, как бы этот сценарий выглядел в framework-е с fine-grained reactivity.
