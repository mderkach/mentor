## Термины урока

- `Server state` — это данные, источник истины для которых находится на сервере.
- `Stale data` — это данные, которые клиент показывает как актуальные, хотя сервер уже мог изменить их вне текущего UI.
- `Invalidation` — это пометка кэша как устаревшего с необходимостью обновить его.
- `Background refetch` — это повторная загрузка данных без полного блокирования интерфейса.
- `Optimistic update` — это временное обновление UI до ответа сервера.
- `Route-level loading` — это загрузка данных на уровне маршрута или meta-framework boundary, а не внутри конкретного виджета.

## Реализация в React

В React проблема быстро проявляется через старый паттерн `fetch in useEffect`.

```tsx
useEffect(() => {
  let cancelled = false;

  async function load() {
    const data = await fetchProducts(categoryId);
    if (!cancelled) {
      setProducts(data);
    }
  }

  void load();

  return () => {
    cancelled = true;
  };
}, [categoryId]);
```

На маленьком экране это еще терпимо. Дальше начинаются вопросы:

- где кэш;
- кто дедуплицирует одинаковые запросы;
- как происходит invalidation после мутации;
- как связать loading/error с маршрутом;
- как делать prefetch и optimistic update.

Именно поэтому React-экосистема обычно приходит к `TanStack Query`, `RTK Query` или к routing/model решениям с data-aware boundaries.

```tsx
const productsQuery = useQuery({
  queryKey: ["products", categoryId],
  queryFn: () => fetchProducts(categoryId),
  staleTime: 30_000,
});
```

Что здесь важно:

- `queryKey` — это API вашего кэша;
- invalidation должна отражать бизнес-смысл, а не "рефетчить все";
- `Suspense`-oriented thinking полезен только там, где у команды уже есть понятные loading boundaries.

---

## Частые ошибки

- складывают server state в обычный client store "для надежности";
- используют `useEffect` как единственную модель запросов;
- не различают stale, loading, refetching и empty states;
- делают optimistic UI без понятного rollback;
- проектируют query keys как случайный набор аргументов без доменной структуры.

---

## Anti-pattern

"Запросы — это просто асинхронный local state компонента".

Нет. Как только данные живут на сервере, локальная модель перестает быть достаточной.

---

## 🔧 Практика

1. Возьми один экран и разложи его состояние на client state, server state и URL state.
1. Для одного запроса сформулируй хороший `queryKey` и объясни, как его инвалидировать после мутации.
1. Найди место, где component-level loading уже мешает, и опиши route-level alternative.
1. Придумай один optimistic update и сразу пропиши rollback-сценарий.
