## Термины урока

- `Composition` — это сборка интерфейса из компонентов с ясными ролями и границами.
- `Component boundary` — это место, где заканчивается одна ответственность и начинается другая.
- `State colocation` — это принцип держать состояние как можно ближе к месту использования.
- `Headless pattern` — это API поведения без жестко зашитого UI.
- `Compound component` — это набор связанных частей с общим внутренним контрактом.
- `Controlled composition` — это модель, где потребитель компонента явно управляет содержимым и состоянием через API, а не через скрытую магию.

## Реализация в React

### `children` как базовая композиция

`children` хороши там, где компонент — это container, layout-layer или boundary, а не владелец деталей вложенного UI.

### Headless и compound patterns

Эти паттерны полезны, когда у UI есть набор связанных частей, но ты не хочешь жестко привязать потребителя к одной разметке.

```tsx
function Tabs({ children }: { children: React.ReactNode }) {
  return <div className="tabs">{children}</div>;
}

Tabs.List = function TabsList({ children }: { children: React.ReactNode }) {
  return <div role="tablist">{children}</div>;
};

Tabs.Panel = function TabsPanel({ children }: { children: React.ReactNode }) {
  return <section>{children}</section>;
};
```

### Context splitting

`context` полезен, когда у нескольких связанных частей есть общий contract. Он вреден, когда превращается в "store на все случаи жизни". Если значения меняются с разной частотой и нужны разным поддеревьям, split context почти всегда лучше одного giant context.

### Portals, error boundaries, `Suspense`, `lazy`

Это не россыпь React-фич, а разные типы архитектурных boundaries:

- `portal` выносит слой UI в другую DOM-зону;
- `error boundary` локализует ошибку;
- `Suspense` и `lazy` управляют loading/code-splitting boundaries.

Именно здесь middle уже должен думать не только о JSX, но и о том, где у UI границы ошибок, загрузки и гидрации.

---

## Частые ошибки

- делают "универсальный" компонент с десятками флагов вместо осмысленной композиции;
- поднимают состояние слишком высоко до появления реальной причины;
- заменяют design boundary на giant `context`;
- выбирают паттерн ради моды, а не ради прозрачного API;
- забывают, что `portal`, `error boundary` и `Suspense` — это именно boundaries, а не декоративные фичи.

---

## Anti-pattern

"Надо сразу сделать максимально абстрактный reusable component, чтобы потом все переиспользовать".

Обычно это значит: компонент еще не понял свою ответственность, а команда уже пытается выжать из него design system.

---

## 🔧 Практика

1. Возьми один сложный экран и разложи его на boundaries: shell, feature shell, presentational blocks, overlay, error/loading boundaries.
1. Для одного reusable блока предложи две формы API: через `children` и через slot-like composition, и сравни их.
1. Найди место, где state можно опустить ближе к потребителю без потери читаемости.
1. Определи, где в этом экране `context` реально оправдан, а где он просто скрывает отсутствие архитектурного решения.
