# 6.6 Server state и data fetching

Как только интерфейс начинает жить не на моках, выясняется неприятная правда: запросы — это не просто `fetch` плюс `setState`. Появляются stale data, invalidation, гонки, background refetch, optimistic updates и связка с routing.

Именно здесь и начинается тема server state как отдельного типа проблемы.

## 🎯 Что ты освоишь

- чем server state отличается от client state;
- почему stale data, invalidation и race conditions нельзя игнорировать;
- как выбирать между component-level и route-level data loading;

---

## Термины урока

- `Server state` — это данные, источник истины для которых находится на сервере.
- `Stale data` — это данные, которые клиент показывает как актуальные, хотя сервер уже мог изменить их вне текущего UI.
- `Invalidation` — это пометка кэша как устаревшего с необходимостью обновить его.
- `Background refetch` — это повторная загрузка данных без полного блокирования интерфейса.
- `Optimistic update` — это временное обновление UI до ответа сервера.
- `Route-level loading` — это загрузка данных на уровне маршрута или meta-framework boundary, а не внутри конкретного виджета.

---

## Core / Fundamentals

Главное различие очень простое: server state не принадлежит вашему UI.

Из этого следуют все последствия:

- данные могут устареть без участия пользователя;
- один и тот же ресурс может понадобиться в нескольких местах;
- нужна стратегия кэширования, dedupe и refetch;
- мутации должны уметь обновлять или инвалидировать кэш;
- ошибки и loading states нельзя сводить к одному `isLoading`.

Полезная базовая классификация:

- `client state` принадлежит интерфейсу;
- `server state` принадлежит серверу и только кэшируется клиентом;
- `route-level data` принадлежит навигационной архитектуре;
- `component-level data` оправдана, если локальна и не влияет на соседние boundaries.

---

## Сравнение моделей

Data fetching можно организовывать по-разному:

- query library в SPA;
- framework-native data model в meta-framework;
- route-level loading;
- server-oriented model, где часть fetching вообще не живет в клиентском компоненте.

Важные сравнения:

- component-level loading проще стартует, но хуже масштабируется;
- route-level loading лучше удерживает navigation/data contract;
- SPA query libraries сильны там, где нужно управлять клиентским кэшем;
- meta-framework data model сильнее там, где routing, SSR и server/client boundaries уже часть архитектуры.

---

## Другие подходы кратко

- `Angular` часто решает часть проблемы через services, resolvers и более явное разделение data access слоя.
- `Remix` и `Next` двигают fetching ближе к маршруту и серверной boundary.
- `GraphQL ecosystems` вроде Apollo добавляют свой кэш и свой mental model вокруг query/mutation normalization.

---

## Частые ошибки

- складывают server state в обычный client store "для надежности";
- используют `useEffect` как единственную модель запросов;
- не различают stale, loading, refetching и empty states;
- делают optimistic UI без понятного rollback;
- проектируют query keys как случайный набор аргументов без доменной структуры.

---

## Anti-pattern

"Запросы — это просто асинхронный local state компонента".

Нет. Как только данные живут на сервере, локальная модель перестает быть достаточной.

---

## 🔧 Практика

1. Возьми один экран и разложи его состояние на client state, server state и URL state.
1. Для одного запроса сформулируй хороший `queryKey` и объясни, как его инвалидировать после мутации.
1. Найди место, где component-level loading уже мешает, и опиши route-level alternative.
1. Придумай один optimistic update и сразу пропиши rollback-сценарий.
