# 6.7 Модели управления состоянием

Тема state management ломается не на выборе библиотеки, а раньше: команда не договорилась, какие типы состояния вообще существуют и где каждому из них место.

Если этого не сделать, в store уезжает все подряд: форма, filters, API-данные, URL state, временные флаги и половина бизнес-логики.

## 🎯 Что ты освоишь

- какие типы состояния есть в UI-приложении;
- когда store действительно нужен, а когда нет;
- чем отличаются event-driven stores, reactive stores и state machines;

---

## Термины урока

- `Local UI state` — это состояние конкретного компонента или близкого поддерева.
- `Shared UI state` — это состояние, которое нужно нескольким связанным частям UI.
- `URL state` — это навигационное состояние, которое должно быть shareable через ссылку.
- `Store` — это отдельный контейнер состояния, живущий дольше одного компонента.
- `Selector` — это функция, которая выбирает и вычисляет нужный кусок состояния.
- `State machine` — это модель, где возможные состояния и переходы между ними описаны явно.

---

## Core / Fundamentals

Полезная классификация состояния:

- local UI state;
- shared UI state;
- URL state;
- server state;
- ephemeral state вроде hover, temporary draft и transient animation flags.

Первый вопрос всегда один: нужен ли store вообще.

Сначала проверь более дешевые варианты:

- local state;
- lifted state;
- component boundary + context;
- URL;
- server-state cache для API-данных.

Store нужен, когда:

- состояние реально разделяется между удаленными частями дерева;
- у клиента есть отдельная предметная модель;
- важны явные переходы, tooling и прозрачность;
- простая комбинация local state + URL + server state уже не держит архитектуру.

---

## Сравнение моделей

Есть несколько осей сравнения:

- centralized store vs distributed state;
- event-driven store vs reactive graph;
- mutable model vs immutable update pipeline;
- selectors/subscriptions vs fine-grained dependency tracking;
- state machine vs набор флагов.

Практическая польза сравнения:

- `Flux/store thinking` хорошо дает прозрачные события и дебаг;
- reactive graph model удобна там, где вычисляемые связи и реактивные зависимости важнее action log;
- state machine полезна, когда UI-сценарий имеет жесткие состояния и переходы, а флаги уже начинают врать.

---

## Другие подходы кратко

- `Angular` все чаще двигается в сторону signals/services и более структурной app-model интеграции.
- `Svelte` дает stores как встроенную модель и проще подводит к distributed reactive state.
- `Solid` строит state вокруг signals и fine-grained subscriptions, поэтому цена грубых подписок и giant store там видна иначе.

---

## Частые ошибки

- заводят store "на старте, потому что взрослый проект";
- смешивают client state, URL state и server state в один контейнер;
- подписываются на весь store вместо адресных selectors/subscriptions;
- дублируют derived values вместо вычисления из source of truth;
- пытаются починить хаотичный workflow еще одним флагом вместо state machine.

---

## Anti-pattern

"Если состояние сложное, его надо вынести в store".

Не всегда. Иногда проблема не в размере состояния, а в том, что команда не развела его по типам и границам.

---

## 🔧 Практика

1. Возьми один сложный экран и классифицируй его состояние по типам: local, shared UI, URL, server, ephemeral.
1. Для каждого типа объясни, почему он должен жить именно там, а не в store по умолчанию.
1. Выбери между `Context`, `Redux Toolkit`, `Zustand`, `MobX` и state machine для одного сценария и аргументируй выбор.
1. Найди одно место, где selectors или normalized state уберут лишнюю связанность.
