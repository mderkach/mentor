# 6.5 Композиция компонентов и архитектурные паттерны

На middle-уровне компонентная разработка перестает быть задачей "распилить страницу на куски". Главный вопрос уже другой: как собрать UI из компонентов так, чтобы границы ответственности, состояние и расширяемость не сгнили через месяц.

Плохая композиция дает мутантов с десятками флагов. Хорошая дает ясные API и управляемую цену изменений.

## Термины урока

- `Composition` — это сборка интерфейса из компонентов с ясными ролями и границами.
- `Component boundary` — это место, где заканчивается одна ответственность и начинается другая.
- `State colocation` — это принцип держать состояние как можно ближе к месту использования.
- `Headless pattern` — это API поведения без жестко зашитого UI.
- `Compound component` — это набор связанных частей с общим внутренним контрактом.
- `Controlled composition` — это модель, где потребитель компонента явно управляет содержимым и состоянием через API, а не через скрытую магию.

---

## Core / Fundamentals

Композиция — это не "прием из React". Это общий способ собирать сложный UI в любом framework-е.

Полезные вопросы:

- где действительно должна лежать ответственность компонента;
- что должно быть на уровне presentation, а что на уровне feature shell;
- где состояние локально, а где уже просится в общий контекст или store;
- какие части API компонента должны быть явными для потребителя.

Два универсальных правила:

- component boundary полезна, если уменьшает связанность, а не просто добавляет еще один файл;
- state colocation почти всегда дешевле и честнее, чем ранняя глобализация состояния.

Container/presentational split все еще полезен как мыслительный прием, но не как догма "всегда два файла на каждый блок". Смысл не в названии, а в том, чтобы поведение, доступ к данным и presentation не смешивались без причины.

---

## Сравнение моделей

Сравнивать полезно не названия паттернов, а способы композиции:

- `children` в React и `slots` во Vue/Svelte решают одну задачу: дать внешнему потребителю контролировать вложенный UI;
- `context` в React и `provide/inject` во Vue решают одну задачу: передать общие данные через дерево без проп-drilling;
- hooks composition и composables решают одну задачу: переиспользовать поведение, а не только разметку;
- `render props` и `HOC` полезно знать как исторический контекст, но сегодня их редко выбирают первыми, если тот же сценарий понятнее выражается через hooks, headless API или slots.

Еще одно важное различие:

- React часто тяготеет к JSX-driven composition;
- Vue и Angular сильнее опираются на template-driven composition;
- это не делает одну модель "лучше", но меняет способ проектирования component API.

---

## Другие подходы кратко

- `Angular` дает структурную композицию через templates, content projection, DI и более жесткую app architecture.
- `Svelte` делает компонентный API ближе к шаблону и компилятору, а не только к runtime abstraction.
- `Solid` усиливает идею fine-grained dependencies: цена слишком широких boundaries и подписок там чувствуется иначе.

---

## Частые ошибки

- делают "универсальный" компонент с десятками флагов вместо осмысленной композиции;
- поднимают состояние слишком высоко до появления реальной причины;
- заменяют design boundary на giant `context`;
- выбирают паттерн ради моды, а не ради прозрачного API;
- забывают, что `portal`, `error boundary` и `Suspense` — это именно boundaries, а не декоративные фичи.

---

## Anti-pattern

"Надо сразу сделать максимально абстрактный reusable component, чтобы потом все переиспользовать".

Обычно это значит: компонент еще не понял свою ответственность, а команда уже пытается выжать из него design system.

---

## 🔧 Практика

1. Возьми один сложный экран и разложи его на boundaries: shell, feature shell, presentational blocks, overlay, error/loading boundaries.
1. Для одного reusable блока предложи две формы API: через `children` и через slot-like composition, и сравни их.
1. Найди место, где state можно опустить ближе к потребителю без потери читаемости.
1. Определи, где в этом экране `context` реально оправдан, а где он просто скрывает отсутствие архитектурного решения.
