## Термины урока

- `Route tree` — это дерево экранов и вложенных layout-границ.
- `Route param` — это идентификатор ресурса или сегмента пути, например `:productId`.
- `Search params` — это query-параметры URL, которые описывают shareable navigation state.
- `Nested layout` — это общий layout или shell для группы вложенных экранов.
- `Route-level data loading` — это загрузка данных на уровне маршрута, а не только внутри компонента.
- `Navigation policy` — это правило доступа, redirect или guard, привязанный к переходу.

---

## Реализация в React

В React эта модель обычно собирается через React Router или router-слой meta-framework-а.

Что важно уметь видеть:

- nested routes как отражение layout boundaries;
- `params` и `search params` как разные типы состояния;
- route-level loading/error boundaries;
- lazy routes и route-based split как естественную оптимизацию.

```tsx
import { Route, Routes, useParams, useSearchParams } from "react-router";

function ProductDetailsPage() {
  const { productId } = useParams();
  const [searchParams, setSearchParams] = useSearchParams();
  const activeTab = searchParams.get("tab") ?? "overview";

  return (
    <>
      <h1>Товар {productId}</h1>
      <button type="button" onClick={() => setSearchParams({ tab: "reviews" })}>
        Отзывы
      </button>
      {activeTab === "overview" && <ProductOverview />}
      {activeTab === "reviews" && <ProductReviews />}
    </>
  );
}

function AppRoutes() {
  return (
    <Routes>
      <Route path="/products/:productId" element={<ProductDetailsPage />} />
    </Routes>
  );
}
```

`React Router` особенно важен там, где router уже начинает управлять не только рендером, но и данными: loaders, actions, route-level error/loading flows и navigation race handling.

## Частые ошибки

- проектируют router как плоский список экранов без layout boundaries;
- складывают shareable navigation state в local state;
- тянут эфемерные UI-флаги в URL без реальной пользы;
- считают client-side guard полноценной security-моделью;
- делают route split только по названию, а не по смысловым boundaries.

---

## Anti-pattern

"Routing — это про ссылки и страницы, а архитектура живет где-то отдельно".

Нет. Как только приложение выходит за пределы трех экранов, routing становится одной из главных структурных осей UI.

---

## 🔧 Практика

1. Возьми один продуктовый сценарий и нарисуй route tree вместе с layout boundaries.
1. Для пяти состояний интерфейса определи, что из этого должно жить в route params, search params или local state.
1. Найди в текущем проекте одну навигационную политику и объясни, это UX guard или реальная авторизация.
1. Определи, где у вас route-level loading/error boundary уже нужна, а где пока можно жить проще.
