# 6.2 Update cycle и реактивность UI framework-ов

Этот урок не про то, куда еще вставить `useEffect`. Он про update cycle UI framework-а: когда происходит render, откуда берутся side effects, чем derived state отличается от эффекта и зачем вообще нужен механизм реактивных обновлений.

Если это не понимать, hooks превращаются в коллекцию ритуалов.

## 🎯 Что ты освоишь

- что такое render, commit, update cycle и side effect в UI framework-е;
- чем derived state отличается от side effect;
- чем explicit dependencies отличаются от implicit reactivity.

---

## Термины урока

- `Render` — это вычисление следующего UI на основе текущих входов и состояния.
- `Commit` — это применение рассчитанных изменений к DOM или другому target runtime.
- `Update cycle` — это полный путь от изменения данных до нового состояния интерфейса.
- `Derived state` — это значение, которое можно вычислить из уже существующих данных без отдельного хранения.
- `Side effect` — это синхронизация с внешней системой: сетью, DOM API, таймером, подпиской, storage.
- `Explicit dependencies` — это модель, где разработчик явно задает, от чего зависит вычисление или эффект.

---

## Core / Fundamentals

Любой UI framework должен решить один и тот же вопрос: как понять, что интерфейс пора обновить, и как отделить "чистый пересчет UI" от действий с внешним миром.

Полезная ментальная модель:

- render должен быть чистым вычислением;
- derived values лучше считать во время render, а не хранить отдельным state;
- side effects нужны только там, где есть внешний мир;
- update cycle не равен старому пониманию lifecycle вида `mounted -> updated -> unmounted`.

То есть правильный вопрос звучит не "в какой lifecycle hook это засунуть", а так:

- это просто вычисляется из текущих данных;
- это нужно переживать между обновлениями;
- это должно синхронизироваться с внешней системой;
- это повторяемая логика, которую лучше вынести.

---

## Сравнение моделей

У разных framework-ов здесь разные акценты:

- в React зависимости эффекта или memo часто задаются явно;
- во Vue и Solid часть зависимостей отслеживается реактивной системой автоматически;
- в Svelte часть реактивности переносится в compile-time;
- в Angular современная история уже выходит за старую lifecycle-терминологию и включает сигналы, шаблоны, DI и более широкий app model.

Особенно важно различать:

- explicit dependencies vs implicit tracking;
- computed value vs effect;
- render-time calculation vs asynchronous reaction to change.

Если это не различать, разработчик начинает использовать effect для всего: вычислений, синхронизации derived state и произвольной бизнес-логики. Именно отсюда и растет половина боли.

---

## Другие подходы кратко

- `Angular` исторически ассоциировался с lifecycle hooks, но современная картина шире: templates, DI, signals и change detection как часть общей модели.
- `Svelte` позволяет описывать реактивность ближе к обычному коду, потому что часть зависимости определяется при компиляции.
- `Solid` делает fine-grained reactivity центральной идеей: обновляется не все компонентное поддерево, а конкретные зависимые вычисления.

---

## Частые ошибки

- используют `useEffect` для вычисления derived state;
- не различают render-логику и side effects;
- забывают cleanup для подписок, таймеров и запросов;
- выносят в custom hook случайный набор строк, а не повторяемую поведенческую модель;
- учат hooks как API-список, а не как способ описать update cycle.

---

## Anti-pattern

"Любое изменение данных нужно отражать через `useEffect`".

Нет. Effect — это не универсальный клей. Если значения можно честно получить из текущего render-контекста, effect только усложнит модель.

---

## 🔧 Практика

1. Возьми компонент с несколькими hooks и для каждого вызова скажи: это local state, ref, effect, reducer или reusable behavior.
1. Найди в проекте один `useEffect`, который на самом деле можно заменить на derived value во время render.
1. Найди один effect, где реально нужен cleanup, и объясни, что сломается без него.
1. Сравни мысленно этот сценарий с Vue-подходом: где здесь был бы `computed`, а где `watch`.
