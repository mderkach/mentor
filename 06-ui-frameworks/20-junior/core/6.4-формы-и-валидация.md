# 6.4 Формы и валидация

Форма — это не просто набор `input`. Это отдельная UI-подсистема со своим lifecycle: значения, touched/dirty-state, submit, ошибки, server feedback, nested fields и пользовательское доверие.

Если форму мыслить как "пару `useState` и дальше разберемся", она быстро начинает ломать UX.

## Термины урока

- `Form state` — это значения полей, ошибки, touched/dirty-состояния и фаза submit.
- `Controlled field` — это поле, чье текущее значение хранится в framework state.
- `Uncontrolled field` — это поле, где текущее значение живет в DOM, а framework читает его по мере необходимости.
- `Schema validation` — это проверка данных по явной схеме.
- `Field array` — это повторяющийся набор полей вроде списка телефонов или адресов.
- `Server error mapping` — это привязка ошибок сервера к конкретным полям или общему состоянию формы.

---

## Core / Fundamentals

Формы быстро становятся отдельной state-проблемой, потому что у них есть свои виды состояния:

- current values;
- touched/dirty;
- submit lifecycle;
- sync validation;
- async validation;
- server errors;
- dynamic fields и nested structures.

Полезные вопросы до выбора библиотеки:

- форма маленькая или большая;
- нужно ли реагировать на каждый ввод;
- есть ли динамические поля;
- нужна ли schema validation;
- нужно ли маппить server-side ошибки на поля;
- насколько критичны лишние ререндеры.

Client validation при этом не отменяет server validation. Она улучшает UX, но не заменяет проверку контракта на сервере.

---

## Сравнение моделей

Сравнивать полезно так:

- controlled vs uncontrolled;
- HTML + немного логики vs form library;
- field-level validation vs schema-driven validation;
- клиентская проверка vs полный lifecycle формы вместе с сервером.

Практическое правило:

- если форма простая, иногда достаточно HTML и немного framework-кода;
- если форма имеет lifecycle, динамику и много полей, библиотека обычно окупается;
- schema-driven validation полезна там, где правила нужно централизовать и типизировать;
- server errors должны быть частью модели формы, а не внезапным `alert("error")`.

---

## Другие подходы кратко

- `Angular reactive forms` дают очень выраженную модель формы как отдельной структуры данных и validation graph.
- `Svelte` и `Nuxt` часто идут через более легкие form abstractions и экосистемные libraries без одной доминирующей модели.
- В любом framework-е форма остается отдельной UI-подсистемой, даже если API вокруг нее выглядит по-разному.

---

## Частые ошибки

- держат большое form state на десятках `useState`;
- дублируют validation rules в UI и schema без причины;
- используют client validation как замену server validation;
- тянут `Controller` туда, где обычный input уже работает;
- забывают про touched/dirty/submit lifecycle и начинают показывать ошибки слишком рано.

---

## Anti-pattern

"Форма — это обычный local state плюс пара `if` на submit".

На маленьком примере это прокатывает. На продуктовой форме быстро превращается в долговую яму.

---

## 🔧 Практика

1. Возьми одну форму и разложи ее на значения, touched/dirty, submit lifecycle, client validation и server errors.
1. Определи, где там controlled-state действительно нужен, а где можно жить проще.
1. Найди один сценарий, где без field array или нормальной schema validation форма станет хрупкой.
1. Придумай, как эта же задача выглядела бы во Vue через composables и form library.
