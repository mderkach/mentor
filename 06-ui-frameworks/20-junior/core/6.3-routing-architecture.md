# 6.3 Routing architecture

Routing — это не список URL и не библиотека для `<Link />`. Это архитектура навигации: дерево экранов, layout boundaries, URL-контракт, policies доступа и место, где очень часто сходятся данные, загрузка и ошибки.

Хороший router делает поведение приложения предсказуемым. Плохой router размазывает навигацию по компонентам и ломает data flow.

---

## Термины урока

- `Route tree` — это дерево экранов и вложенных layout-границ.
- `Route param` — это идентификатор ресурса или сегмента пути, например `:productId`.
- `Search params` — это query-параметры URL, которые описывают shareable navigation state.
- `Nested layout` — это общий layout или shell для группы вложенных экранов.
- `Route-level data loading` — это загрузка данных на уровне маршрута, а не только внутри компонента.
- `Navigation policy` — это правило доступа, redirect или guard, привязанный к переходу.

---

## Core / Fundamentals

Routing как архитектура отвечает на несколько вопросов сразу:

- как разложить продукт на экраны и вложенные разделы;
- какие части UI должны жить в общих layout-границах;
- какое состояние должно жить в URL;
- где должны обрабатываться loading и error states;
- как навигация связана с данными и правами доступа.

Практическое правило:

- `route params` нужны для идентичности ресурса;
- `search params` нужны для shareable navigation state;
- local state нужен для короткоживущих UI-эпизодов, которым не место в URL.

Если фильтры, вкладки, сортировка и текущая страница должны переживать refresh и шариться ссылкой, это уже не local state.

---

## Сравнение моделей

Есть несколько осей сравнения:

- `config-based routing` vs `file-based routing`;
- SPA router vs meta-framework router;
- router как слой навигации vs router как слой навигации плюс data loading.

Нормальная взрослая оптика такая:

- routing — это часть application architecture;
- router влияет не только на URL, но и на layout, loading, errors и code splitting;
- meta-framework-и вроде Next, Nuxt, Remix, SvelteKit связывают routing с серверными и route-level data concerns сильнее, чем классический SPA router.

---

## Другие подходы кратко

- `Angular Router` силен в large-scale app structure, guards, resolvers и декларативной настройке маршрутов.
- `Next App Router`, `Nuxt` и `SvelteKit` двигают routing ближе к meta-framework модели: route file structure, data loading, server/client boundaries и streaming concerns.
- `Remix` особенно силен там, где navigation, mutations и route-level data рассматриваются как единая система.

---

## Частые ошибки

- проектируют router как плоский список экранов без layout boundaries;
- складывают shareable navigation state в local state;
- тянут эфемерные UI-флаги в URL без реальной пользы;
- считают client-side guard полноценной security-моделью;
- делают route split только по названию, а не по смысловым boundaries.

---

## Anti-pattern

"Routing — это про ссылки и страницы, а архитектура живет где-то отдельно".

Нет. Как только приложение выходит за пределы трех экранов, routing становится одной из главных структурных осей UI.

---

## 🔧 Практика

1. Возьми один продуктовый сценарий и нарисуй route tree вместе с layout boundaries.
1. Для пяти состояний интерфейса определи, что из этого должно жить в route params, search params или local state.
1. Найди в текущем проекте одну навигационную политику и объясни, это UX guard или реальная авторизация.
1. Определи, где у вас route-level loading/error boundary уже нужна, а где пока можно жить проще.
