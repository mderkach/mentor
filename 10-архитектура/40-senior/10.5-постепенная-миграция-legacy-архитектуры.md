# 10.5 Постепенная миграция legacy-архитектуры

Почти ни одно достаточно большое frontend-приложение не начинается с идеальной архитектуры. Точнее, архитектура их есть, но высока вероятность, что будут вводиться дополнительные ограничения или упрощения. Senior-уровень начинается не там, где ты можешь нарисовать красивую схему, а там, где ты умеешь провести миграцию без остановки разработки и без попытки переписать всё за один заход.

Во время миграции временные исключения действительно бывают допустимы. Но только при одном условии: исключение должно быть оформлено как временное и контролируемое, а не как новая вечная норма.

## 🎯 Что ты освоишь

- как мигрировать код по границам, а не по случайным папкам;
- как использовать фасад, strangler-подход и слой совместимости;
- какие временные компромиссы допустимы, а какие уже ломают архитектуру;
- как не перепутать “перенесли файлы” с реальной архитектурной миграцией.

## Термины урока

- `Strangler pattern` — это постепенная замена старой реализации новой через управляемую boundary, а не переписывание всего за один заход.
- `Facade` — это единая точка входа, скрывающая старые и новые реализации за одним контрактом.
- `Compatibility layer` — это временный слой совместимости между legacy и новой моделью.
- `Migration budget` — это заранее оговорённая цена компромиссов и временных решений.
- `Freeze rule` — это правило “новый код сюда больше не кладём”, даже если старая зона ещё жива.

## Core / Fundamentals

Плохая миграция выглядит так:

- выбрали новую архитектуру;
- несколько недель переносили файлы;
- старые импорты остались;
- новые фичи продолжают строиться поверх legacy;
- через полгода в проекте уже две архитектуры и ни одна не управляется.

Хорошая миграция начинается не с файлов, а с boundary:

1. определить, какой старый модуль нужно изолировать;
1. поставить фасад или public API;
1. запретить новые прямые импорты в legacy-внутренности;
1. мигрировать потребителей по одному сценарию за раз;
1. удалить временный слой, когда он перестал быть нужен.

## Плохой и хороший вариант boundary

Плохо:

```ts
// любой новый код продолжает импортировать legacy напрямую
import { fetchRoom } from '@/legacy/RoomService';
import { parseLegacyRoom } from '@/legacy/helpers';
import { roomStore } from '@/legacy/stores/roomStore';
```

Хорошо:

```ts
// entities/room/index.ts
export { getRoomGateway } from './api/getRoomGateway';
export type { Room } from './model/Room';
```

```ts
// entities/room/api/getRoomGateway.ts
import { legacyRoomService } from '@/shared/legacy/roomService';
import { RoomSchema } from '../model/RoomSchema';

export async function getRoomGateway(id: string) {
  const dto = await legacyRoomService.fetchRoom(id);
  return RoomSchema.parse(dto);
}
```

Теперь новые потребители зависят не от старого сервиса напрямую, а от новой boundary. Это уже реальная миграция.

## Временные компромиссы допустимы, если они честные

Иногда приходится держать временную “грязную” зону, особенно если старый проект слишком сцеплен.

Нормальный вариант:

```text
shared/
  legacy/
    room/
      roomService.ts
      roomMapper.ts
```

Ненормальный вариант:

```text
shared/
  room/
  user/
  chat/
  notifications/
```

Во втором случае `shared` уже не compatibility-zone, а новое имя для legacy-монолита.

Полезные правила для временной зоны:

- явно назвать её `legacy`, `compat`, `migration` или аналогично;
- не добавлять туда новый доменный код по умолчанию;
- назначить ответственного и критерий удаления;
- документировать исключения, а не прятать их.

## Strangler-подход на примере

Представь, что старый виджет чата напрямую тянет DTO, store и websocket-helpers из разных мест. Переписывать всё разом опасно.

Тогда можно сделать так:

```ts
// features/chat/model/chatFacade.ts
import { getLegacyMessages } from '@/shared/legacy/chat/getLegacyMessages';
import { ChatMessageSchema } from '@/entities/chat/model/ChatMessageSchema';

export async function getMessages(roomId: string) {
  const items = await getLegacyMessages(roomId);
  return items.map((item) => ChatMessageSchema.parse(item));
}
```

Дальше:

- новые сценарии идут через `chatFacade`;
- старые вызовы постепенно переподключаются на неё;
- когда последний consumer ушёл с `getLegacyMessages`, старый импорт можно вырезать.

Это и есть strangler pattern в прикладном виде, без мистики.

## Как понять, что миграция идёт по-настоящему

Не по количеству “перенесённых файлов”. Полезнее смотреть на:

- сколько потребителей уже ушло на новый public API;
- есть ли запрет на новые legacy-imports;
- сокращается ли число точек, где DTO и old helpers протекают в новый код;
- исчезают ли временные фасады, а не копятся ли новые.

Если файлов стало больше, а старые зависимости всё ещё везде, миграция носит косметический характер.

## Частые ошибки

- сначала переносят код, а потом пытаются придумать границы;
- допускают временный слой совместимости, но не ставят срок или критерий удаления;
- дают новым фичам строиться поверх старого слоя “пока так быстрее”;
- меряют успех миграции числом moved files;
- делают временный компромисс постоянной архитектурой.

## Anti-pattern

"Надо сначала переписать всё, а потом уже включать новые правила".

Это удобная формула, если хочется никогда не закончить миграцию.

## 🔧 Практика

1. Выбери один legacy-модуль и опиши его будущий public API.
1. Поставь вокруг него слой совместимости и запрети новые прямые импорты.
1. Опиши, как ты будешь измерять прогресс миграции: сколько мест уже переведено на новый API, сколько прямых импортов удалено, где legacy DTO больше не попадают в новый код.
1. Найди один временный компромисс в проекте и сформулируй правило его удаления.

## ✅ Чеклист

- [ ] Я умею ставить boundary перед legacy-модулем
- [ ] Я отличаю временный слой совместимости от нового постоянного слоя
- [ ] Я не считаю “перенос файлов” достаточной миграцией
- [ ] Я умею вводить правило заморозки для старой зоны
- [ ] Я понимаю, как strangler-подход снижает риск полной переписи
