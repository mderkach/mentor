# 10.4 Composition root, dependency inversion, ports/adapters и anti-corruption layer

На senior-уровне архитектура перестаёт быть разговором “куда положить файл”. Главный вопрос уже другой: как сделать так, чтобы бизнес-сценарий не зависел напрямую от конкретного HTTP-клиента, backend DTO, websocket-формата или случайного legacy-сервиса.

Здесь и появляются `composition root`, `dependency inversion`, `ports/adapters` и `anti-corruption layer`. Не как backend-фетиш, а как способ не протащить внешний хаос в ядро клиентского приложения.

И да: даже если у тебя `contract-first`, OpenAPI, codegen и swagger “собирается честно”, это не гарантирует, что UI защищён. Контракт может не обновиться вовремя, backend может отдать неожиданный shape, а фронт может всё ещё быть слишком тесно сцеплен с DTO. `Contract-first` полезен, но без runtime-boundary и адаптеров он не спасает от боли.

В этом уроке `Zod` используется как конкретный инструментальный пример, потому что на нём удобно показать проверку входных данных и нормализацию DTO. Но архитектурная идея не зависит от `Zod`: тот же `port`, `adapter` и защитную границу вокруг внешнего контракта можно построить и вручную, и на другой библиотеке, и без runtime-схем вообще.

## 🎯 Что ты освоишь

- что на фронтенде реально можно считать `composition root`;
- как применять `dependency inversion` без контейнеров ради контейнеров;
- как проектировать `port` и `adapter` вокруг API, storage, websocket и legacy SDK;
- как использовать `Zod` как один из способов построить защитную границу вокруг внешнего контракта.

## Термины урока

- `Composition root` — это место, где приложение собирает зависимости и связывает абстракции с конкретными реализациями.
- `Dependency inversion` — это принцип, при котором сценарий зависит от абстракции, а не от конкретной transport-реализации.
- `Port` — это интерфейс нужного нам поведения со стороны доменной логики.
- `Adapter` — это реализация порта для конкретной внешней системы.
- `Anti-corruption layer` — это boundary, который изолирует внутреннюю модель приложения от кривого или нестабильного внешнего контракта.

## Core / Fundamentals

Во frontend `composition root` обычно живёт не в одном “волшебном контейнере”, а в сборке приложения:

- инициализация `QueryClient`;
- создание API-клиентов;
- wiring providers;
- регистрация маршрутов;
- подключение websocket/startup logic;
- выбор конкретных адаптеров для окружения.

Если бизнес-сценарий напрямую импортирует `axios`, `localStorage`, `window.location` и backend DTO, никакой реальной архитектурной границы нет.

## Port и adapter на простом примере

Предположим, фича должна загрузить пользователя. Ей не обязательно знать, откуда именно он приедет.

```ts
// entities/user/model/User.ts
export interface User {
  id: string;
  name: string;
}

export interface UserGateway {
  getById(id: string, signal?: AbortSignal): Promise<User>;
}
```

```ts
// entities/user/api/HttpUserGateway.ts
import { z } from 'zod';
import type { User, UserGateway } from '../model/User';

const UserDtoSchema = z
  .object({
    user_id: z.number(),
    user_name: z.string(),
  })
  .transform(
    (dto): User => ({
      id: String(dto.user_id),
      name: dto.user_name.trim(),
    }),
  );

export class HttpUserGateway implements UserGateway {
  constructor(private readonly api: { get: <T>(url: string, config?: { signal?: AbortSignal }) => Promise<T> }) {}

  async getById(id: string, signal?: AbortSignal): Promise<User> {
    const dto = await this.api.get<unknown>(`/users/${id}`, { signal });
    return UserDtoSchema.parse(dto);
  }
}
```

Здесь `Zod` не просто валидирует. Он:

- проверяет runtime-форму данных;
- нормализует DTO;
- превращает внешний контракт в внутреннюю модель.

Это уже полноценный adapter boundary. Именно этот акцент ты и прислал в тексте про `Zod`: схема может быть не только валидатором, но и runtime-adapter.

Но важно не перепутать архитектурную идею и конкретный инструмент:

- `port` и `adapter` можно спроектировать без `Zod`;
- `anti-corruption layer` можно построить без `Zod`;
- `Zod` здесь просто даёт компактный способ совместить runtime-проверку и преобразование DTO в одном месте.

## Тот же самый adapter без Zod

Ниже тот же подход, но без библиотеки схем:

```ts
// entities/user/api/parseUserDto.ts
import type { User } from '../model/User';

interface UserDTO {
  user_id: number;
  user_name: string;
}

function isUserDto(value: unknown): value is UserDTO {
  if (typeof value !== 'object' || value === null) {
    return false;
  }

  const dto = value as Record<string, unknown>;

  return typeof dto.user_id === 'number' && typeof dto.user_name === 'string';
}

export function parseUserDto(value: unknown): User {
  if (!isUserDto(value)) {
    throw new Error('Некорректный ответ API пользователя');
  }

  return {
    id: String(value.user_id),
    name: value.user_name.trim(),
  };
}
```

```ts
// entities/user/api/HttpUserGateway.ts
import type { User, UserGateway } from '../model/User';
import { parseUserDto } from './parseUserDto';

export class HttpUserGateway implements UserGateway {
  constructor(private readonly api: { get: <T>(url: string, config?: { signal?: AbortSignal }) => Promise<T> }) {}

  async getById(id: string, signal?: AbortSignal): Promise<User> {
    const dto = await this.api.get<unknown>(`/users/${id}`, { signal });
    return parseUserDto(dto);
  }
}
```

Архитектурно здесь всё то же самое:

- внешний ответ сервера не течёт в UI напрямую;
- преобразование сосредоточено в одном месте;
- точка отказа явная;
- фича по-прежнему зависит от `UserGateway`, а не от DTO.

Разница только в том, что с `Zod` короче писать runtime-проверку и удобнее масштабировать такие адаптеры.

## Почему это лучше ручного маппинга “на честном слове”

Ручной адаптер сам по себе не плох:

```ts
function userAdapter(dto: { user_id: number; user_name: string }): User {
  return {
    id: String(dto.user_id),
    name: dto.user_name,
  };
}
```

Проблема в другом: очень легко забыть проверить вход и продолжать жить в иллюзии, что внешние данные “ну, наверное, корректные”.

Схема вроде `Zod` полезна тем, что объединяет:

- runtime-проверку;
- адаптацию;
- явную точку отказа;
- удобство тестирования.

Но важно не скатиться в магию. Если transform внутри схемы начинает делать полбизнес-процесса, это уже плохой adapter.

## Contract-first не отменяет адаптер

Частая ошибка на проектах с кодогенерацией звучит так:

"Раз типы приехали из OpenAPI, значит можно просто пронести их до компонента".

На короткой дистанции это удобно. На длинной дистанции это приводит к тому, что snake_case, nullable-артефакты и backend-специфика начинают жить в UI как дома.

```ts
// Backend DTO
interface UserDTO {
  _id: number;
  user_id: number;
  user_name: string;
  user_lastname: string;
}

// Frontend model
interface User {
  id: number;
  name: string;
  lastName: string;
}

function adaptUser(dto: UserDTO): User {
  return {
    id: dto.user_id,
    name: dto.user_name,
    lastName: dto.user_lastname,
  };
}
```

Что это даёт на практике:

- UI зависит от своей модели, а не от backend naming;
- неиспользуемые поля вроде `_id` не текут дальше;
- изменение API ловится в одном boundary, а не в пятидесяти компонентах;
- adapter можно тестировать отдельно.

Именно поэтому “неконсистентность интерфейсов” между backend и frontend часто не проблема, а осознанная защита от избыточной связности.

## Composition root во frontend

Ниже не “идеальная Clean Architecture”, а нормальный practical wiring:

```tsx
// app/providers/AppProviders.tsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { ServicesContext } from './ServicesContext';
import { HttpUserGateway } from '@/entities/user/api/HttpUserGateway';

const queryClient = new QueryClient();
const apiClient = {
  get: async <T,>(url: string, config?: { signal?: AbortSignal }): Promise<T> => {
    const response = await fetch(url, { signal: config?.signal });
    return (await response.json()) as T;
  },
};

const services = {
  userGateway: new HttpUserGateway(apiClient),
};

export function AppProviders({ children }: { children: React.ReactNode }) {
  return (
    <QueryClientProvider client={queryClient}>
      <ServicesContext.Provider value={services}>{children}</ServicesContext.Provider>
    </QueryClientProvider>
  );
}
```

```ts
// features/profile/model/useProfile.ts
import { useServices } from '@/app/providers/ServicesContext';
import { useQuery } from '@tanstack/react-query';

export function useProfile(userId: string) {
  const { userGateway } = useServices();

  return useQuery({
    queryKey: ['user', userId],
    queryFn: ({ signal }) => userGateway.getById(userId, signal),
  });
}
```

Фича зависит от `UserGateway`, а не от конкретного HTTP-клиента и DTO.

## Где здесь место инструменту: пример с TanStack Query `meta`

Базовый архитектурный принцип здесь такой:

- fetch-функция должна заниматься получением данных;
- политика нотификаций, логирования и глобальной обработки ошибок должна жить в инфраструктурном слое;
- конкретный инструмент для этого может быть любым.

`TanStack Query meta` — это уже не база, а инструментальный способ реализовать этот принцип.

```ts
import {
  MutationCache,
  QueryCache,
  QueryClient,
  useMutation,
  useQuery,
} from '@tanstack/react-query';

export interface EnrichedQueryMeta extends Record<string, unknown> {
  readonly showError?: boolean;
  readonly showSuccess?: boolean;
  readonly errorMessageKey?: string;
  readonly successMessageKey?: string;
}

declare module '@tanstack/react-query' {
  interface Register {
    queryMeta: EnrichedQueryMeta;
    mutationMeta: EnrichedQueryMeta;
  }
}

export const queryClient = new QueryClient({
  queryCache: new QueryCache({
    onError(error, query) {
      if (query.meta?.showError && query.meta.errorMessageKey) {
        showToast({
          severity: 'error',
          message: `${query.meta.errorMessageKey}: ${String(error)}`,
        });
      }
    },
  }),
  mutationCache: new MutationCache({
    onSuccess(_data, _variables, _context, mutation) {
      if (mutation.meta?.showSuccess && mutation.meta.successMessageKey) {
        showToast({
          severity: 'success',
          message: mutation.meta.successMessageKey,
        });
      }
    },
  }),
});
```

```ts
async function fetchUserProfile(userId: string) {
  const response = await api.get(`/users/${userId}`);
  return response.data;
}

export function useUserProfile(userId: string) {
  return useQuery({
    queryKey: ['user', userId],
    queryFn: () => fetchUserProfile(userId),
    meta: {
      showError: true,
      errorMessageKey: 'errors.user_profile.load_failed',
    },
  });
}

export function useRenameUser() {
  return useMutation({
    mutationFn: renameUser,
    meta: {
      showSuccess: true,
      successMessageKey: 'success.user.rename',
    },
  });
}
```

Здесь важно не перепутать уровни:

- `TanStack Query meta` — это инструмент;
- вынос error/success policy из fetch-функции в infrastructure boundary — это архитектурная идея.

Если завтра ты уйдёшь с `TanStack Query`, сам принцип останется полезным.

## Anti-corruption layer нужен не только при микросервисах

Он нужен в любой ситуации, где внешний контракт:

- кривой;
- нестабильный;
- исторически странный;
- не совпадает с моделью UI.

Пример:

```ts
const RoomDtoSchema = z
  .object({
    room_id: z.string(),
    is_archived: z.boolean().optional(),
    title: z.string().nullable(),
  })
  .superRefine((dto, ctx) => {
    if (!dto.title || dto.title.trim().length === 0) {
      ctx.addIssue({
        code: z.ZodIssueCode.custom,
        message: 'title не должен быть пустым',
      });
    }
  })
  .transform((dto) => ({
    id: dto.room_id,
    title: dto.title!.trim(),
    archived: dto.is_archived ?? false,
  }));
```

Это уже anti-corruption layer:

- внешний DTO остаётся снаружи;
- внутренняя модель не заражается именами и nullable-странностями API;
- если backend сломался, ошибка возникает на boundary, а не расползается по UI.

## Где обычно перегибают

Есть две крайности:

1. “Никаких адаптеров, просто DTO прямо в компоненты”.
1. “На каждый `fetch` делаем три интерфейса, четыре сервиса и DI-контейнер”.

Обе крайности плохи.

Хороший ориентир:

- если контракт внешний и нестабильный, boundary нужен;
- если интеграция тривиальна и одноразова, не надо строить архитектуру из бронзы;
- если фича зависит от transport-деталей напрямую, ты платишь этим в каждом изменении API.

## Частые ошибки

- подменяют dependency inversion обычной передачей “любого объекта” без явного контракта;
- делают порт слишком низкоуровневым, повторяя API-клиент один в один;
- тащат DTO внутрь store, selectors и UI-модели;
- превращают `Zod`-схему в смесь валидации, трансформации и бизнес-логики;
- считают, что `contract-first` автоматически решает runtime-несовместимость;
- привязывают UX-нотификации к query/fetch-функциям вместо инфраструктурного слоя;
- путают composition root с местом, куда просто свалили все providers.

## Anti-pattern

"Если использовать `Zod`, отдельный adapter больше не нужен никогда".

Не совсем. `Zod` отлично подходит для границы рантайм-проверки и нормализации данных. Но если у тебя сложная координация сценария, несколько источников данных или нетривиальная доменная политика, схема не заменит архитектурный слой сама по себе. И наоборот: если `Zod` в проекте нет, это не мешает строить нормальные адаптеры и анти-коррупционный слой вручную.

## 🔧 Практика

1. Возьми один внешний API-контракт и опиши для него `port`, `adapter` и внутреннюю модель.
1. Найди место, где DTO прямо сейчас протекает в UI, и поставь перед ним boundary.
1. Покажи, что в твоём приложении является composition root, а что только делает вид.
1. Перепиши один ручной adapter через runtime-схему с `parse`/`safeParse` и transform.
