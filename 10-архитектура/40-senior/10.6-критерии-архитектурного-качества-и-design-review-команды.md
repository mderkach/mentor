# 10.6 Критерии архитектурного качества и design review команды

`Критерии архитектурного качества` — это набор обязательных требований, по которым команда одинаково оценивает архитектурные решения в коде и на ревью.

Проблема начинается, когда эти критерии либо слишком общие, либо слишком жёсткие и выдают локальные привычки за универсальные правила.

## 🎯 Что ты освоишь

- как превращать “пишем аккуратно” в наблюдаемые архитектурные критерии;
- какие правила стоит автоматизировать, а какие оставлять предметом design review;
- как использовать проектные правила и чеклисты как основу системы качества;
- как строить review без вкусовщины и бессмысленных придирок.

## Термины урока

- `Критерии качества` — это набор обязательных требований, которые команда умеет одинаково интерпретировать.
- `Design review` — это проверка архитектурных и продуктовых решений, а не только синтаксиса и форматирования.
- `Автоматически проверяемое правило` — это правило, которое можно проверить линтером, тестом, CI-проверкой или другим инструментом.
- `Правило для инженерного ревью` — это правило, которое нельзя надёжно проверить автоматически без контекста задачи.
- `Порядок оформления исключений` — это способ явно фиксировать осознанные отклонения от стандарта.

## Core / Fundamentals

Хорошие критерии качества отвечают на четыре вопроса:

1. Что обязательно.
1. Что желательно.
1. Что автоматизируется.
1. Что требует инженерного суждения.

Если этого разделения нет, команда обычно получает одно из двух:

- либо правила игнорируются;
- либо review превращается в спор о вкусе, а реальные архитектурные риски проходят мимо.

## Как раскладывать правила команды

Проектные правила, чеклисты ревью и соглашения команды лучше раскладывать на три группы.

### 1. Правила, которые можно проверить автоматически

Их надо кодировать в инструментах проверки.

Например:

- запрещён прямой импорт внутренностей модуля в обход его публичного API;
- `index.ts` не должен содержать побочные эффекты;
- нельзя импортировать `features/*` из других `features/*` того же уровня;
- нельзя использовать `any` без осознанного исключения.

```ts
// eslint.config.ts
export default [
  {
    rules: {
      'no-restricted-imports': [
        'error',
        {
          patterns: [
            '@/entities/*/model/*',
            '@/features/*/ui/*',
          ],
        },
      ],
    },
  },
];
```

### 2. Правила, которые проверяются на ревью

Их нельзя надёжно поймать линтером.

Например:

- этот компонент правда shared или просто случайно переиспользован два раза;
- этот adapter изолирует внешний контракт или просто дублирует DTO;
- этот hook улучшает архитектуру или просто прячет хаос под новый файл.

### 3. Локальные соглашения

Например:

- суффикс `Proto`;
- `classnames as cx`;
- порядок импортов;
- конкретная файловая конвенция в проекте.

Они могут быть разумными. Но их нельзя выдавать за универсальные правила для любого frontend-проекта.

## Как выглядит хороший design review

Плохо:

- “мне не нравится это название”;
- “я бы положил файл в другое место”;
- “у нас вроде так не принято”.

Хорошо:

- “этот `feature` лезет во внутренности `entity` в обход её публичного API; следующее изменение разломает как минимум два места использования”;
- “этот shared-компонент знает доменную операцию удаления, значит reuse здесь ложный”;
- “здесь DTO течёт в UI напрямую, поэтому boundary между transport и внутренней моделью отсутствует”.

То есть review должен говорить языком риска и границ, а не языком вкуса.

## Мини-кейс

Ниже код, который формально “работает”, но по review должен получить блокирующие комментарии:

```tsx
import { useUserStore } from '@/entities/user/model/user.store';
import { deleteRoom } from '@/legacy/roomApi';

export function RoomDangerZone({ roomId }: { roomId: string }) {
  const currentUser = useUserStore((state) => state.currentUser);

  return (
    <button
      onClick={() => {
        if (currentUser?.role === 'admin') {
          deleteRoom(roomId);
        }
      }}
    >
      Delete
    </button>
  );
}
```

Что здесь плохо:

- UI напрямую тянет внутренности `entity`;
- feature зависит от legacy API в обход boundary;
- permission-policy зашита в кнопку;
- нет явного use case или adapter boundary для destructive action.

Нормализованный вариант:

```tsx
import { Button } from '@/shared/ui/Button';
import { useDeleteRoom } from '../model/useDeleteRoom';

export function RoomDangerZone({ roomId }: { roomId: string }) {
  const { canDelete, remove } = useDeleteRoom(roomId);

  if (!canDelete) {
    return null;
  }

  return <Button onClick={remove}>Delete</Button>;
}
```

Теперь review обсуждает уже `useDeleteRoom` и его границы, а не распутывает бизнес-логику из кнопки.

## Порядок оформления исключений важнее, чем кажется

Если стандарт не допускает исключений, команда начнёт их делать молча.

Нормальный путь:

- исключение фиксируется явно;
- у него есть причина;
- у него есть ответственный;
- у него есть срок жизни или критерий удаления.

Иначе “временная уступка” быстро становится новой архитектурой проекта по факту.

## Частые ошибки

- пытаются линтером ловить то, что требует инженерного суждения;
- спорят о локальном стиле вместо обсуждения реальных архитектурных рисков;
- копируют правила команды из одного проекта в другой без адаптации контекста;
- пишут чеклист из двадцати пунктов, которым никто не следует;
- не различают must-have rules и рекомендательные советы.

## Anti-pattern

"Раз правило попало в чеклист, значит оно одинаково важно и всегда универсально".

Нет. Часть правил — про архитектуру. Часть — про локальное удобство разработки. Часть — вообще про исторические договорённости конкретной команды.

## 🔧 Практика

1. Возьми один проектный чеклист и разложи его на `проверять автоматически / проверять на ревью / локальное соглашение`.
1. Для трёх архитектурных правил предложи формулировку в терминах риска, а не вкуса.
1. Придумай один порядок оформления исключения для временного legacy-обхода.
1. Найди правило, которое команда пытается автоматизировать зря, и объясни, почему там нужна инженерная проверка.

## ✅ Чеклист

- [ ] Я умею отличать автоматически проверяемое правило от правила для инженерного ревью
- [ ] Я не путаю design review с ревью только по стилю кода
- [ ] Я умею формулировать замечание через риск и границу
- [ ] Я понимаю, какие проектные стандарты являются локальной конвенцией
- [ ] Я умею оформлять осознанные исключения из стандарта
