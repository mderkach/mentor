# 10.2 FSD: слои, слайсы, сегменты и Public API

`Feature-Sliced Design (FSD)` — это подход к организации frontend-кода по слоям ответственности и предметным модулям. Его цель — сделать зависимости между частями приложения явными и не дать коду расползтись в набор случайных импортов.

Этот урок не про то, как правильно назвать папки. Он про то, как использовать FSD как модель границ: что можно импортировать, где проходит ответственность модуля и зачем нужен `Public API`.

## 🎯 Что ты освоишь

- как читать FSD как модель архитектурных границ;
- как делить код на слои, слайсы и сегменты без механического копипаста структуры;
- зачем нужен `Public API` и почему прямой импорт внутренностей модуля может ломать границы;
- как жить с временными компромиссами при миграции legacy, не выдавая их за норму.

## Термины урока

- `Feature-Sliced Design (FSD)` — это способ организовать код приложения по уровням ответственности и предметным модулям.
- `Layer` — это уровень ответственности в приложении: `app`, `pages`, `widgets`, `features`, `entities`, `shared`.
- `Slice` — это модуль предметной области внутри слоя.
- `Segment` — это группировка кода по назначению внутри слоя или слайса: `ui`, `api`, `model`, `lib`.
- `Public API` — это контролируемая точка входа в модуль, обычно через `index.ts`.

## Core / Fundamentals

FSD решает не проблему “как назвать папки”, а проблему “как не превратить приложение в сеть неявных зависимостей”.

Практическая логика такая:

- слой отвечает на вопрос “на каком уровне живёт код”;
- слайс отвечает на вопрос “к какой предметной области он относится”;
- сегмент отвечает на вопрос “что именно этот код делает”;
- public API отвечает на вопрос “что из этого модуля вообще разрешено использовать снаружи”.

Если эта модель работает, команда получает несколько выгод сразу:

- проще понимать зону ответственности модулей;
- проще проводить код-ревью;
- проще ограничивать связанность;
- проще мигрировать legacy по частям, а не переписывать всё одним махом.

Важно: FSD не является обязательным стандартом для любого проекта. Это один из способов организовать код, который особенно полезен там, где приложение растёт по фичам, а границы модулей начинают реально влиять на скорость изменений и число ошибок.

## Как это выглядит в коде

```text
src/
  app/
    providers/
    router/
  pages/
    profile/
  widgets/
    profile-header/
  features/
    edit-profile/
  entities/
    user/
      api/
      model/
      ui/
      index.ts
  shared/
    ui/
    lib/
```

Полезная эвристика:

- `entities/user` хранит модель пользователя и то, что связано именно с сущностью `user`;
- `features/edit-profile` хранит сценарий редактирования профиля, а не “всё, где встречается user”;
- `pages/profile` собирает экран;
- `shared` не должен становиться папкой “всё непонятное кидаем сюда”.

## Public API важнее, чем кажется

Без публичной поверхности модуль формально существует, но по факту любой файл внутри него становится общедоступным. Это быстро убивает возможность безопасно менять внутренности.

```ts
// entities/user/index.ts
export { UserCard } from './ui/UserCard';
export { useUserQuery } from './api/useUserQuery';
export type { User } from './model/types';
```

```ts
// features/edit-profile/ui/EditProfileButton.tsx
import { UserCard, useUserQuery } from '@/entities/user';

// Плохо: feature знает внутреннюю файловую структуру entity.
// import { userStore } from '@/entities/user/model/user.store';
```

Полезное правило:

- если внешний код импортирует файл глубже `index.ts`, значит public API либо не спроектирован, либо сознательно обходится;
- если `index.ts` начинает содержать побочные эффекты, вычисления и скрытую магию, это уже не public API, а точка утечки.

Важно не переоценивать эту тему. Это не отдельная “главная боль фронтенда” уровня prop drilling. Проблема становится заметной там, где у модулей действительно есть публичный API: в монорепах, библиотеках компонентов, FSD-структуре и больших кодовых базах с явными архитектурными границами.

## Сегменты нужны не для красоты

Обычно в FSD-проектах используются сегменты вроде:

- `ui` — представление;
- `api` — доступ к внешнему миру;
- `model` — состояния, контракты, бизнес-правила;
- `lib` — локальные утилиты, нужные именно этому слайсу.

Это не религия. Но если всё лежит в одной папке, через пару месяцев различить UI, доменную логику и сетевой код становится неприятно даже автору.

```ts
// entities/user/model/getDisplayName.ts
import type { User } from './types';

export function getDisplayName(user: User): string {
  return user.fullName?.trim() || user.username;
}
```

```tsx
// entities/user/ui/UserCard.tsx
import { getDisplayName } from '../model/getDisplayName';
import type { User } from '../model/types';

export function UserCard({ user }: { user: User }) {
  return <div>{getDisplayName(user)}</div>;
}
```

Здесь граница читается сразу: UI использует модель, но модель не зависит от UI.

## Что делать при миграции legacy

На миграции иногда приходится делать временные послабления, особенно вокруг `shared`. Это не повод объявить “архитектура не взлетела”, но и не повод узаконить хаос.

Нормальная миграционная стратегия выглядит так:

- явно пометить compatibility-зону;
- не добавлять туда новый доменный код без причины;
- иметь план выноса;
- запрещать новым модулям строиться поверх временного обходного слоя.

Плохой вариант:

```text
shared/
  api/
  user/
  edit-profile/
  pages/
```

Хороший временный вариант:

```text
shared/
  legacy/
    user/
      userFacade.ts
```

Разница в том, что `shared/legacy` честно показывает: это карантин, а не новая архитектурная норма.

## Где FSD уместен, а где уже начинается карго-культ

FSD полезен, когда:

- у тебя не один экран и не один автор;
- приложение растёт по фичам, а не только по количеству компонентов;
- нужны явные правила импорта и понятные границы ответственности.

FSD легко переусложнить, когда:

- приложение очень маленькое;
- команда механически дублирует слои без реальной причины;
- каждый модуль получает `ui/api/model/lib`, даже если внутри два файла.

Если модуль состоит из одного простого компонента и пары helper-функций, ему не нужна искусственно усложнённая структура.

## Частые ошибки

- объявляют `shared` свалкой общего кода;
- режут по FSD только файловую систему, но не правила импорта;
- дают `features` напрямую тянуть внутренности других `features`;
- делают giant `index.ts`, который реэкспортирует всё подряд без смысла;
- начинают с “идеальной” структуры, не понимая, какие доменные границы вообще существуют.

## Anti-pattern

"Мы перешли на FSD, значит архитектурные проблемы решены".

Нет. Если бизнес-сценарии всё равно текут между слоями, FSD превращается в формальную схему каталогов без реальной пользы.

## 🔧 Практика

1. Возьми один экран и разложи его на `page`, `widget`, `feature`, `entity`, `shared`.
1. Для двух модулей сформулируй их public API и отметь, где прямой импорт внутренних файлов начнёт ломать границы.
1. Найди один кусок кода, который сейчас лежит в `shared`, но по смыслу принадлежит конкретной предметной области.
1. Опиши одно миграционное исключение, которое допустимо временно, и зафиксируй, когда оно должно исчезнуть.
