# 10.3 UI-границы: shared, dumb components, compound components и hooks

UI-границы — это правила, по которым мы отделяем универсальные компоненты интерфейса от компонентов, которые знают о данных, правах доступа и бизнес-сценарии.

Ошибки здесь начинаются с небольших решений: “давайте положим этот компонент в `shared`, вдруг переиспользуем”. Через некоторое время оказывается, что “общий” компонент знает про права доступа, серверные запросы, аналитику, маршрутизацию и половину бизнес-правил. Это уже не shared UI, а замаскированный feature-компонент.

## 🎯 Что ты освоишь

- как удерживать границу между универсальным UI и бизнес-сценарием;
- как использовать `dumb components` без превращения их в бесполезные div-обёртки;
- когда нужен паттерн `compound components`;
- как и зачем выносить сложное поведение из компонента в hook.

## Термины урока

- `Dumb component` — это компонент, который не знает бизнес-контекста и работает через явные `props`.
- `Smart component` — это компонент или модуль, где живёт сценарий использования, координация действий и привязка к данным.
- `Compound components` — это набор связанных компонентов с общим API вроде `Modal.Header`, `Modal.Body`, `Modal.Footer`.
- `Headless component` — это компонент или hook, который даёт поведение без жёстко зашитого UI.
- `Business state` — это состояние, связанное с предметной логикой, а не только с локальным взаимодействием.

## Core / Fundamentals

Полезное правило: shared UI не должен содержать бизнес-состояние. Это хорошая эвристика, если понимать её правильно.

Правильная трактовка:

- универсальный UI может иметь локальное техническое состояние;
- универсальный UI не должен знать доменные сценарии;
- бизнес-эффекты, проверки прав доступа, [запросы к серверу](../../06-ui-frameworks/30-middle/core/6.6-server-state-и-data-fetching.md) и оркестрация сценария должны жить выше.

Пример нормального локального UI-состояния в generic-компоненте:

- открыт ли раскрывающийся блок или панель;
- какая вкладка активна;
- где находится фокус внутри всплывающего элемента.

Пример бизнес-состояния, которое туда уже лучше не тянуть:

- можно ли пользователю удалить комнату;
- нужно ли после клика отправить запрос на удаление или изменение данных;
- какой toast показывать по бизнес-ошибке;
- какую аналитику отправлять для конкретного бизнес-события.

## Плохой и хороший вариант

Плохо:

```tsx
// shared/ui/DeleteButton.tsx
import { useDeleteRoomRequest } from '@/features/delete-room';
import { usePermissions } from '@/shared/auth';

export function DeleteButton({ roomId }: { roomId: string }) {
  const canDelete = usePermissions('room.delete');
  const request = useDeleteRoomRequest();

  if (!canDelete) {
    return null;
  }

  return <button onClick={() => request.send({ roomId })}>Удалить</button>;
}
```

Этот компонент не shared. Он знает предметную операцию, права доступа и логику запроса на удаление.

Лучше:

```tsx
// shared/ui/Button.tsx
export function Button({
  children,
  disabled,
  onClick,
}: {
  children: React.ReactNode;
  disabled?: boolean;
  onClick?: () => void;
}) {
  return (
    <button disabled={disabled} onClick={onClick}>
      {children}
    </button>
  );
}
```

```tsx
// features/delete-room/ui/DeleteRoomButton.tsx
import { Button } from '@/shared/ui/Button';
import { useDeleteRoom } from '../model/useDeleteRoom';

export function DeleteRoomButton({ roomId }: { roomId: string }) {
  const { canDelete, remove, isPending } = useDeleteRoom(roomId);

  if (!canDelete) {
    return null;
  }

  return (
    <Button disabled={isPending} onClick={remove}>
      Удалить комнату
    </Button>
  );
}
```

Здесь shared-компонент остался generic, а бизнес-сценарий живёт в `feature`.

## Когда выносить поведение в hook

Хорошая инженерная эвристика: если компонент быстро набирает сложное состояние, обработчики и эффекты, обычно лучше вынести поведение в hook.

Это полезно потому что:

- компонент остаётся читаемым;
- оркестрацию проще тестировать отдельно;
- позднее проще заменить локальное состояние на store, query или state machine.

```tsx
// features/invite-user/model/useInviteUserForm.ts
export function useInviteUserForm() {
  const [query, setQuery] = useState('');
  const [selectedIds, setSelectedIds] = useState<string[]>([]);

  const canSubmit = query.trim().length > 0 && selectedIds.length > 0;

  function toggleUser(id: string) {
    setSelectedIds((prev) =>
      prev.includes(id) ? prev.filter((item) => item !== id) : [...prev, id],
    );
  }

  return {
    query,
    selectedIds,
    canSubmit,
    setQuery,
    toggleUser,
  };
}
```

```tsx
// features/invite-user/ui/InviteUserForm.tsx
import { useInviteUserForm } from '../model/useInviteUserForm';

export function InviteUserForm() {
  const { query, selectedIds, canSubmit, setQuery, toggleUser } = useInviteUserForm();

  return (
    <section>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <p>Выбрано: {selectedIds.length}</p>
      <button disabled={!canSubmit}>Пригласить</button>
    </section>
  );
}
```

Сам hook не обязан жить в `shared`. Если он знает бизнес-термины и сценарий, это часть конкретной `feature`.

## Compound components нужны не всегда

`Compound components` полезны, когда у блока есть несколько связанных частей и ты хочешь дать потребителю выразительный API.

```tsx
<Dropdown>
  <Dropdown.Trigger>Открыть</Dropdown.Trigger>
  <Dropdown.Content>
    <Dropdown.Item value="edit">Редактировать</Dropdown.Item>
    <Dropdown.Item value="delete">Удалить</Dropdown.Item>
  </Dropdown.Content>
</Dropdown>
```

Это хороший вариант, если:

- части действительно используются вместе;
- между ними есть общий внутренний контракт;
- API становится понятнее, чем набор флагов.

Это плохой вариант, если:

- у компонента по факту одна точка использования;
- API сложнее самого сценария;
- compound-паттерн выбран “потому что выглядит солидно”.

## Где проходит реальная граница shared

Хороший вопрос для ревью:

“Если завтра бизнес-правило поменяется, придётся ли менять этот shared-компонент?”

Если ответ “да, почти наверняка”, то это уже не shared UI, а модуль предметной логики под видом UI-кита.

## Частые ошибки

- кладут в `shared/ui` компонент, который знает про серверный запрос, права доступа и маршрутизацию;
- понимают `dumb component` как “вообще без состояния”, хотя локальное интерактивное состояние может быть нормальным;
- выносят любой handler в hook просто ради моды;
- строят `compound components` для блока, который нужен ровно в одном месте;
- делают generic API слишком рано и ломают читаемость ради гипотетического reuse.

## Anti-pattern

"Раз компонент переиспользуется два раза, значит его надо срочно поднять в `shared`".

Нет. Сначала проверь, совпадает ли у этих двух мест именно ответственность, а не только похожая разметка.

## 🔧 Практика

1. Возьми три компонента из проекта и для каждого определи: `shared`, `entity`, `feature` или `widget`.
1. Найди один shared-компонент, который знает слишком много про бизнес-сценарий.
1. Разбей один перегруженный компонент на UI-часть и hook с поведением.
1. Придумай пример, где `compound components` реально делают API чище, и пример, где они только мешают.
