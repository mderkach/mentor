## 🎯 Что ты освоишь

- пройти инструментальную реализацию без смешения с общей инженерной моделью;
- увидеть, где конкретный инструмент помогает, а где создает ограничения;
- использовать core-слой как базу, а не как кусок внутри tool-урока.

---

## Термины урока

- `Chunk` — это отдельная часть output bundle.
- `Code splitting` — это стратегия разбиения кода на части для разной загрузки.
- `Source map` — это карта соответствия между исходным кодом и результатом сборки.
- `Waterfall loading` — это каскадная загрузка чанков друг за другом.
- `Разбор проблем сборочного артефакта` — это диагностика проблем, связанных со сборкой, output и runtime behavior артефакта.

---

## Реализация через инструмент

В Vite, Webpack и похожих системах split обычно рождается из:

- `dynamic import`;
- route-level lazy loading;
- manual chunk strategies;
- plugin behavior.

```ts
const AdminPage = lazy(() => import("./pages/AdminPage"));
const ReportsPage = lazy(() => import("./pages/ReportsPage"));
```

Source maps тоже настраиваются инструментом:

```ts
build: {
  sourcemap: true
}
```

Но сам инструмент не решает за вас:

- где должна пройти boundary;
- кто и как будет использовать source maps;
- что делать, если local и prod bundle ведут себя по-разному.

---

## Границы применимости инструмента

- больше чанков не означает автоматически лучший performance;
- source maps полезны для диагностики, но требуют политики публикации и доступа;
- один и тот же split strategy может быть хороша для admin surface и плоха для user-critical path;
- инструмент умеет нарезать код, но не знает, какие boundaries действительно осмысленны для продукта.

---

## Альтернативы кратко

- `Vite` и `Webpack` по-разному подходят к chunk graph и plugin ecosystem, но логика boundaries одна и та же.
- `Rspack` и другие новые bundlers меняют performance profile, но не отменяют trade-offs over-splitting и waterfall loading.

---

## Production failure modes

- over-splitting делает сеть и загрузку менее стабильной;
- route split выглядит "умно", но в реальности создает waterfall;
- source maps публикуются без понимания security trade-offs;
- local build работает, а prod bundle ломается из-за plugin, alias или transform mismatch.

---

## Anti-pattern

"Если появился `lazy()`, значит performance уже улучшили".

Нет. Вы могли просто перенести стоимость загрузки в другой момент и сделать UX менее предсказуемым.

---

## 🔧 Практика

1. Найди один тяжелый экран и объясни, route-level это candidate для splitting или feature-level.
1. Посмотри на output сборки и опиши, какие чанки образовались и почему.
1. Проверь политику source maps в проекте: где они генерируются, кто ими пользуется, где они доступны.
1. Смоделируй одну bundle-проблему и опиши, как ты будешь искать причину между local и prod behavior.
