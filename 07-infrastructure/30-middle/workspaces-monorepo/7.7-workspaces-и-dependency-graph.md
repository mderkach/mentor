# 7.7 Workspaces: пакетные границы и dependency graph

Это отдельный инструментальный слой темы. Общую инженерную модель смотри в [Workspaces, границы monorepo и dependency graph](../core/7.7-workspaces-границы-monorepo-и-dependency-graph.md).

- общая модель темы: [Workspaces, границы monorepo и dependency graph](../core/7.7-workspaces-границы-monorepo-и-dependency-graph.md)

## 🎯 Что ты освоишь

- пройти инструментальную реализацию без смешения с общей инженерной моделью;
- увидеть, где конкретный инструмент помогает, а где создает ограничения;
- использовать core-слой как базу, а не как кусок внутри tool-урока.

---

## Термины урока

- `Workspace` — это механизм управления несколькими пакетами в одном репозитории.
- `Monorepo` — это подход, при котором несколько приложений и библиотек живут в одной кодовой базе.
- `Dependency graph` — это карта связей между пакетами и задачами.
- `Package boundary` — это явно ограниченный public API пакета и его зона ответственности.
- `Shared coupling` — это скрытая связанность через "общий" код без ясного владельца и контракта.

---

## Реализация через инструмент

На практике workspaces чаще всего оформляются через:

- `pnpm workspaces`;
- `npm workspaces`;
- `yarn workspaces`.

```yaml
packages:
  - apps/*
  - packages/*
```

Инструменты дают механику:

- установку нескольких пакетов;
- локальные ссылки между пакетами;
- workspace-aware scripts.

Но не дают автоматически:

- хорошего public API;
- модель зон ответственности;
- clean dependency graph;
- orchestration и affected execution.

---

## Границы применимости инструмента

- workspaces полезны, когда реально есть пакетные границы, а не просто много папок;
- monorepo окупается, если shared code и общий tooling дают больше пользы, чем coordination cost;
- shared package без owner'а быстро становится источником боли;
- workspaces не заменяют ни архитектурные границы, ни orchestration tooling.

---

## Альтернативы кратко

- `pnpm`, `npm`, `yarn` workspaces решают одну базовую задачу, но отличаются DX и dependency handling.
- `Nx` и `Turborepo` нужно мыслить как следующий слой orchestration, а не как способ "внезапно сделать monorepo зрелым".

---

## Production failure modes

- monorepo создается раньше, чем возникла реальная межпакетная проблема;
- внутренности пакетов импортируются напрямую, а public API игнорируется;
- dependency graph грязный, но команда этого не видит;
- shared package тащится всеми, но никто им не владеет.

---

## Anti-pattern

"Сделаем monorepo, и порядок появится сам".

Чаще происходит обратное: плохие границы становятся общими быстрее и дешевле.

---

## 🔧 Практика

1. Возьми проект с несколькими пакетами и нарисуй его dependency graph в упрощенном виде.
1. Для каждого shared package назови owner'а и public API.
1. Найди одно место, где shared code уже стал shared coupling.
1. Аргументируй, monorepo здесь реально окупается или пока создает ложное чувство порядка.
