# 7.4 Vite: env и базовая сборка frontend-приложения

Это отдельный инструментальный слой темы. Общую инженерную модель смотри в [Build systems и env](../core/7.4-build-systems-и-env.md).

---

## Что уже должно быть знакомо

- общая модель темы: [Build systems и env](../core/7.4-build-systems-и-env.md)

---

## 🎯 Что ты освоишь

- пройти инструментальную реализацию без смешения с общей инженерной моделью;
- увидеть, где конкретный инструмент помогает, а где создает ограничения;
- использовать core-слой как базу, а не как кусок внутри tool-урока.

---

## Термины урока

- `Build pipeline` — это путь от исходников до deployable artifact.
- `Dev server` — это среда локальной разработки с быстрым feedback loop.
- `Build-time config` — это конфигурация, вшиваемая в артефакт на этапе сборки.
- `Runtime config` — это конфигурация, которая доступна уже после сборки в окружении доставки.
- `Env drift` — это рассинхрон между окружениями по конфигу и поведению.

---

## Реализация через инструмент

В Vite эта модель обычно выглядит так:

- dev server дает быстрый локальный feedback;
- `import.meta.env` читает публичные env;
- `VITE_` префикс делает переменную доступной клиентскому коду;
- `vite.config` описывает aliases, plugins и build behavior;
- HMR помогает разрабатывать быстрее, но не равен production behavior.

```ts
import { defineConfig } from "vite";
import { fileURLToPath, URL } from "node:url";

export default defineConfig({
  resolve: {
    alias: {
      "@": fileURLToPath(new URL("./src", import.meta.url))
    }
  }
});
```

```env
VITE_API_BASE_URL=https://api.example.com
```

---

## Границы применимости инструмента

- `Vite` прекрасен для современного frontend DX, но не является синонимом build system как такового;
- `Vite` не решает автоматически runtime config, cache strategy и delivery architecture;
- если проект требует более специфических интеграций, сложного legacy-bundling или platform-level customization, одной familiarity с Vite может быть мало.

---

## Альтернативы кратко

- `Webpack` остается важным там, где нужна зрелая и очень гибкая plugin ecosystem.
- `Rspack` предлагает похожую экосистемную форму с другим performance profile.
- `esbuild-oriented tooling` делает ставку на скорость и простоту, но не закрывает все сценарии одинаково.
- `Turbopack` пока стоит мыслить как evolving story, а не как зрелый "дефолт мира".

---

## Production failure modes

- секреты попадают в клиентский env;
- aliases синхронизированы в `vite.config`, но забыты в `tsconfig` или тестовой среде;
- локальный dev server ведет себя не так, как production build, и команда это замечает слишком поздно;
- env drift между stage и prod ломает приложение при формально успешной сборке.

---

## Anti-pattern

"Мы используем Vite, значит инфраструктурная часть сборки уже понятна".

Нет. Это значит только то, что у вас есть один конкретный инструмент. Понять нужно саму build-систему и границы ее применимости.

---

## 🔧 Практика

1. Возьми текущий проект и разложи его build pipeline: dev server, transforms, bundling, output artifact, config injection.
1. Назови все env, которые реально уходят в клиент, и объясни, почему там нет секретов.
1. Проверь, синхронизированы ли aliases между build, typecheck и test tooling.
1. Определи один риск env drift между local, stage и prod.
