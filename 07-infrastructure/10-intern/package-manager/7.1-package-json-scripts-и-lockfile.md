## 🎯 Что ты освоишь

- пройти инструментальную реализацию без смешения с общей инженерной моделью;
- увидеть, где конкретный инструмент помогает, а где создает ограничения;
- использовать core-слой как базу, а не как кусок внутри tool-урока.

---

## Термины урока

- `Манифест` — это файл, который описывает метаданные проекта, зависимости и его интерфейс запуска.
- `Script` — это именованная команда проекта, запускаемая через package manager.
- `Lockfile` — это файл с точными версиями и графом зависимостей.
- `Peer dependency` — это зависимость, которую пакет ожидает получить из окружения проекта.

---

## Реализация через инструмент

На практике разные менеджеры пакетов ведут себя чуть по-разному:

- `npm`, `pnpm`, `yarn` по-разному работают с lockfile и workspace scripts;
- `npx` и `pnpm dlx` полезны для одноразовых CLI;
- scripts, умеющие работать в workspace-среде, становятся важны, когда проект растет в monorepo.

Типовой manifest выглядит так:

```json
{
  "scripts": {
    "dev": "vite",
    "lint": "eslint .",
    "test": "vitest run",
    "build": "vite build"
  },
  "dependencies": {
    "react": "^19.0.0"
  },
  "devDependencies": {
    "vite": "^7.0.0"
  }
}
```

---

## Границы применимости инструмента

- scripts удобны как интерфейс команды, но не заменяют полноценный task orchestration на масштабе monorepo;
- `npx`/`pnpm dlx` хороши для одноразовых CLI, но не должны подменять project-level tooling contract;
- lockfile полезен только если команда признает один manager и один источник истины.

---

## Альтернативы кратко

- `npm`, `pnpm`, `yarn` дают общий интерфейс scripts, но отличаются по workspace behavior и управлению зависимостями.
- Для сложных task chains иногда приходят task runners и orchestration tools, но manifest все равно остается точкой входа.

---

## Production failure modes

- в репозитории живет несколько lockfile, и никто уже не понимает, какой из них основной;
- важные команды существуют в wiki, но не в scripts;
- `peerDependencies` игнорируются, пока не ломается build или runtime;
- script chaining превращается в нечитаемый набор shell-хаков, который никто не умеет поддерживать.

---

## Anti-pattern

"Я просто знаю, что надо писать `npm run dev`, и этого достаточно".

Нет. Нужно понимать, что это за контракт, где он описан и какие еще команды проект считает обязательными.

---

## 🔧 Практика

1. Открой `package.json` любого проекта и объясни каждую dependency section.
1. Назови минимальный локальный цикл проверок, который должен быть виден в `scripts`.
1. Найди основной lockfile и объясни, почему другой package manager в этом проекте будет проблемой.
1. Определи один script, который сейчас выглядит как мусорный ящик, и предложи, как сделать его понятнее.
