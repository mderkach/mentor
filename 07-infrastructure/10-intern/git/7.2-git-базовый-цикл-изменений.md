## 🎯 Что ты освоишь

- пройти инструментальную реализацию без смешения с общей инженерной моделью;
- увидеть, где конкретный инструмент помогает, а где создает ограничения;
- использовать core-слой как базу, а не как кусок внутри tool-урока.

---

## Термины урока

- `Working tree` — это текущее состояние файлов в каталоге проекта.
- `Index` — это staged-слой перед коммитом.
- `Commit graph` — это граф истории изменений.
- `Atomic commit` — это коммит с одним понятным смыслом.
- `Branch` — это временная линия изоляции работы.
- `Pull request` — это точка review и интеграции в основную ветку.

---

## Реализация через инструмент

На практике базовый цикл выглядит так:

```bash
git switch -c feature/header-search
git status
git add src/components/Header.tsx
git commit -m "add header search field"
git pull --rebase origin main
git push -u origin feature/header-search
```

Командный слой поверх Git обычно включает:

- PR flow;
- protected branches;
- merge policy;
- squash/rebase conventions.

Именно этот слой превращает Git из набора команд в воспроизводимый процесс изменений.

---

## Границы применимости инструмента

- Git хранит историю изменений, но сам по себе не гарантирует хорошую историю — это делает дисциплина команды.
- `rebase` удобен для чистой линейной истории, но не заменяет понимание commit graph.
- `merge` сохраняет контекст ветвления, но может делать историю шумной, если ветка и коммиты неаккуратны.

---

## Альтернативы кратко

- `merge`, `rebase`, `squash` — это не "лучший инструмент навсегда", а разные способы управлять историей.
- GitHub, GitLab и другие платформы добавляют PR/MR workflow, но underlying model истории остается той же.

---

## Production failure modes

- ветка смешивает фичу, форматирование, случайные фиксы и обновление зависимостей;
- конфликты откладывают до последнего, а потом чинят в панике;
- PR скрывает смысл изменений, потому что история неряшливая;
- команда спорит о merge/rebase, но на самом деле у нее нет базового правила для atomic commits.

---

## Anti-pattern

"Главное запушить, а там в review разберутся".

Обычно это означает, что разбираться придется всей команде и уже под давлением времени.

---

## 🔧 Практика

1. Возьми один локальный набор изменений и разложи его на working tree, staged changes и будущий commit.
1. Проверь, можно ли этот commit назвать atomic.
1. Опиши, какая merge/rebase policy была бы разумна для вашей команды и почему.
1. Смоделируй маленький конфликт и объясни, как ты удержишь смысл истории после его разрешения.
