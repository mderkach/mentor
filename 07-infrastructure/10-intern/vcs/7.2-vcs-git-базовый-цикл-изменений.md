## Зачем это нужно на практике

На этом уровне Git чаще ломается не на сложных сценариях, а на банальной рутине:

- случайно закоммитили лишние файлы;
- настроили глобальный конфиг и сломали поведение сразу в нескольких репозиториях;
- переименовали путь, а потом полдня ловят "модуль не найден";
- локально видят ветки, которых на сервере уже давно нет.

То есть проблема обычно не в "сложности Git", а в том, что базовый цикл и базовая гигиена не собраны в понятную привычку.

## 🎯 Что ты освоишь

- понимать, где заканчивается общая модель VCS и начинается конкретная механика Git;
- настроить репозиторий через `git config --local` под правила команды;
- пройти безопасный базовый цикл изменений от working tree до push;
- разложить по полочкам частые operational-кейсы: rename, reindex, cleanup stale refs.

---

## Термины урока

- `Repository` — это каталог проекта с Git-историей и служебными данными.
- `Remote` — это удаленная копия репозитория, с которой ты синхронизируешься.
- `Upstream branch` — это удаленная ветка, с которой связана текущая локальная ветка.
- `Локальный конфиг` — это Git-конфиг, который действует только внутри текущего репозитория.
- `Index` — это staged-слой перед коммитом.
- `Atomic commit` — это коммит с одним понятным смыслом.

---

## Реализация через инструмент

### Git как конкретная реализация VCS

`Git` — это распределенная VCS. У тебя локально есть не только рабочая папка, но и история изменений. Поэтому важны не только команды `push/pull`, но и локальная дисциплина:

- читать `git status` и `git diff`, а не гадать;
- собирать коммит осознанно через index;
- настраивать поведение репозитория под правила команды, а не жить в дефолтах вслепую.

### Где и как хранится Git-конфиг

У Git есть три уровня конфигурации:

```bash
git config --system
git config --global
git config --local
```

Практическая логика такая:

- `--system` почти не трогаешь руками;
- `--global` подходит для личных настроек на машине;
- `--local` нужен, когда конкретный репозиторий живет по своей политике.

Чтобы понять, откуда вообще приехало текущее значение, полезно начинать с:

```bash
git config --list --show-origin
```

Что часто имеет смысл держать именно локально:

```bash
git config --local pull.rebase true
git config --local fetch.prune true
git config --local rebase.autoStash true
git config --local merge.conflictStyle zdiff3
```

Дополнительно локальный `user.name` и `user.email` полезны, если ты работаешь с несколькими аккаунтами или разными корпоративными репозиториями.

Живой кейс:

у тебя на машине есть рабочий репозиторий, где команда живет через `rebase`, и личный проект, где тебе удобнее обычный `merge`. Если ты бездумно прописал `pull.rebase=true` глобально, то внезапно поменял поведение обоих проектов сразу. Именно для таких случаев и нужен `git config --local`.

Важно:

- не копируй чужой конфиг без понимания, что именно меняется;
- `pull.rebase` должен совпадать с командной политикой;
- настройки вроде `core.autocrlf` или `filemode` без контекста лучше не дергать "по гайду из интернета".

### Базовый ежедневный цикл без магии

На практике безопасный цикл выглядит так:

```bash
git switch -c feature/header-search
git status
git diff
git add src/components/Header.tsx
git diff --cached
git commit -m "add header search field"
git fetch origin
git rebase origin/main
git push -u origin feature/header-search
```

Что здесь важно:

- `git diff` проверяет, что ты вообще меняешь;
- `git diff --cached` проверяет именно будущий коммит, а не весь working tree;
- `git fetch` + `git rebase origin/main` обычно прозрачнее, чем бездумный `git pull`;
- `git push -u` связывает локальную ветку с upstream и снимает часть рутины дальше.

Если изменений много и они разнородные, полезно не жать `git add .` на автомате, а добавлять файлы точечно или через `git add -p`.

### Операционные кейсы, которые реально случаются в проектах

Это уже не теория, а типовые рабочие проблемы:

1. Изменился `.gitignore` или Git потерял корректную индексацию пути:

```bash
git rm -r --cached <path>
```

Команда удаляет путь из индекса, но не с диска. После этого Git пересобирает представление о файлах при следующем `add`.

1. Файл или папка переехали, а инструменты начали путаться в путях:

```bash
git mv old/path new/path
```

Это безопаснее, чем надеяться, что IDE и Git одинаково поймут переименование во всех окружениях.

1. Локально висят устаревшие ссылки на удаленные ветки после удаления веток на сервере:

```bash
git fetch --prune
```

Это не "оптимизация", а нормальная гигиена репозитория.

1. Репозиторий подозрительно ведет себя на уровне объектов и индекса:

```bash
git fsck
git gc --aggressive
```

Это уже не ежедневный цикл, а операция по диагностике и уборке. Не надо превращать ее в ритуал "на всякий случай".

---

## Границы применимости инструмента

- Локальный конфиг помогает подстроить репозиторий под процесс команды, но не заменяет командный стандарт.
- `git rm --cached` и `git mv` решают конкретные проблемы индекса и путей, но требуют внимательно читать `git status` перед коммитом.
- `git gc --aggressive` не должен становиться универсальной таблеткой от любой странности в репозитории.

---

## Альтернативы кратко

- IDE-кнопки и GUI-клиенты могут быть удобными, но они не отменяют понимание working tree, index и upstream branch.
- GitHub Desktop, SourceTree и аналоги нормальны как интерфейс, если ты понимаешь, какие Git-операции они делают под капотом.

---

## Production failure modes

- разработчик слепо ставит глобальный конфиг с чужого ноутбука и ломает поведение всех репозиториев на машине;
- `git add .` случайно тащит в коммит мусор, который никто не собирался ревьюить;
- файл переехал, но индекс и инструменты это поняли по-разному, и проект начинает "терять" модули;
- устаревшие ссылки на удаленные ветки создают иллюзию, что локально все еще живут ветки, которых давно нет на remote.

---

## Anti-pattern

"Один раз что-то настроил глобально, и теперь любой репозиторий должен жить по тем же правилам".

Обычно это заканчивается тем, что проблемы одного проекта расползаются на все остальные, а сам конфиг превращается в магию без владельца.

---

## 🔧 Практика

1. Выполни `git config --list --show-origin` и объясни, какие настройки пришли из `global`, а какие из `local`.
1. Настрой локально хотя бы `fetch.prune`, `pull.rebase` и `merge.conflictStyle`, а потом объясни, зачем каждая из этих опций нужна именно в репозитории.
1. Возьми один набор изменений и собери atomic commit через `git status`, `git diff`, `git add`, `git diff --cached`, `git commit`.
1. Смоделируй переименование файла и покажи, как бы ты проверил, что Git и проект одинаково поняли перемещение.

## ❓ Вопросы для самопроверки

1. В каком случае настройка должна жить в `--global`, а в каком лучше оставить ее в `--local`?
1. Почему `git diff --cached` полезнее, чем просто смотреть на `git status` перед коммитом?
1. Чем `git mv` практичнее ручного перемещения файла в проектах со сборкой и импортами?
1. Почему `git fetch --prune` полезен не "для красоты", а для реальной гигиены репозитория?
