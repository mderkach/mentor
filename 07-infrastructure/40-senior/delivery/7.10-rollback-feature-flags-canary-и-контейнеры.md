## 🎯 Что ты освоишь

- пройти инструментальную реализацию без смешения с общей инженерной моделью;
- увидеть, где конкретный инструмент помогает, а где создает ограничения;
- использовать core-слой как базу, а не как кусок внутри tool-урока.

---

## Термины урока

- `Rollback` — это возврат к предыдущему стабильному состоянию.
- `Progressive delivery` — это постепенная доставка изменения с контролем риска.
- `Canary` — это выкладка на часть трафика, пользователей или окружений.
- `Feature flag` — это runtime-переключатель поведения отдельно от факта деплоя.
- `Version skew` — это ситуация, когда разные части системы видят разные версии артефактов или API.

---

## Реализация через инструмент

На практике delivery strategy часто выражается через:

- feature flag systems;
- canary approaches;
- Docker/container delivery;
- static hosting vs container runtime vs SSR runtime.

```dockerfile
FROM node:22-alpine AS build
WORKDIR /app
COPY package.json pnpm-lock.yaml ./
RUN corepack enable && pnpm install --frozen-lockfile
COPY . .
RUN pnpm build

FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
```

Этот пример полезен не потому, что "Docker обязателен", а потому что показывает:

- build и runtime можно развести;
- итоговый образ можно сделать тонким;
- delivery contract можно сделать воспроизводимым.

---

## Границы применимости инструмента

- feature flags полезны для risk management, но только если у команды есть lifecycle cleanup;
- canary без метрик и наблюдаемости почти превращается в театр безопасности;
- containerization полезна, когда реально решает reproducible delivery или runtime packaging, а не потому что "так серьезнее";
- static hosting, container runtime и SSR runtime — это разные delivery modes, а не три варианта одного и того же.

---

## Альтернативы кратко

- Feature flags можно реализовывать разными системами, но смысл всегда один: управлять поведением после деплоя.
- Контейнер — не единственный способ доставки frontend: часто хватает static hosting или platform runtime.
- Canary бывает по трафику, по окружению, по user cohort — важно не название, а measured blast radius.

---

## Production failure modes

- rollback невозможен из-за CDN/cache/version skew;
- flags копятся без cleanup и усложняют поведение системы;
- canary запускают без метрик и четкого критерия отката;
- контейнер используют как ритуал, но runtime config injection остается хаотичной;
- команда говорит "релиз", имея в виду просто успешный deploy job.

---

## Anti-pattern

"Если что-то пойдет не так, быстро поправим новой выкладкой".

Это не стратегия. Это отсутствие стратегии.

---

## 🔧 Практика

1. Опиши текущий delivery path проекта: build artifact, config injection, hosting/runtime, rollout, rollback.
1. Назови, где у вас реально rollback, а где только надежда "быстро перезальем".
1. Для одной фичи предложи rollout strategy: direct release, flag, canary или staged rollout.
1. Определи один frontend-specific риск version skew и способ его уменьшить.
