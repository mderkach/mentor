# 2.9 Адаптивность, responsive и units

Адаптивность — это не "ужмём десктоп до мобильного", а проектирование интерфейса под разные устройства и сценарии.

---

## Что уже должно быть знакомо

- базовый layout через [2.4 Flexbox](../10-intern/2.4-flexbox.md) и [2.5 Grid и двумерные сетки](./2.5-grid-и-двумерные-сетки.md);
- смысл `viewport` и мобильной среды из [1.1.2 Мета-теги, title, charset, viewport](../../01-html/10-intern/1.1.2-Мета-теги-title-charset-viewport.md).

---

## 🎯 Что ты освоишь

- mobile-first подход;
- выбор единиц измерения (`px`, `rem`, `%`, `vw`, `dvh`);
- медиазапросы без хаоса;
- `clamp()` для гибкой типографики и отступов;
- базу container queries и container units (`cqi/cqb`).

---

## Что важно понять

- Responsive-дизайн строится от поведения контента, а не от списка популярных устройств.
- Mobile-first нужен не ради моды, а чтобы базовый интерфейс был собран в более жестких ограничениях.
- Единицы измерения и брейкпоинты надо выбирать как инженерные инструменты, а не как привычные значения из прошлого проекта.

---

## Термины урока

- `Mobile-first` — это подход, где базовые стили пишутся под мобильный экран, а дальше расширяются под большие размеры.
- `Breakpoint` — это точка, где layout нужно перестроить из-за поведения контента.
- `Fluid value` — это гибкое значение (`clamp`, `%`, `vw`), которое масштабируется без жёсткой привязки к одному экрану.

---

## Mobile-first

Пиши базовые стили для мобильного, затем расширяй под большие экраны.

```css
.container {
  padding: 16px;
}

@media (min-width: 768px) {
  .container {
    padding: 24px;
  }
}
```

Так проще поддерживать и меньше конфликтов.

---

## Container queries и container units

Иногда компонент живёт в разных колонках страницы, и `@media` по viewport уже не хватает.

```css
.product-list {
  container-name: cards;
  container-type: inline-size;
}

@container cards (min-width: 42rem) {
  .product-card {
    grid-template-columns: 1fr 2fr;
    gap: 4cqi;
  }
}
```

- `container-type` включает возможность контейнерных запросов;
- `container-name` помогает адресовать нужный контейнер;
- `cqi/cqb` масштабируются от размеров контейнера.

Больше о modern CSS-контексте и поддержке: [2.10 Современный CSS 2024-2026](../30-middle/2.10-современный-css-2024-2026.md).

---

## Units: что когда

- `px` — точные значения для границ, иконок, мелких деталей.
- `rem` — размеры шрифта, отступы и масштабируемые размеры интерфейса.
- `em` — локальное масштабирование относительно родителя.
- `ch` — ограничение длины текстовой строки.
- `lh` — размеры от текущей высоты строки.
- `%` — относительные ширины/высоты внутри контейнера.
- `vw/vh` — привязка к viewport.
- `dvh/svh/lvh` — более предсказуемая высота на мобильных.

Практика:

- типографика и spacing чаще в `rem`;
- ширины компонентов часто `%`/`fr`/`minmax()`;
- большие герои и full-screen блоки аккуратно через `dvh`.

---

## `clamp()` для гибких значений

```css
.hero__title {
  font-size: clamp(1.75rem, 1.2rem + 2vw, 3rem);
}
```

`clamp(min, preferred, max)` позволяет не плодить лишние брейкпоинты.

---

## Брейкпоинты

Не копируй "магические" числа из чужих проектов.

Опирайся на:

- поведение контента;
- переломы layout;
- реальные устройства аудитории.

Чаще достаточно 2-4 ключевых точек.

---

## Медиа-контент в адаптиве

Для изображений и видео важны:

- `object-fit` (`cover`/`contain`);
- `object-position`;
- в табличных/inline-сценариях иногда нужен `vertical-align`.

```css
.media {
  inline-size: 100%;
  aspect-ratio: 4 / 3;
  object-fit: cover;
  object-position: center;
}
```

`Resource adaptation` на уровне frontend — это когда CSS-решения по layout согласованы с адаптацией ресурсов (изображений, шрифтов, приоритетов загрузки), а не живут отдельно.
Практика по ресурсам: [1.2.1 Адаптивные изображения и медиа](../../01-html/20-junior/1.2.1-Адаптивные-изображения-и-медиа.md) и [1.3.5 Resource Hints и приоритеты загрузки](../../01-html/30-middle/1.3.5-Resource-Hints-и-приоритеты-загрузки.md).

---

## Что проверять в DevTools

- 320 / 375 / 768 / 1024 / 1440;
- горизонтальный скролл;
- длину заголовков и кнопок на реальном контенте;
- состояние меню, форм, таблиц, карточек.

Если на мобильном начинают "дрожать" метрики и рендер, смотри урок про CSS-производительность: [2.13 CSS-производительность: критический рендер, contain, content-visibility](../30-middle/2.13-CSS-производительность-критический-рендер-contain-content-visibility.md).

---

## Частые ошибки

- Сначала верстают desktop, потом "ломают" всё на mobile.
- Везде используют `px`, из-за чего интерфейс плохо масштабируется.
- Дублируют медиазапросы без системы.
- Не проверяют промежуточные ширины между брейкпоинтами.

---

## Anti-pattern

"Сначала собираем desktop, а потом ужмем всё под mobile".

Именно так чаще всего и рождаются хрупкие адаптивные интерфейсы с бесконечными патчами.

---

## 🔧 Практика

1. Сделай mobile-first страницу с 3 брейкпоинтами.
1. Примени `clamp()` к заголовку и вертикальным отступам.
1. Прогони проверку на 5 ширинах и составь список багов.
