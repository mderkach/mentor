# 2.8 Трансформации, переходы и анимации

Анимация должна помогать понимать интерфейс: что изменилось, откуда появилось, куда исчезло. Если движение не объясняет действие, оно мешает.

- как учитывать `prefers-reduced-motion`;
- как управлять `transition-*` и `animation-*` без хаоса.

---

## Что важно понять

- Движение в интерфейсе должно объяснять изменение состояния, а не доказывать, что мы умеем в анимацию.
- У анимации есть цена и по производительности, и по доступности.
- Хороший motion чаще незаметен как эффект, но заметен как помощь в понимании интерфейса.

---

## `transform`: движение без перерасчёта layout

Чаще всего используем:

- `translate`;
- `scale`;
- `rotate`.

```css
.card:hover {
  transform: translateY(-4px);
}
```

Свойства `transform` и `opacity` обычно анимируются плавнее, чем `top/left/width/height`.

Больше о влиянии анимаций на метрики рендера: [2.13 CSS-производительность: критический рендер, contain, content-visibility](../30-middle/2.13-CSS-производительность-критический-рендер-contain-content-visibility.md).

---

## Точка трансформации и 3D-эффекты

- `transform-origin` задаёт точку, вокруг которой вращается/масштабируется элемент;
- `perspective`, `transform-style` и `backface-visibility` нужны для 3D-анимаций карточек.

```css
.scene {
  perspective: 900px;
}

.card {
  transform-origin: center top;
  transform-style: preserve-3d;
  transition: transform 300ms ease;
}

.scene:hover .card {
  transform: rotateY(14deg) translateZ(12px);
}

.card--flip {
  backface-visibility: hidden;
}
```

Практический принцип:
- движение через `transform` обычно дешевле, чем через `top/left`;
- 3D-эффекты делают UI выразительным, но легко перегрузить интерфейс и железо.

Разбор, почему это так в пайплайне рендера (layout/paint/composite, CPU/GPU): [2.13 CSS-производительность: критический рендер, contain, content-visibility](../30-middle/2.13-CSS-производительность-критический-рендер-contain-content-visibility.md).

## `transition`: для состояния элемента

Подходит для hover/focus/active.

```css
.button {
  transition: background-color 180ms ease, transform 180ms ease;
}

.button:hover {
  transform: translateY(-1px);
}
```

Не ставь `transition: all` без причины.

Чаще лучше явно:

```css
.chip {
  transition-property: transform, opacity;
  transition-duration: 160ms;
  transition-timing-function: ease-out;
  transition-delay: 0ms;
}
```

---

## `animation` и `@keyframes`

Используй, когда движение автономное или многошаговое.

```css
@keyframes pulse {
  0%,
  100% {
    transform: scale(1);
  }

  50% {
    transform: scale(1.03);
  }
}

.badge {
  animation: pulse 1.8s ease-in-out infinite;
}
```

Полезно знать отдельные параметры:

- `animation-name`;
- `animation-delay`;
- `animation-direction`;
- `animation-fill-mode`;
- `animation-play-state`.

Scroll-driven анимации (`view-timeline`, `animation-timeline`) оставь на middle+ и внедряй только с fallback.

---

## Accessibility: уважай пользователя

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

Минимум: отключай или упрощай лишние анимации при `reduced-motion`.

Если нужно смотреть на motion уже как на часть accessibility-контракта, а не только как на аккуратный UX, дальше есть отдельный урок: [2.14 CSS accessibility, system preferences и адаптивное взаимодействие](../30-middle/2.14-CSS-accessibility-system-preferences-и-адаптивное-взаимодействие.md).

## Частые ошибки

- Анимируют `width/height` там, где можно `transform`.
- Делают слишком долгие и "тяжелые" движения.
- Вешают бесконечную анимацию на важный контент.
- Забывают про `prefers-reduced-motion`.

---

## Anti-pattern

"Если интерфейс кажется скучным, добавим больше движения".

Скучный UI редко лечится анимацией, а перегруженный motion почти всегда ухудшает UX.

---

## 🔧 Практика

1. Сделай hover-анимацию карточки через `transform + transition`.
1. Добавь аккуратное появление модального окна.
1. Реализуй fallback для `prefers-reduced-motion`.

---

## ✅ Чеклист

- [ ] Отличаю `transition` от `animation`
- [ ] Для движения в основном использую `transform/opacity`
- [ ] Не использую `transition: all` без нужды
- [ ] Учитываю `prefers-reduced-motion`
