# 2.3 Поток документа, блочная модель и позиционирование

Большинство "непонятных" багов в верстке — это не магия, а проблемы потока, box model и неправильного `position`.

---

## Что уже должно быть знакомо

- база каскада и чтение стилей в DevTools: [0.0.1 CSS в DevTools: каскад, слои и layout-debug](../00-zero/0.0.1-CSS-в-DevTools-каскад-слои-layout-debug.md), [2.1 Базовые знания CSS](./2.1-базовые-знания-css.md).

---

## 🎯 Что ты освоишь

- как работает normal flow;
- как считать размеры блока через box model;
- когда использовать `relative`, `absolute`, `fixed`, `sticky`;
- как не ломать layout из-за `z-index` и переполнения;
- где полезны `aspect-ratio`, `inline-size`, `inset`;
- как писать RTL-safe отступы через `margin-inline`/`margin-block`.

---

## Что важно понять

- Поток документа — это базовое состояние страницы, а не «что-то, что сразу надо сломать позиционированием».
- `position`, `overflow` и размеры надо читать вместе: по отдельности они редко объясняют баг.
- Если не понимать box model и контекст позиционирования, дальше будут системно болеть flex, grid, sticky и слои интерфейса.

---

## Поток документа

По умолчанию элементы идут сверху вниз:

- блочные (`div`, `section`) занимают всю строку;
- строчные (`span`) живут внутри строки.

Если резко менять `position` или `float` без понимания потока, верстка "плывёт".

---

## Box model без сюрпризов

Размер блока = `content + padding + border + margin`.

С `box-sizing: border-box` проще:

- заданная `width` уже включает `padding` и `border`;
- меньше сюрпризов на адаптиве.

Полезно всегда проверять в DevTools вкладку `Computed`, если блок "не влез".

### Логические отступы для LTR/RTL

Если интерфейс может быть не только LTR, лучше сразу использовать логические свойства:

```css
.card {
  margin-block: 12px;
  margin-inline: 16px;
  padding-block: 12px;
  padding-inline: 16px;
  border-inline-start: 4px solid #1a73e8;
}
```

Что это даёт:

- не нужно вручную зеркалить `margin-left/right` для RTL;
- один и тот же CSS корректно работает для `dir="ltr"` и `dir="rtl"`;
- меньше условных "rtl-патчей" в проекте.

Связь с HTML-семантикой направления и `dir`: [1.3.8 Интернационализация в HTML: lang, dir, hreflang, bdi, bdo](../../01-html/30-middle/1.3.8-Интернационализация-в-HTML-lang-dir-hreflang-bdi-bdo.md).

## Позиционирование: что когда

- `position: relative` — база для смещения и точка отсчёта для `absolute`-потомков.
- `position: absolute` — для бейджей, иконок, локальных оверлеев внутри конкретного блока.
- `position: fixed` — элементы, привязанные к viewport (например, floating action button).
- `position: sticky` — "прилипание" в пределах родительского контейнера.

Пример:

```css
.card {
  position: relative;
}

.card__badge {
  position: absolute;
  top: 8px;
  right: 8px;
}

.sidebar {
  position: sticky;
  top: 16px;
}
```

Короткая подсказка по modern-синтаксису:

- вместо `top/right/bottom/left` можно использовать `inset`;
- для RTL/LTR-безопасности удобны `inset-inline` и `inset-block`.

### Что дороже для рендера при движении

- `top/left` чаще триггерят пересчёт layout и repaint;
- `transform` чаще ограничивается compositing и обычно работает плавнее;
- `fixed/sticky` удобны, но на скролле могут быть дороже по repaint.

Подробный разбор пайплайна (`critical rendering path`, `reflow/repaint/composite`, CPU/GPU): [2.13 CSS-производительность: критический рендер, contain, content-visibility](../30-middle/2.13-CSS-производительность-критический-рендер-contain-content-visibility.md).

## Размеры и пропорции без костылей

- `inline-size` и `block-size` — логические аналоги `width/height`;
- `aspect-ratio` — фиксирует пропорцию блока без JS и "padding-хаков".

```css
.thumb {
  inline-size: 100%;
  aspect-ratio: 16 / 9;
  object-fit: cover;
}
```

---

## `z-index` и контексты наложения

`z-index` работает только в своём stacking context.

Что часто ломает ожидания:

- `transform`, `opacity < 1`, `position` + `z-index` у предков могут создать новый контекст;
- `isolation: isolate` принудительно создаёт новый stacking context;
- поэтому "z-index: 9999" не всегда спасает.

Если нужно собрать это в production-картину вместе с clipping, `overflow`, sticky/fixed и nested scroll, дальше есть отдельный урок: [2.15 Stacking context, overflow, scroll и слои интерфейса](../30-middle/2.15-stacking-context-overflow-scroll-и-слои-интерфейса.md).

---

## Overflow и скролл

- `overflow: hidden` может обрезать тени/всплывашки;
- `overflow: auto` даёт локальный скролл;
- `overflow-x` и `overflow-y` помогают чинить только проблемную ось;
- `scrollbar-gutter: stable` уменьшает скачки layout при появлении скроллбара;
- `overscroll-behavior` контролирует "протекание" скролла в родителя;
- глобальный горизонтальный скролл обычно означает проблему ширин/отступов.

`contain` и `content-visibility` тоже влияют на поведение layout/рендера, но это уже middle-тема. Больше о ней: [2.13 CSS-производительность: критический рендер, contain, content-visibility](../30-middle/2.13-CSS-производительность-критический-рендер-contain-content-visibility.md).

## Частые ошибки

- Пытаются выровнять layout через `position: absolute` вместо flex/grid.
- Используют `fixed`, когда нужен `sticky`.
- Не понимают, от какого блока считается `absolute`.
- Лечат наложение через бесконечный рост `z-index`.
- В мультиязычном UI продолжают писать только `margin-left/right` и ловят RTL-баги.
- Двигают элементы через `top/left` в анимациях вместо `transform`.

---

## Anti-pattern

"Если layout не сходится, поставим `position: absolute` и руками подвинем".

Это почти всегда превращает нормальную верстку в хрупкий набор координат.

---

## 🔧 Практика

1. Сделай карточку с бейджем через `relative + absolute`.
1. Сделай колонку фильтров через `position: sticky`.
1. Найди баг с наложением слоёв и исправь через корректный stacking context.
