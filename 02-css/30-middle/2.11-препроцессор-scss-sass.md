# 2.11 SCSS/SASS: когда нужен, а когда modern CSS уже достаточно

SCSS всё ещё полезен, но его роль изменилась.
Раньше он закрывал дыры CSS, сегодня чаще выступает как compile-time слой для зрелого проекта, а не как дефолтный выбор "потому что так принято".

Перед внедрением SCSS полезно свериться с тем, что уже можно сделать нативно: [2.10 Современный CSS 2024-2026](./2.10-современный-css-2024-2026.md).

---

## Большая идея

SCSS сегодня не "обязательная часть CSS-стека", а инструмент с ценой.
Хороший вопрос звучит не "умеем ли мы в Sass", а "какую проблему он решает в этом проекте лучше, чем modern CSS и tokens".

---

## Что важно понять

- SCSS больше не должен появляться в проекте «по привычке».
- Это compile-time инструмент, который хорошо решает одни задачи и плохо подходит для других, особенно для runtime theming.
- Правильный уровень зрелости здесь не «знаю синтаксис Sass», а «умею обосновать, зачем он нужен или не нужен».

---

## Когда SCSS стоит использовать

Хорошие кейсы:

- большой проект с повторяющимися паттернами;
- legacy-кодовая база, где SCSS уже является рабочим контрактом;
- сложная кодовая база, где нужна compile-time модульность и генерация;
- проект, где хватает build-time токенов и не нужен сложный runtime-theming.

Когда можно обойтись нативным CSS:

- маленький лендинг;
- локальный виджет;
- проект без сложной темы и без долгого жизненного цикла;
- продукт, где modern CSS уже закрывает основную боль через `@layer`, nesting, custom properties и container queries.

---

## Практичный decision framework

### SCSS чаще оправдан

- когда кодовая база уже живёт на Sass и миграция не окупается;
- когда нужен compile-time слой для карт, циклов и генерации ограниченного числа утилит;
- когда команде важны модули `@use/@forward`, но нет отдельной styling-платформы.

### SCSS чаще не нужен по умолчанию

- когда проект стартует с нуля и основная боль уже закрывается modern CSS;
- когда ключевая задача проекта — runtime-темизация, theme switching и design tokens в браузере;
- когда команда и без того страдает от сложного каскада и длинных селекторов.

### Красные флаги

- SCSS используется как оправдание для глубокой вложенности;
- тема собирается через импорт разных SCSS-файлов вместо runtime-переключения токенов;
- миксины скрывают важные продуктовые решения и их уже нельзя прочитать из итогового CSS.

---

## Переменные: SCSS vs CSS custom properties

- SCSS-переменные (`$color-primary`) вычисляются на этапе сборки.
- CSS-переменные (`--color-primary`) живут в runtime и могут меняться динамически.

Практика в production:

- статические вычисления -> SCSS;
- темизация и runtime-переключения -> CSS custom properties.

Глубже про runtime-токены: [2.12 CSS Custom Properties и Design Tokens](./2.12-CSS-custom-properties-и-design-tokens.md).

Короткое правило:

- SCSS хорошо считает заранее;
- CSS custom properties хорошо живут в runtime.

Если проекту нужна полноценная theming-strategy на уровне React/Vue, монорепы или дизайн-системы, дальше смотри: [2.16 CSS в современном frontend stack](./2.16-CSS-в-современном-frontend-stack.md).

---

## Модульный Sass: `@use` и `@forward`

`@import` в Sass считается устаревшим подходом. Рабочая база в 2026 — модули:

```scss
// styles/_tokens.scss
$space-4: 1rem;

// styles/_index.scss
@forward 'tokens';

// component.scss
@use '../styles' as s;

.card {
  padding: s.$space-4;
}
```

- `@use` подключает модуль с неймспейсом;
- `@forward` собирает публичный API для других файлов.

---

## Миксины и функции

```scss
@mixin focus-ring($color) {
  outline: 2px solid $color;
  outline-offset: 2px;
}

.button:focus-visible {
  @include focus-ring(#0b57d0);
}
```

Используй миксины для повторяемых, осмысленных паттернов, а не для каждой пары свойств.

---

## Maps, циклы и генерация утилит

```scss
$spaces: (
  2: 0.5rem,
  4: 1rem,
  6: 1.5rem
);

@each $name, $value in $spaces {
  .u-m-#{$name} {
    margin: $value;
  }
}
```

Так можно генерировать утилиты, но держи это под контролем: без ограничений легко раздуть итоговый CSS.

`@for` и `@each` в проде встречаются часто, `@while` — редко и обычно только в очень специфичных генераторах.

Хороший вопрос перед генерацией:
"мы реально уменьшаем ручной труд или просто фабрикуем сотни классов заранее?"

---

## Вложенность: главная ловушка

Плохо:

- 4-6 уровней вложенности;
- селекторы длиной в полэкрана;
- зависимость от хрупкой структуры DOM.

Хорошо:

- 1-2 уровня вложенности;
- компонентные классы;
- читаемые селекторы и низкая специфичность.

Если без 4-5 уровней вложенности компонент "не собирается", проблема обычно уже не в отсутствии Sass, а в архитектуре HTML/CSS.

---

## Структура файлов

Рабочий минимум:

- `abstracts/` (переменные, миксины, функции);
- `base/` (reset, typography);
- `components/`;
- `layouts/`;
- `utilities/`.

Важно: структура должна совпадать с архитектурой проекта, а не быть "ради моды".

Если нужно зафиксировать архитектуру CSS-слоёв на уровне команды, смотри: [2.17 CSS-архитектура команды: слои, соглашения, масштабирование](../40-senior/2.17-CSS-архитектура-команды-слои-соглашения-масштабирование.md).

---

## Source maps и дебаг

Компилируемый SCSS без source maps быстро превращается в боль при отладке. Включай source maps в dev-сборке, чтобы DevTools показывал исходный `*.scss`, а не только итоговый CSS.

---

## Где SCSS начинает вредить

- когда переменные, миксины и maps дублируют систему design tokens;
- когда runtime-theme пытаются симулировать через пересборку разных stylesheet;
- когда вложенность и `&`-магия важнее читаемости итогового селектора;
- когда команда больше обсуждает структуру partial-файлов, чем контракт самих компонентов.

SCSS должен упрощать поддержку, а не прятать архитектурный долг под compile-time синтаксис.

---

## Частые ошибки

- Используют SCSS как оправдание для сложных селекторов.
- Дублируют токены в SCSS и CSS-переменных без стратегии.
- Пытаются сделать runtime-тему только на Sass.
- Прячут важные решения в миксины, которые никто не понимает.
- Не контролируют итоговый размер скомпилированного CSS.

---

## Anti-pattern

"Подключим SCSS на старте проекта, а там разберемся, пригодится ли".

По умолчанию это чаще добавляет слой сложности, чем реальную инженерную пользу.

---

## 🔧 Практика

1. Вынеси цвета/spacing в токены и подключи их в компонентах.
1. Напиши 2-3 миксина для повторяющихся паттернов (`focus`, `truncate`, `visually-hidden`).
1. Сократи вложенность в проблемном модуле до читаемого уровня.
1. Для текущего проекта письменно ответь: почему здесь SCSS оправдан или почему пора от него уходить.
