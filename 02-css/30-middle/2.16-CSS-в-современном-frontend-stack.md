# 2.16 CSS в современном frontend stack

Хороший CSS как web platform навык уже недостаточен, если ты работаешь в React/Vue-продукте, дизайн-системе, монорепе или microfrontend-ландшафте.
Здесь важен не только синтаксис, но и то, где живут стили, кто ими владеет и как они масштабируются.

---

## 🎯 Что ты освоишь

- различия между CSS Modules, CSS-in-JS, vanilla-extract, SCSS и utility CSS;
- как стили живут рядом с компонентами в React/Vue;
- где проходит граница между токенами, компонентными стилями и локальными утилитами;
- как выбирать стратегию theming для дизайн-системы;
- что меняется в monorepo, microfrontends и Module Federation.

## Большая идея

Стратегия стилей отвечает на три вопроса:

1. Где живут стили: рядом с компонентом, в глобальном слое, в токенах, в utility-классе?
1. Когда вычисляются стили: на build-time, в runtime, на сервере, в браузере?
1. Как контролируются границы: именами классов, CSS scoping, токенами, архитектурой слоев?

---

## Что важно понять

- Выбор styling-подхода почти никогда не про вкус команды, а про ownership, runtime-cost и границы системы.
- Один и тот же инструмент может быть удачным в одном продукте и разрушительным в другом.
- Этот урок не про «назвать победителя», а про то, чтобы научиться принимать архитектурное решение по стилям осознанно.

---

## Краткая карта подходов

### CSS Modules

Хорошо подходят, когда:

- нужен локальный scope классов;
- команда хочет обычный CSS без runtime-магии;
- проект компонентный, но без сложной styling-платформы.

Слабые стороны:

- темизация и shared-variants быстро требуют отдельной системы токенов;
- глобальные правила и cross-cutting layers все равно надо проектировать отдельно.

### CSS-in-JS

Хорошо подходит, когда:

- стили завязаны на runtime state и variants;
- команда сознательно принимает runtime/build-time компромиссы;
- экосистема уже построена вокруг конкретной библиотеки.

Слабые стороны:

- выше стоимость инфраструктуры и дебага;
- легко получить styling-энтропию, если нет design-token discipline.

### vanilla-extract

Хорошо подходит, когда:

- нужен typed styling и build-time extraction;
- важны токены, theme contracts и более строгая инженерная модель;
- команда готова к более системному подходу.

Слабые стороны:

- входной порог выше обычного CSS;
- не в каждом продукте окупает сложность.

### SCSS

Хорошо подходит, когда:

- нужен compile-time слой поверх CSS;
- проект legacy или большой, но без отдельной styling-платформы;
- есть повторяемые паттерны и модульная структура Sass.

Слабые стороны:

- не решает runtime theming сам по себе;
- легко превращается в вложенный селекторный лес.

### Utility CSS

Хорошо подходит, когда:

- команде нужен быстрый и единообразный способ собирать UI;
- есть четкие design constraints;
- продукт живет в высокой скорости поставки.

Слабые стороны:

- можно потерять смысловые границы компонентов;
- при слабой дисциплине HTML превращается в носитель всех styling-решений сразу.

---

## Как выбирать стратегию

- Если приоритет — локальность и простота, чаще выигрывают CSS Modules.
- Если приоритет — design tokens, typed theming и build-time discipline, чаще выигрывает vanilla-extract.
- Если проект уже живет на Sass и не испытывает боли, SCSS может оставаться рабочим слоем, но не по умолчанию.
- Если стиль тесно завязан на runtime state и variants, CSS-in-JS может быть оправдан, но только с ясной ценой.
- Если продукту нужна высокая скорость сборки UI из ограниченного набора primitive-компонентов, utility CSS может быть рациональным.

Ключевая мысль:

- не существует "лучшего" инструмента вне контекста;
- существует стратегия, которая лучше совпадает с архитектурой продукта и зрелостью команды.

---

## Theming strategy для дизайн-системы

Практичный минимум:

- raw tokens и semantic tokens живут отдельно;
- runtime theme строится через CSS custom properties;
- компонент не должен знать конкретный hex, если он может опираться на semantic token;
- глобальная тема и локальные overrides должны иметь понятных владельцев.

Плохой сигнал:

- тема меняется через ручной импорт другого SCSS-файла;
- токены дублируются в JS, SCSS и CSS без единого источника истины.

---

## Частые ошибки

- выбирают styling tool до того, как описали требования продукта;
- смешивают несколько подходов без явной ownership-модели;
- недооценивают цену runtime-styling и дебага;
- пытаются решать theming без token-слоя и договоренностей о границах.

---

## Monorepo, microfrontends, Module Federation

Здесь стили ломаются не "из-за CSS", а из-за отсутствия контрактов.

Что нужно контролировать:

- глобальные reset/base styles;
- shared tokens и их versioning;
- naming/scoping стратегию;
- кто имеет право менять foundational layers;
- как избегать style bleed между приложениями.

Если продукта несколько, главный вопрос уже не "какой синтаксис удобнее", а "как стили живут между границами команд и сборок".

---

## Anti-pattern

"Выберем styling tool по вкусу команды и потом как-нибудь договоримся".

Потом обычно начинается смесь из глобального CSS, локальных хаков, дублированных токенов и конфликтующих owner-границ.

---

## 🔧 Практика

1. Возьми один реальный продукт или pet-project.
1. Опиши, где сейчас живут стили: global, component, utility, token layer.
1. Сравни текущий подход с двумя альтернативами из урока.
1. Сформулируй decision note: почему текущая стратегия подходит или почему пора мигрировать.
