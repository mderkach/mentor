# 2.12 CSS Custom Properties и Design Tokens

## Уровень

Middle

## Кратко о теме

Этот урок про систему токенов и runtime-переменных: как сделать стили масштабируемыми, а темы и брендовые вариации — управляемыми.

## Что уже должно быть знакомо

- каскад и базовая работа с CSS-переменными на уровне синтаксиса;
- границы между compile-time и runtime из [2.11 SCSS/SASS: когда нужен, а когда modern CSS уже достаточно](./2.11-препроцессор-scss-sass.md).

## Термины урока

- `Design token` — это именованная дизайн-константа (`color`, `spacing`, `radius`, `font-size`), используемая по всему продукту.
- `Semantic token` — это токен уровня смысла (`--color-text-primary`), а не конкретного цвета (`--blue-600`).
- `CSS custom property` — это переменная CSS, которая живёт в runtime и может переопределяться по контексту.

Для advanced-сценариев с типизацией и анимацией переменных смотри `@property`: [2.10 Современный CSS 2024-2026](./2.10-современный-css-2024-2026.md).

## Зачем это нужно на практике

- Сокращает дубли и случайные значения в стилях.
- Упрощает тему/брендинг без переписывания компонентов.
- Делает код-ревью по стилям предсказуемым.

## Что нужно знать

- Разницу между raw-токенами и semantic-токенами.
- Где задавать корневые токены (`:root`), где — контекстные (тема, компонент).
- Когда оставить значение инлайном, а когда вынести в токен.

## Что нужно уметь

- Строить минимальную токен-модель для реального проекта.
- Переопределять тему через атрибут/класс без каскадного хаоса.
- Мигрировать legacy-значения на токены поэтапно.

## Что важно понять

- Токены нужны не ради модного словаря, а ради управляемости, договоренностей и уменьшения случайных значений.
- Хорошая token-система разделяет сырые значения и смысловые роли.
- Если токены не помогают менять тему, бренд или слой компонентов предсказуемо, значит система спроектирована слабо.

## Практичный шаблон

```css
:root {
  --space-2: 0.5rem;
  --space-4: 1rem;
  --radius-md: 0.75rem;

  --color-surface: #ffffff;
  --color-text-primary: #0f172a;
  --color-border-default: #cbd5e1;
}

[data-theme='dark'] {
  --color-surface: #0b1220;
  --color-text-primary: #e2e8f0;
  --color-border-default: #334155;
}

.card {
  padding: var(--space-4);
  border-radius: var(--radius-md);
  color: var(--color-text-primary);
  border: 1px solid var(--color-border-default);
  background: var(--color-surface);
}
```

## Типичные ошибки

- Называют токены по цвету (`--red`) вместо роли (`--color-danger`).
- Смешивают SCSS-переменные и CSS-переменные без стратегии.
- Создают сотни токенов заранее без реального сценария.

Про границы ответственности SCSS и CSS-переменных подробнее здесь: [2.11 Препроцессор SCSS/SASS](./2.11-препроцессор-scss-sass.md).

## Anti-pattern

"Сделаем токены потом, пока пишем значениями".

Потом это превращается в дорогую миграцию и бесконечные дубли.

## 🔧 Практика

1. Возьми один экран и выпиши повторяющиеся цвета, отступы и радиусы.
1. Раздели найденное на raw-токены и semantic-токены.
1. Переведи один компонент на новые токены без смены его структуры.

## ✅ Чеклист

- [ ] Я различаю raw tokens и semantic tokens
- [ ] Я не называю токены по случайному цвету, если важна роль
- [ ] Я понимаю, где токен должен жить глобально, а где локально
- [ ] Я вижу, как токены помогают теме и ревью, а не просто заменяют литералы

## Как проверить понимание

1. Возьми 1 экран и инвентаризируй повторяющиеся цвета/spacing.
1. Вынеси минимум в raw + semantic tokens.
1. Подключи светлую/тёмную тему через переопределение токенов.
1. Убедись, что компонентный CSS не изменялся по структуре.

Признак освоения: изменение темы делается через токены, а не через ручную правку десятков компонентов.
