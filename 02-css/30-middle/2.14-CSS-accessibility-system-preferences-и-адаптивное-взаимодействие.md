# 2.14 CSS accessibility, system preferences и адаптивное взаимодействие

Хороший HTML еще не гарантирует доступный интерфейс.
CSS легко может добить UX: убрать фокус, перегрузить motion, сломать high-contrast режим или сделать hover единственным способом взаимодействия.

---

## 🎯 Что ты освоишь

- `:focus-visible` как базовый контракт интерактива;
- `prefers-reduced-motion`, `prefers-contrast`, `forced-colors`;
- `color-scheme` и `accent-color` без самообмана;
- `hover` / `pointer` media queries для разных типов устройств;
- как уважать системные настройки пользователя, а не бороться с ними.

## Рабочий пример

```css
:root {
  color-scheme: light dark;
  accent-color: #1a73e8;
}

.button {
  transition: transform 160ms ease, background-color 160ms ease;
}

.button:focus-visible {
  outline: 2px solid currentColor;
  outline-offset: 3px;
}

@media (hover: hover) and (pointer: fine) {
  .button:hover {
    transform: translateY(-1px);
  }
}

@media (prefers-reduced-motion: reduce) {
  .button {
    transition: none;
  }
}

@media (forced-colors: active) {
  .button {
    border: 1px solid ButtonText;
  }
}
```

---

## Что важно понять

- CSS accessibility не ограничивается `outline: none`.
- Пользовательские предпочтения и системные режимы надо считать входными данными, а не "неудобным edge case".
- `:hover` не универсален: на touch-устройствах его может не быть, а у части пользователей вообще нет точного указателя.
- `forced-colors` и high-contrast режимы могут переопределить палитру. Это не баг, а попытка системы сделать интерфейс читаемым.
- `color-scheme` и `accent-color` полезны, но не заменяют нормальную систему цветов и контрастов.

---

## Частые ошибки

- Убирают фокусный outline и не дают замену.
- Прячут важные состояния только в hover.
- Делают motion обязательной частью понимания интерфейса.
- Красиво стилизуют light/dark, но игнорируют high contrast и forced colors.
- Пытаются "перекрасить обратно" системные режимы вместо того, чтобы работать с ними.

---

## Термины по-человечески

- `:focus-visible` — показываем фокус там, где он реально нужен пользователю, особенно при клавиатурной навигации.
- `prefers-reduced-motion` — пользователь просит меньше движения.
- `prefers-contrast` — пользователь предпочитает более высокий или более низкий контраст.
- `forced-colors` — система сама задает ограниченную палитру для лучшей читаемости.
- `hover` / `pointer` — характеристики устройства ввода: умеет ли оно в hover и насколько точен указатель.

---

## Практичный decision framework

### Когда правило почти обязательное

- `:focus-visible` — для любого кликабельного UI;
- `prefers-reduced-motion` — если у тебя есть заметные переходы, анимации, parallax, scroll effects;
- `hover` / `pointer` — если hover-стили влияют не только на косметику, но и на поведение.

### Когда это особенно важно

- `forced-colors` и `prefers-contrast` — для B2B, enterprise, госпродуктов, кабинетов и долгоживущих интерфейсов;
- `color-scheme` — если ты поддерживаешь light/dark и хочешь, чтобы нативные элементы не выбивались;
- `accent-color` — когда нужны аккуратные нативные чекбоксы, radio, progress без кастомной обвязки.

### Чего не делать

- не строй критичный UX только на `hover`;
- не делай motion единственным способом показать изменение состояния;
- не ломай системные режимы ради "идеального пикселя".

---

## Anti-pattern

"Доступность в CSS — это просто не убирать outline".

Нет. Это только минимальный старт. Реальная задача — уважать способы ввода, пользовательские настройки и системные режимы.

---

## 🔧 Практика

1. Возьми реальный экран с кнопками, формой и анимацией.
1. Проверь клавиатурный фокус через `:focus-visible`.
1. Прогони интерфейс с `prefers-reduced-motion` и `forced-colors`.
1. Убери hover-зависимости там, где без них теряется смысл.

---

## ✅ Чеклист

- [ ] У интерактива есть понятный `:focus-visible`
- [ ] Motion можно упростить через `prefers-reduced-motion`
- [ ] UI не зависит только от `hover`
- [ ] Системные high-contrast/forced-colors режимы не ломают интерфейс
- [ ] Я понимаю разницу между light/dark theming и accessibility preferences
