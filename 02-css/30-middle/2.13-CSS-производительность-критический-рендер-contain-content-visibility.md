# 2.13 CSS-производительность: критический рендер, contain, content-visibility

## Уровень

Middle

## Кратко о теме

CSS напрямую влияет на скорость первого экрана и стабильность интерфейса. Этот урок про то, как ускорять рендер без «магических твиков».

Если нужна целостная картина браузерного пайплайна до CSS-деталей, начни с: [4.0 Браузер: от URL до пикселя (network, fetch, render)](../../04-dom-и-browser/30-middle/4.0-браузер-от-url-до-пикселя-network-fetch-render.md).

## Что уже должно быть знакомо

- базовый layout и позиционирование из [2.3 Поток документа, блочная модель и позиционирование](../10-intern/2.3-поток-документа-блочная-модель-и-позиционирование.md);
- анимации и transform из [2.8 Трансформации, переходы и анимации](../20-junior/2.8-трансформации-переходы-и-анимации.md);
- базовая диагностика в [0.0.1 CSS в DevTools: каскад, слои и layout-debug](../00-zero/0.0.1-CSS-в-DevTools-каскад-слои-layout-debug.md).

## Термины урока

- `Critical Rendering Path (CRP)` — это путь от HTML/CSS до первого отрисованного пикселя.
- `Reflow (layout)` — это пересчёт геометрии элементов: размеров и позиций.
- `Repaint` — это перерисовка пикселей без пересчёта геометрии.
- `Composite` — это сборка готовых слоёв в финальный кадр.
- `Critical CSS` — это минимальные стили, нужные для первого экрана.
- `contain` — это свойство изоляции части рендера, чтобы уменьшить область перерасчётов.
- `content-visibility` — это отложенный рендер невидимого контента до появления в viewport.

## Зачем это нужно на практике

- Улучшает LCP/CLS и стабильность интерфейса по Core Web Vitals. Подробно про метрики и DevTools: [1.3.7 Core Web Vitals в HTML-практике](../../01-html/30-middle/1.3.7-Core-Web-Vitals-в-HTML-практике.md).
- Снижает стоимость layout/paint на длинных страницах.
- Делает интерфейс отзывчивее на слабых устройствах.

## Что нужно знать

- Стадии критического пути рендера: `DOM -> CSSOM -> Render Tree -> Layout -> Paint -> Composite`.
- Почему большой CSS-файл может тормозить первый рендер.
- Когда `contain` и `content-visibility` помогают, а когда ломают UX.
- Как связать изменения CSS с Web Vitals.
- Почему одни CSS-изменения грузят CPU сильнее, а другие чаще уходят в compositing (и часто в GPU).

## Что нужно уметь

- Выделять критичные стили и откладывать второстепенные.
- Применять `contain` для локальных виджетов.
- Безопасно использовать `content-visibility` для контента ниже первого экрана.
- Отличать изменения, которые вызывают reflow/repaint, от изменений на уровне composite.
- Находить в DevTools, где проблема: layout, paint или compositing.

## Что важно понять

- CSS-производительность нельзя улучшать по ощущениям: только через наблюдаемый эффект и замеры.
- Не каждое «тяжелое» свойство вредно всегда, и не каждое «GPU-friendly» свойство бесплатно.
- Middle-уровень здесь не в количестве терминов, а в умении связать изменение стилей с конкретной метрикой и стадией рендера.

## Критический путь рендера (CRP)

Упрощённо браузер делает так:
1. Парсит HTML в DOM.
2. Парсит CSS в CSSOM.
3. Собирает render tree.
4. Считает layout (геометрия).
5. Делает paint (пиксели).
6. Делает composite (сборка слоёв).

Почему это важно:
- пока нет нужного CSS, первый экран может не отрисоваться;
- чем больше дорогих layout/paint-операций, тем хуже отзывчивость и скролл.

## Reflow, Repaint, Composite: что обычно триггерит

- Чаще `layout/reflow`: `width/height`, `margin/padding`, `top/left/right/bottom`, изменения потока и позиционирования.
- Чаще `paint`: `color`, `background`, `box-shadow`, `border-radius`, `filter`.
- Чаще `composite`: `transform`, `opacity`.

Ключевая мысль: `composite` обычно дешевле, но не бесплатен.

## Позиционирование и стоимость рендера

- `relative/static` + частые геометрические изменения -> риск лишних reflow.
- `absolute` может уменьшить влияние на соседей, но layout всё равно может пересчитываться в своём контексте.
- `fixed/sticky` удобны для UX, но на скролле могут увеличивать repaint-нагрузку.
- Для анимационного движения почти всегда лучше `transform: translate(...)`, чем `top/left`.

## CPU и GPU без мифов

- Layout и большая часть paint-работы — в CPU.
- Composite часто использует GPU, особенно для transform/opacity-слоёв.
- Перевод всего подряд в отдельные слои (`will-change` везде) увеличивает память и может ухудшить производительность.
- Тяжёлые эффекты (`backdrop-filter`, большие blur/тени) могут быть дорогими даже при compositing.

## Где смотреть это в DevTools

1. `Performance` запись: смотри доли `Layout`, `Paint`, `Composite Layers`.
2. `Rendering`/`Paint flashing`: видно, что часто перерисовывается.
3. `Layers`: видно, какие элементы ушли в отдельные слои.
4. Сравни до/после изменения на одной и той же странице и сценарии.

## Практичный шаблон

```css
/* Критичные стили первого экрана */
.hero {
  min-height: 60dvh;
  display: grid;
  place-items: center;
}

/* Низ страницы: можно отложить рендер */
.feed-section {
  content-visibility: auto;
  contain-intrinsic-size: 800px;
}

/* Изолированный виджет */
.widget {
  contain: layout paint;
}
```

## Типичные ошибки

- Добавляют `content-visibility` на интерактивные блоки первого экрана.
- Ставят `contain: strict` без понимания побочных эффектов.
- Меряют только Lighthouse и игнорируют field-данные.
- Анимируют `top/left/width/height` там, где нужен `transform/opacity`.
- Считают, что "GPU всё решит", и не смотрят реальные профили в DevTools.

## Anti-pattern

"Добавим contain везде — будет быстрее".

Без замеров это легко ломает ожидаемое поведение layout и не даёт реальной выгоды.

## 🔧 Практика

1. Возьми страницу с длинным контентом и hero-блоком.
1. Найди один CSS-кандидат на улучшение первого экрана и один кандидат ниже фолда.
1. Примени точечные изменения через `content-visibility`, `contain` или оптимизацию критичных стилей.
1. Сравни результат в `Performance` и зафиксируй, что именно изменилось.

## ✅ Чеклист

- [ ] Я понимаю, какие CSS-изменения чаще бьют по layout, paint и composite
- [ ] Я не внедряю perf-трюки без замеров
- [ ] Я различаю оптимизацию первого экрана и оптимизацию длинного контента
- [ ] Я умею показать эффект через DevTools, а не только описать его словами

## Как проверить понимание

1. Замерь baseline LCP/CLS на реальной странице.
1. Выдели критичные и некритичные CSS-блоки.
1. Примени 1-2 изменения (`critical css`, `content-visibility`, `contain`).
1. Проверь в `Performance`, уменьшились ли `Layout/Paint` тайминги.
1. Сравни метрики и зафиксируй эффект.

Признак освоения: ты показываешь измеримый результат, а не «кажется стало быстрее».
