# 2.17 CSS-архитектура команды: слои, соглашения, масштабирование

## Уровень

Senior / Senior+

## Кратко о теме

На senior-уровне CSS — это не набор свойств, а архитектурные правила, которые позволяют команде изменять продукт без каскадных аварий.

## Что уже должно быть знакомо

- modern CSS и `@layer` из [2.10 Современный CSS 2024-2026](../30-middle/2.10-современный-css-2024-2026.md);
- token-layer и theming из [2.12 CSS Custom Properties и Design Tokens](../30-middle/2.12-CSS-custom-properties-и-design-tokens.md);
- styling strategy внутри продукта из [2.16 CSS в современном frontend stack](../30-middle/2.16-CSS-в-современном-frontend-stack.md).

## Термины урока

- `CSS-архитектура` — это договорённости о структуре, слоях и правилах стилей в проекте.
- `Cascade layers` — это уровни приоритета (`@layer`), которые фиксируют порядок каскада как контракт.
- `Utility class` — это малый класс с одной задачей (например, `u-mt-16`) для типовых локальных правок.

Если команда ещё не выровняла базу по modern CSS-фичам, начни с этого: [2.10 Современный CSS 2024-2026](../30-middle/2.10-современный-css-2024-2026.md).

## Зачем это нужно на практике

- Снижает число style-regressions между фичами.
- Ускоряет ревью и onboarding новых разработчиков.
- Делает рост проекта предсказуемым.

## Что нужно знать

- Как разделять стили на reset/base/components/utilities.
- Где граница между компонентными стилями и утилитами.
- Как фиксировать правила в документации и проверках.

Токены как отдельный слой лучше заранее согласовать с дизайн-системой: [2.12 CSS Custom Properties и Design Tokens](../30-middle/2.12-CSS-custom-properties-и-design-tokens.md).

## Что нужно уметь

- Спроектировать архитектуру слоёв под конкретный продукт.
- Внедрить naming/style conventions без релизного шока.
- Управлять миграцией legacy-CSS по этапам.

## Что важно понять

- CSS-архитектура не существует в виде абстрактной «красивой схемы»: она должна быть объяснима, внедряема и проверяема в ревью.
- На senior-уровне важны не только правила слоев, но и owners, миграция, исключения и границы между командами.
- Если команда не может ответить, где должен жить новый стиль и почему, архитектуры по факту нет.

## Базовый каркас

```css
@layer reset, base, tokens, components, utilities;

@layer reset {
  * { box-sizing: border-box; }
}

@layer tokens {
  :root {
    --space-4: 1rem;
  }
}

@layer components {
  .card { padding: var(--space-4); }
}

@layer utilities {
  .u-hidden { display: none !important; }
}
```

## Типичные ошибки

- Вводят архитектуру «на словах», без реальных правил и примеров.
- Позволяют любому стилю лежать в любом слое.
- Смешивают дизайн-токены, компонентные стили и хак-утилиты в одном файле.

## Anti-pattern

"Давайте просто договоримся писать аккуратно".

Без фиксированных слоёв и quality gates это не работает на масштабе.

## 🔧 Практика

1. Составь карту текущих CSS-слоев в одном реальном проекте.
1. Отметь, где смешаны tokens, components, overrides и utilities.
1. Предложи минимальный migration plan без остановки разработки.
1. Зафиксируй owner-правила: кто может менять foundation-слои и как проходят исключения.

## ✅ Чеклист

- [ ] Я могу описать CSS-архитектуру проекта не общими словами, а конкретными слоями и владельцами
- [ ] Я понимаю, как внедрять архитектуру поэтапно, а не «переписать всё сразу»
- [ ] Я различаю командные договоренности и реально enforceable правила
- [ ] Я вижу, где CSS-архитектура пересекается с design system и product architecture

## Как проверить понимание

1. Составь карту текущих CSS-слоёв в проекте.
1. Выдели 3 источника конфликтов каскада.
1. Предложи миграцию на `@layer` в 2-3 итерации.
1. Зафиксируй правила ревью по слоям.

Признак освоения: команда перестает спорить «почему стиль победил», потому что порядок предопределён архитектурой.
