# 🌐 4. DOM и Browser

В этом разделе разбирается, как работает браузер: `DOM`, события, браузерные API, сеть, observers, workers и DevTools.

Здесь важно не запоминать API списком, а понимать картину целиком: как страница загружается, как код меняет интерфейс, откуда берутся события, где узкое место и почему браузер ведёт себя именно так.

## Как проходить раздел

- Раздел сознательно начинается с `Junior`: без базы по [HTML](../01-html/README.md), [CSS](../02-css/README.md) и [JavaScript](../03-javascript/README.md) браузерный слой быстро превращается в магию и случайные хаки.
- Идти по ядру так: **Junior -> Middle -> Senior**.
- Если плавает общая картина `network -> parse -> execute -> render`, сначала пройти [4.0 Браузер: от URL до пикселя](./30-middle/4.0-браузер-от-url-до-пикселя-network-fetch-render.md).
- Держать в голове простое правило: `DOM` не хранит бизнес-логику и не должен быть твоей базой данных состояния.
- Если тема уходит в безопасность, производительность или архитектуру приложения, использовать этот раздел как базу и дальше идти в соседний домен.
- Самая полезная оптика для этого раздела: причинно-следственная цепочка. Что именно вызвало `layout`, `paint` или `composite`, где событие стало скрытой шиной, где offline-идея превратилась в устаревшие данные и почему интерфейс деградирует именно так.

## ✅ Ожидания по уровням

| Уровень | Что должен уметь |
|---|---|
| Junior | Понимать DOM как рабочий слой поверх HTML, безопасно работать с узлами, событиями, URL, storage и жизненным циклом страницы без хрупких селекторов и случайных побочных эффектов. |
| Middle | Осознанно выбирать transport и browser API под задачу, использовать observers и workers, читать DevTools-таймлайны и объяснять проблемы языком пайплайна браузера, а не «что-то тормозит». |
| Senior / Senior+ | Проектировать взаимодействие частей браузерного приложения, учитывать ограничения вкладок и потоков, выбирать подход для offline-режима и системно разбирать деградацию интерфейса. |
