# 4.4 Сеть и обмен данными: Fetch API, streams, SSE, WebSocket

## Уровень

Middle

## Кратко о теме

На этом уровне уже недостаточно уметь делать `fetch("/api")`. Нужно понимать саму модель обмена данными: запрос-ответ, потоковая доставка, серверные события, двусторонний канал, отмена запросов, backpressure и связь transport-слоя с UX.

Именно здесь часто рождаются хрупкие решения: запросы без отмены, вечные WebSocket там, где достаточно SSE, огромные JSON-ответы без стриминга и сетевые "оптимизации", которые на самом деле относятся к HTML-авторингу, а не к приложенческой сети.

## Зачем это нужно на практике

- Делать предсказуемый data-fetching в реальном UI, где есть гонки запросов, отмены и повторные действия пользователя.
- Не смешивать сетевой transport, кэширование, синхронизацию состояния и рендер в одну кучу.
- Выбирать правильную модель для live-обновлений, чатов, логов, экспорта и долгих серверных операций.
- Объяснять сетевые проблемы через модель обмена, а не только словами «API тормозит».

## Термины урока

- `Fetch API` — это браузерный интерфейс для HTTP-запросов и работы с `Request`/`Response`.
- `Streaming` — это обработка данных по частям, без ожидания полного ответа целиком.
- `SSE` — это односторонний поток событий от сервера к клиенту поверх HTTP.
- `WebSocket` — это постоянное двустороннее соединение между клиентом и сервером.

## Нормальный baseline для fetch

Хороший middle-код вокруг сети должен отвечать минимум на четыре вопроса:

1. Что считается успешным ответом, кроме "запрос не упал"?
2. Как отменяется устаревший запрос?
3. Что произойдёт при гонке нескольких запросов?
4. Какая часть ошибок реально показывается пользователю, а какая уходит в retry/logging?

Простой, но здоровый baseline:

```js
let activeController;

async function searchUsers(query) {
  activeController?.abort();
  activeController = new AbortController();

  const response = await fetch(`/api/users?q=${encodeURIComponent(query)}`, {
    signal: activeController.signal,
    headers: {
      Accept: "application/json",
    },
  });

  if (!response.ok) {
    throw new Error(`Search failed with status ${response.status}`);
  }

  return response.json();
}
```

Здесь важно не только то, что есть `fetch`, а то, что учтён реальный UX: пользователь мог ввести новый запрос раньше, чем закончился старый.

## Когда тебе реально нужен streaming

Streams нужны не для красоты, а когда ответ приходит долго, большой по объёму или по природе своей кусочный:

- сервер отдаёт большой CSV/лог;
- UI получает токены или частичные сообщения постепенно;
- нужно обработать ответ по мере прихода, а не ждать весь payload;
- требуется трансформация потока без полной материализации в памяти.

Пример чтения текстового потока:

```js
const response = await fetch("/api/export");
const reader = response.body
  .pipeThrough(new TextDecoderStream())
  .getReader();

let chunk = "";

while (true) {
  const { value, done } = await reader.read();

  if (done) break;

  chunk += value;
  renderPartial(chunk);
}
```

Если задача обычная и ответ маленький, `response.json()` всё ещё лучше. Стриминг без реальной причины только усложняет отладку.

## SSE, WebSocket, WebRTC: не всё что realtime, то WebSocket

- `SSE` хорошо подходит, когда сервер только толкает события клиенту: лента статусов, live-лог, оповещения, progress долгой операции.
- `WebSocket` нужен, когда связь действительно двусторонняя и низколатентная: чат, кооперативное редактирование, игровые сценарии, интерактивные dashboards.
- `WebRTC` нужен не для "ещё одного realtime API", а для peer-to-peer, медиа и близких сценариев.

Хороший инженерный вопрос здесь не "что современнее", а "какая модель обмена минимально достаточна для нашей задачи".

## Что не надо путать с transport-уровнем

Здесь легко смешать `Fetch API` и `preload/prefetch/preconnect/dns-prefetch`, но это разные уровни:

- `fetch`, streams, SSE, WebSocket — это приложенческий обмен данными;
- `resource hints` — это HTML-сигналы браузеру про загрузку ресурсов.

Если ты лечишь гонки API через `preconnect`, значит, ты решаешь не ту проблему.

## Типичные ошибки

- Пишут `await fetch(...)` и считают, что всё ок, не проверяя `response.ok`.
- Не отменяют устаревшие запросы и потом ловят гонки в search/autocomplete.
- Тянут `WebSocket` "на будущее", хотя у задачи сервер только шлёт редкие обновления и подошёл бы `SSE`.
- Запихивают в realtime-канал слишком много ответственности: бизнес-логика, state sync, retry, reconnect, analytics и локальный cache в одном месте.
- Используют streams там, где payload маленький и обычный JSON проще, понятнее и дешевле в поддержке.

## Как проверить понимание

1. Возьми UI с поиском или фильтрацией и добавь отмену устаревших запросов.
2. Для одного live-сценария выбери между polling, `SSE` и `WebSocket` и аргументируй выбор.
3. Найди в проекте один сетевой вызов без проверки `response.ok` и перепиши его с явной моделью ошибок.
4. Объясни, почему `preload` не решает проблему устаревшего API-ответа.

Признак освоения: ты можешь защитить выбор transport-модели под реальный продуктовый сценарий, а не просто назвать знакомые API.
