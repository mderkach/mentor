# 4.12 Системный дебаг деградации UI

## Уровень

Senior

## Кратко о теме

Middle-уровень уже умеет открыть DevTools и найти long task, layout shift или memory retention. Senior-уровень должен уметь больше: не просто увидеть симптом, а быстро собрать расследование, определить owner-слой и привести команду к правке, а не к бесконечной дискуссии.

UI-деградация редко приходит в чистом виде. Чаще это смесь:

- тяжёлого interaction path;
- сетевой нестабильности;
- лишнего re-render или layout thrash;
- stale cache;
- фоновых скриптов;
- low-end device pressure;
- неподходящей продуктовой логики.

## Зачем это нужно на практике

- Чтобы баг "иногда всё фризит" превращался в инженерную задачу, а не в фольклор.
- Чтобы команда знала, что мерить и кто владелец проблемы.
- Чтобы не лечить symptoms патчами вроде "поставим debounce ещё на один handler".
- Чтобы regression analysis был воспроизводимым и полезным для следующих релизов.

## Что ты освоишь

- Как строить расследование деградации от symptoms к root cause.
- Как отличать локальный UI-баг от системной platform/runtime-проблемы.
- Как работать с воспроизводимостью, low-end сценариями и owner-слоями.
- Как превращать findings в план исправления, а не только в красивый скриншот trace.

## Термины урока

- `Symptom` — наблюдаемая проблема: лаг, дёргание, рост памяти, stale UI, баг после restore.
- `Repro path` — минимальный воспроизводимый сценарий с фиксированными шагами.
- `Owner-layer` — слой, который реально должен чинить проблему: UI feature, shared infra, bundling, API, third-party, performance platform.
- `Control point` — место, где ты можешь измерить до/после и доказать эффект правки.

## Не начинай с инструмента, начинай с симптома

Плохой процесс:

- "открыл Performance, вроде есть длинная полоса";
- "кажется проблема в React";
- "давайте оптимизируем render".

Сильный процесс:

1. Зафиксируй symptom человеческим языком.
2. Опиши repro path.
3. Определи среду: браузер, устройство, network, tab-state, dataset size.
4. Выбери первый инструмент под гипотезу.
5. Получи артефакт: trace, heap snapshot, waterfall, запись RUM, лог restore event.
6. Найди owner-layer.
7. Сними до/после.

## Типы деградации и что они обычно означают

### Interaction lag

Часто указывает на:

- long tasks;
- expensive event handlers;
- synchronous compute на main thread;
- тяжёлый reconciliation/render path;
- third-party скрипты на критичном пути.

### Дёргание интерфейса

Часто связано с:

- layout shift;
- асинхронными данными без зарезервированного места;
- измерениями DOM после записи;
- анимациями без ясного ownership.

### Деградация со временем

Подозревай:

- memory retention;
- недочищенные listeners/timers;
- бесконечный cache growth;
- накопление DOM-узлов и state.

### Только после background/restore

Проверяй:

- page lifecycle assumptions;
- stale network state;
- broken reconnection logic;
- cache/session revalidation.

## Быстрый triage: что смотреть в первую очередь

Когда приходит жалоба "UI лагает", полезно очень быстро разделить класс проблемы:

### CPU-bound

Признаки:

- длинные JS task;
- expensive compute;
- main thread забит обработчиками и преобразованием данных.

### Layout-bound / render-bound

Признаки:

- forced reflow;
- частые layout/paint блоки;
- дёргание на scroll/resize/input;
- проблема особенно заметна при DOM-measure/write паттернах.

### Logic-bound

Признаки:

- неверный порядок async-операций;
- stale updates;
- дублирующиеся side effects;
- UI "ломается логически", даже если trace не выглядит чудовищно.

### Network-bound

Признаки:

- UI долго ждёт данные;
- waterfall показывает задержку до рендера полезной части;
- main thread сам по себе не перегружен, но продуктовый сценарий всё равно воспринимается медленным.

Эта быстрая классификация нужна не ради красоты, а чтобы не лечить CPU-проблему CSS-фиксом и не дебажить layout там, где на самом деле сломан async contract.

## Сначала докажи воспроизводимость

Senior-уровень особенно важен здесь: команда тратит огромное количество времени на невоспроизводимые баги.

Хороший repro path должен фиксировать:

- последовательность действий;
- размер данных;
- device/network profile;
- есть ли background/restore;
- свежая вкладка или long-lived session;
- влияние extensions и third-party.

Если этого нет, отладка быстро скатывается в "у меня не воспроизводится".

## Owner-layer важнее, чем название инструмента

Одна и та же деградация может принадлежать разным владельцам:

- UI feature team;
- shared component layer;
- routing/state orchestration;
- performance/platform;
- API/backend;
- сторонний vendor.

Пример:

- фильтр лагает;
- trace показывает длинную JS task;
- но root cause не в фильтре как UI, а в аналитическом SDK, который синхронно шлёт heavy payload в том же click path.

Если это не выяснить, feature-команда будет "оптимизировать render" вместо удаления настоящего bottleneck.

## Кейс: каталог начинает лагать только после 20 минут работы

Разбор без магии:

1. Симптом: typing lag и рост памяти после долгой сессии.
2. Repro: открыть каталог, 20 минут фильтровать, открывать карточки, сворачивать/разворачивать панели.
3. Инструмент: heap snapshot + performance trace на позднем этапе.
4. Наблюдение: куча удерживает старые search subscriptions и detached DOM nodes.
5. Owner-layer: shared filter panel, где cleanup listeners не происходит после remount.
6. Исправление: нормализовать lifecycle и повторно снять snapshot/trace.

Здесь важно не то, что ты умеешь сделать snapshot. Важно, что ты вывел проблему на owner и доказал эффект правки.

## Как оформлять findings

Хороший результат расследования содержит:

- symptom;
- repro path;
- артефакт измерения;
- root cause;
- owner;
- proposed fix;
- expected metric or behavioral change;
- residual risk.

Если findings выглядят как "кажется, render тяжёлый", это ещё не инженерный результат.

## Не чини следствие вместо причины

Очень частый сценарий:

- симптом виден в рендере;
- команда ставит `debounce`, memoization или локальный костыль;
- метрика чуть улучшается;
- root cause остаётся и вернётся в следующем сценарии.

Примеры:

- lag на вводе лечат `debounce`, хотя проблема в тяжёлом синхронном compute;
- jank на scroll лечат "оптимизацией компонента", хотя код thrash-ит layout;
- stale UI лечат повторным refetch, хотя сломан ownership async-result.

Senior-подход здесь очень прямой:

- сначала классифицируй деградацию;
- потом найди owner-layer;
- только потом выбирай тип фикса.

## Что нужно знать

- типовые классы UI-деградации и их признаки;
- разницу между symptom, signal, root cause и owner-layer;
- importance воспроизводимости и control points;
- как связывать DevTools, RUM и product scenario.

## Что нужно уметь

- собирать reproducible investigation;
- выбирать правильный артефакт под гипотезу;
- отделять локальную причину от системной;
- превращать дебаг в конкретный fix plan.

## Типичные ошибки

1. Сразу лезут в оптимизации без воспроизводимого сценария.
2. Лечат symptom не у того owner-layer.
3. Доверяют одному локальному trace без подтверждения в production-паттерне.
4. Не фиксируют device/network/session conditions.
5. Закрывают баг словами "не смогли воспроизвести" без карты проверок.

## Anti-pattern

"Раз DevTools показал длинную задачу, причина уже найдена".

Нет. Trace показывает проявление. Root cause может лежать в другом слое: данных, третьей стороне, cache policy, lifecycle или даже продуктовой orchestration.

## Как проверить понимание

1. Возьми один плавающий UI-баг и опиши для него repro matrix: device, browser, data size, tab state.
2. Назови первый и второй инструмент, которые ты откроешь, и почему именно в таком порядке.
3. Для одной деградации укажи возможные owner-слои и критерий, как сузить поиск.
4. Сформулируй findings в формате symptom -> evidence -> root cause -> owner -> fix.

Признак освоения: ты умеешь делать из UI-деградации короткое инженерное расследование, а не бесконечный поток догадок.
