# 4.8 Event architecture в браузерных приложениях: ownership, contracts, cross-context

## Уровень

Senior

## Кратко о теме

На junior- и middle-слое события обычно воспринимаются как `click`, `submit`, `input` и делегирование в DOM. На senior-слое вопрос уже другой: как устроить событийную архитектуру так, чтобы приложение не превратилось в сеть невидимых подписок, глобальных bus-ов и "магии", которую никто не может отладить.

В браузерном приложении есть минимум три разных класса событий:

- DOM-события интерфейса;
- application events между модулями;
- cross-context сигналы между вкладками, iframes, workers и service worker.

Если эти слои не разводить, код быстро становится хрупким.

## Зачем это нужно на практике

- Чтобы не тащить бизнес-координацию в случайные DOM-обработчики.
- Чтобы модуль мог публиковать понятный контракт, а не "случайно что-то дергать по дереву".
- Чтобы несколько вкладок, embedded-режимы и background-процессы не расходились по состоянию.
- Чтобы cleanup подписок и ownership были очевидны на ревью и в отладке.

## Что ты освоишь

- Как разложить события по слоям: UI, domain, platform.
- Когда `CustomEvent` и `EventTarget` достаточно, а когда уже нужен store, router или explicit service layer.
- Как использовать `BroadcastChannel` и `postMessage` без фантомной синхронизации.
- Почему cleanup подписок, имена событий и форма payload-а важнее, чем "какой bus взять".

## Термины урока

- `Event contract` — это явное соглашение о том, кто публикует событие, в какой форме, с каким жизненным циклом и кто имеет право на него реагировать.
- `Application event` — это событие уровня бизнес-потока, а не пользовательского DOM-взаимодействия.
- `Cross-context messaging` — это обмен сообщениями между разными execution contexts: вкладками, окнами, iframes, workers.
- `Ownership` — это ответ на вопрос, какой модуль создаёт событие, кто его обслуживает и кто отвечает за cleanup.

## Сначала раздели слои событий

Плохой признак зрелого фронтенда: одно слово "событие" означает сразу всё.

Практичный расклад такой:

- DOM-события отвечают за пользовательское взаимодействие и не должны быть transport-слоем доменной логики.
- Application events описывают то, что произошло в системе: `cart:item-added`, `session:expired`, `editor:autosave-failed`.
- Cross-context сигналы нужны там, где несколько вкладок или процессов должны договориться о состоянии.

Когда эти слои не разведены, начинается следующее:

- `click` на кнопке приводит к прямому вызову analytics, mutation, toast, router transition и обновлению нескольких unrelated-секций;
- сервисы начинают слушать DOM;
- вкладки синхронизируются через `localStorage`-хаки без явного контракта.

## DOM delegation — baseline, а не архитектура целого приложения

Делегирование событий остаётся отличным приёмом для UI-слоя:

- список с большим числом элементов;
- интерактивная таблица;
- лениво дорисовываемые карточки;
- embedded DOM, где прямые ссылки на каждый узел не нужны.

Но delegation плохо подходит как замена application event layer. Если у тебя один обработчик на `document`, который по `data-*` и `closest()` управляет половиной бизнес-логики приложения, это уже не "элегантная оптимизация", а неявный runtime-router.

Senior-правило простое:

- делегирование — для интерфейсного уровня;
- доменные переходы — через явные команды, сервисы, store, router или локальные event contracts.

## `CustomEvent` и `EventTarget`: хороший локальный контракт, если границы узкие

`CustomEvent` полезен, когда нужно явно связать соседние части системы без глобального bus-а:

```js
const profileEvents = new EventTarget();

profileEvents.dispatchEvent(
  new CustomEvent("avatar:upload-failed", {
    detail: { reason: "file-too-large" },
  }),
);
```

Это нормальный выбор, если:

- область действия локальна;
- событие относится к конкретной feature-зоне;
- payload небольшой и предсказуемый;
- есть явный владелец `EventTarget`.

Важно не превращать `new EventTarget()` в "микро-Redux без правил". Если event target лежит в `shared/utils/eventBus.ts` и им пользуется весь проект, ты уже строишь глобальную шину, только без формальных гарантий.

## Cleanup и lifecycle подписок важнее, чем выбор API

Утечки и фантомные реакции чаще всего появляются не из-за плохого API, а из-за плохого lifecycle management.

Полезный baseline:

- каждая подписка живёт рядом с owner-модулем;
- точка отписки понятна;
- одноразовые слушатели маркируются явно;
- для browser listeners используется `AbortController`, если это упрощает cleanup.

```js
const controller = new AbortController();

window.addEventListener("visibilitychange", handleVisibility, {
  signal: controller.signal,
});

// later
controller.abort();
```

Если cleanup "как-нибудь переживёт", он обычно переживает слишком долго и вылезает как memory retention, duplicate handler firing или баг "после третьего открытия модалки всё вызывается по два раза".

## Cross-context messaging: сначала сформулируй, что именно надо синхронизировать

Для нескольких вкладок и embedded-контекстов нужны уже другие инструменты:

- `BroadcastChannel` — если контексты одного origin должны обмениваться сигналами;
- `window.postMessage()` — если говорим между окнами/iframes и нужно контролировать origin;
- `ServiceWorker` или `Shared Worker` — если у тебя есть общий background-owner для сети, кеша или shared runtime;
- `navigator.locks` — если нужна координация доступа к общему ресурсу между вкладками.

Пример здравой синхронизации между вкладками:

- вкладка A завершила logout;
- отправила `session:revoked` через `BroadcastChannel`;
- остальные вкладки сбросили чувствительное состояние и ушли на экран логина.

Пример плохой синхронизации:

- вкладки пишут полу-структурированные флаги в `localStorage`;
- каждая трактует их по-своему;
- часть cleanup идёт по `storage`, часть по polling;
- через месяц никто не помнит, кто должен удалять ключ.

## Кейс: корзина, несколько вкладок и embedded checkout

Допустим, у продукта есть:

- каталог в одной вкладке;
- embedded checkout в iframe;
- возможность открыть корзину в нескольких вкладках.

Разумная архитектура будет такой:

- DOM-события остаются локальными внутри каждой feature;
- application event `cart:item-added` живёт внутри cart-domain;
- iframe общается с parent через `postMessage` и strict origin-check;
- межвкладочная синхронизация остатков и logout идёт через `BroadcastChannel`;
- конфликт записи в локальный draft защищается `navigator.locks` или явной leader-election стратегией.

Это сложнее, чем "давайте бросим всё в один bus". Но это и есть зрелая браузерная архитектура: разные проблемы получают разные каналы коммуникации.

## Что нужно знать

- разницу между DOM-, application- и cross-context событиями;
- границы `CustomEvent`, `EventTarget`, `BroadcastChannel`, `postMessage`;
- роль cleanup и subscription ownership;
- почему глобальный bus без правил быстро создаёт невидимую связность.

## Что нужно уметь

- выбрать минимальный канал коммуникации под конкретный сценарий;
- формулировать event contract: имя, payload, владелец, listeners, cleanup;
- отличать локальную событийную координацию от необходимости в store/router/service layer;
- проектировать межвкладочную синхронизацию без `localStorage`-магии.

## Типичные ошибки

1. Используют один глобальный event bus как замену архитектуре.
2. Тянут доменную координацию в DOM delegation на `document`.
3. Не фиксируют owner-модуль события и получают фантомных подписчиков.
4. Передают в payload слишком много данных и начинают зависеть от внутренней формы чужого состояния.
5. Синхронизируют вкладки через `localStorage` без протокола и cleanup.

## Anti-pattern

"Развязка через события всегда лучше прямой зависимости".

Нет. События полезны, когда уменьшают связанность без потери ясности. Если событие только скрывает ownership и делает поток данных невидимым, это деградация архитектуры, а не зрелость.

## Как проверить понимание

1. Возьми одну feature и разложи её события на DOM, application и cross-context.
2. Для сценария logout в нескольких вкладках выбери `BroadcastChannel`, `postMessage`, polling или ничего из этого и объясни почему.
3. Назови точку cleanup для каждого long-lived listener в одном реальном модуле.
4. Объясни, почему один глобальный bus здесь хуже, чем локальный owner плюс явный contract.

Признак освоения: ты можешь показать карту событий системы и объяснить, где событие реально снижает связанность, а где его нужно заменить более явным механизмом.
