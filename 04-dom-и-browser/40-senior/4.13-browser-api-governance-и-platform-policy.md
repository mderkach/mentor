# 4.13 Browser API governance и platform policy

## Уровень

Senior

## Кратко о теме

Почти каждый сильный frontend-проект рано или поздно упирается в вопрос:

"Можно ли уже брать этот browser API в прод?"

Плохой ответ:

"На моей машине работает".

Senior-ответ включает другое:

- какой пользовательский сценарий это закрывает;
- какая поддержка в целевых браузерах;
- есть ли fallback;
- нужен ли secure context или permission;
- кто владелец API-слоя;
- как мы его отключим или заменим при проблеме.

Это и есть governance. Не бюрократия, а защита продукта от случайной platform-зависимости.

## Зачем это нужно на практике

- Чтобы новые API не попадали в продукт без стратегии graceful degradation.
- Чтобы команда не строила критичный сценарий на limited-availability возможностях без оговорок.
- Чтобы permission-sensitive фичи не ломали UX и доверие пользователя.
- Чтобы поддержка браузеров, polyfills и deprecation decisions были управляемыми.

## Что ты освоишь

- Как принимать решение о внедрении browser API не по хайпу, а по policy.
- Почему feature detection лучше UA-sniffing и где всё равно нужен support matrix.
- Как работать с permission-aware UX и secure-context требованиями.
- Как фиксировать owner, fallback и exit strategy для platform-sensitive возможностей.

## Термины урока

- `Governance` — это набор правил, по которым команда вводит, ограничивает и сопровождает browser API.
- `Feature detection` — это проверка наличия возможности в текущей среде вместо угадывания браузера по UA.
- `Support matrix` — это явный список целевых сред и того, что в них считается обязательным, опциональным или недоступным.
- `Exit strategy` — это способ выключить или заменить API, если он даёт regressions или не оправдывает цену поддержки.

## Любой browser API должен пройти через 5 вопросов

### 1. Какой пользовательский сценарий он улучшает?

Если ответа нет, внедрение скорее всего делается "потому что можно".

### 2. Что будет в среде без поддержки?

Если fallback нет, feature либо не готова, либо должна считаться optional enhancement.

### 3. Есть ли secure-context или permission gate?

Если да, это меняет rollout и UX.

### 4. Кто владелец?

Если API добавили "внутрь одной фичи", а страдать от него будут все, ownership должен быть поднят выше.

### 5. Как выключить или заменить?

Фича без kill-switch или деградационного режима плохо переживает реальный прод.

## Feature detection лучше, чем browser sniffing

Причина простая: UA ненадёжен, меняется, редуцируется и плохо описывает реальную поддержку возможностей.

Практичный baseline:

```js
if ("clipboard" in navigator && window.isSecureContext) {
  await navigator.clipboard.writeText(value);
} else {
  fallbackCopy(value);
}
```

Полезный вывод:

- сначала проверяй capability;
- затем уже решай, какой degraded path показать;
- UA sniffing оставляй только для действительно вынужденных compatibility workarounds, и то как локальный exception с owner-ом и сроком жизни.

## Permission-aware UX — часть дизайна, а не техническая деталь

Если API живёт за permission gate, важны вопросы:

- когда спрашивать доступ;
- почему именно сейчас;
- что увидит пользователь при `prompt`, `granted`, `denied`;
- можно ли продолжить ценностный сценарий без разрешения.

Пример плохого rollout:

- приложение сразу при открытии просит Notifications, Geolocation и Clipboard;
- пользователь жмёт `deny`;
- дальше команда считает, что "люди не хотят feature".

Пример зрелого rollout:

- запрос permissions происходит в ясной точке сценария;
- value proposition объяснима;
- degraded mode тоже рабочий;
- метрики permission conversion и usage читаются отдельно.

## Support matrix должна быть явной

Не всё обязано работать одинаково везде. Но это нужно зафиксировать.

Удобная модель:

- `required baseline` — без этого продукт не может выполнить критичный сценарий;
- `progressive enhancement` — улучшает опыт, но не ломает core;
- `experimental / limited availability` — включается осознанно и ограниченно.

Иначе внутри команды быстро начинается путаница:

- один разработчик считает API baseline;
- второй оборачивает всё в fallback;
- третий ship-ит без него критичный flow.

## Governance-кейс: Clipboard, Web Locks и scheduler API

Три похожих по названию API, но разная policy:

- `Clipboard API` часто идёт как enhancement с secure-context и permission-aware UX;
- `Web Locks API` может быть полезен для multi-tab coordination, но нужен не всем и не должен silently становиться hard dependency;
- `scheduler.*` может улучшить responsiveness, но из-за ограниченной доступности требует feature detection и fallback.

У этих решений разные:

- бизнес-обоснование;
- fallback story;
- owner-слой;
- risk profile.

Одна governance policy должна помогать команде видеть эту разницу.

## Что нужно знать

- принципы feature detection и ограничения UA-sniffing;
- роль secure contexts и Permissions API;
- смысл support matrix, ownership и exit strategy;
- как различать baseline, enhancement и experimental feature.

## Что нужно уметь

- оформлять решение по новому browser API как engineering decision;
- проектировать fallback и degraded mode;
- формулировать policy на permission-sensitive и limited-availability возможности;
- не допускать скрытого platform lock-in через "маленькую удобную утилиту".

## Типичные ошибки

1. Ship-ят новый API без fallback, потому что "поддержка уже вроде нормальная".
2. Считают feature detection избыточным и проверяют `navigator.userAgent`.
3. Просят permissions слишком рано и без контекста.
4. Не фиксируют owner и получают orphaned platform-debt.
5. Оставляют compatibility workaround навсегда без даты пересмотра.

## Anti-pattern

"Если API улучшает DX разработчика, значит его можно quietly встроить в core flow".

DX команды не важнее runtime-реальности пользователя. Каждая platform-sensitive возможность должна пройти через продуктовую и эксплуатационную проверку.

## Как проверить понимание

1. Возьми один browser API и опиши его как `baseline`, `enhancement` или `experimental`.
2. Для permission-sensitive фичи распиши `granted/prompt/denied` UX.
3. Покажи пример feature detection и fallback без UA-sniffing.
4. Сформулируй exit strategy: как команда выключит API при проблемах в проде.

Признак освоения: ты умеешь вводить browser APIs так, чтобы продукт не становился заложником одной платформенной ставки.
