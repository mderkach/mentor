# 4.3 Browser API в практике: lifecycle страницы, URL, storage, clipboard и history

## Уровень

Junior

## Кратко о теме

После DOM и событий следующий слой браузера — это page lifecycle, адресная строка, история навигации, хранение данных, clipboard и соседние API, которые делают интерфейс частью реального браузерного окружения, а не просто набором обработчиков на странице.

На этом уровне важно понять две вещи:

- у страницы есть жизненный цикл, и инициализация не сводится к одному `window.onload`;
- у каждого browser API есть цена: по поддержке, по UX, по безопасности и по сроку жизни данных.

## Зачем это нужно на практике

- Синхронизировать фильтры, табы и состояние поиска с URL, чтобы ссылкой можно было делиться.
- Хранить черновики, пользовательские настройки и кэш там, где это действительно оправдано.
- Реагировать на уход страницы в background, возврат пользователя, закрытие вкладки, смену ориентации и разрешений.
- Пользоваться `clipboard`, `postMessage`, `BroadcastChannel` и `History API` без хрупких костылей.

## Термины урока

- `Page lifecycle` — это этапы жизни документа: загрузка, активная работа, скрытие вкладки, возможное восстановление из cache, выгрузка.
- `Persisted state` — это данные, которые переживают перерисовку, перезагрузку страницы или даже закрытие вкладки.
- `Navigation state` — это часть состояния интерфейса, которая должна жить в URL и истории браузера.
- `Storage scope` — это граница жизни и доступности данных: только вкладка, только origin, долгое хранение, кэш ресурсов и так далее.

## Жизненный цикл страницы: не сводится к `load`

- `DOMContentLoaded` нужен, когда DOM уже построен и тебе не нужно ждать загрузку картинок, шрифтов и остальных ресурсов.
- `load` нужен гораздо реже. Обычно он нужен, если ты действительно зависишь от полной загрузки страницы и всех вложенных ресурсов.
- `visibilitychange` полезен для мягкой синхронизации: остановить polling, сохранить черновик, пометить вкладку как неактивную.
- `beforeunload` не должен быть основой бизнес-логики. Это аварийный стоп-кран для защиты от потери данных, и браузеры сильно ограничивают его поведение.

## URL и History как часть UX-контракта

Если пользователь может поделиться состоянием страницы ссылкой, это почти всегда лучше, чем держать всё только в памяти.

```js
const params = new URLSearchParams(location.search);
const initialView = params.get("view") ?? localStorage.getItem("issues-view") ?? "board";

function updateView(nextView) {
  const nextParams = new URLSearchParams(location.search);
  nextParams.set("view", nextView);

  history.replaceState(null, "", `${location.pathname}?${nextParams.toString()}`);
  localStorage.setItem("issues-view", nextView);
  renderView(nextView);
}
```

Здесь URL решает задачу shareable-state, а `localStorage` — задачу удобного дефолта. Это хороший пример разделения ответственности.

## Как выбирать хранилище без суеверий

- `localStorage` — для небольших простых данных, когда важна простота, а не объём и не транзакции.
- `sessionStorage` — когда данные должны жить только в рамках вкладки.
- `IndexedDB` — когда данные уже похожи на реальное локальное хранилище: объём, офлайн, индексы, сложные структуры.
- `Cache API` — не про «любые данные вообще», а про кэширование ресурсов и ответов.
- `cookies` — не универсальное хранилище для фронтенд-состояния. Они сильно завязаны на сервер, сеть и безопасность.

Если разговор заходит про токены, auth и чувствительные данные, это уже не просто browser API, а security-решение: [11. Безопасность](../../11-безопасность/README.md).

## Clipboard, permissions и межвкладочное общение

- `Clipboard API` полезен для явных пользовательских действий: «Скопировать ссылку», «Скопировать код». Он не должен работать скрытно и неожиданно.
- `Permissions API` помогает понять, какой статус у чувствительных возможностей вроде геолокации, но не отменяет нормальный UX и graceful fallback.
- `BroadcastChannel` и `postMessage` хороши, когда тебе действительно нужно общение между вкладками, окнами или iframe. Если задача решается через один источник данных и обычный рендер, не усложняй.

## Типичные ошибки

- Всю инициализацию вешают на `window.onload`, хотя нужен был только готовый DOM.
- Держат фильтры и активные табы только в памяти, поэтому ссылкой нельзя поделиться, а back/forward ведут себя странно.
- Используют `localStorage` как мини-базу данных без стратегии миграций, инвалидации и обработки ошибок.
- Завязывают важную логику на `beforeunload`, хотя браузер вообще не обязан вести себя одинаково во всех сценариях.
- Складывают в storage то, что уже должно жить на сервере, в URL или в доменной модели приложения.

## Как проверить понимание

1. Возьми страницу с фильтрами и перенеси активный фильтр в `URLSearchParams`.
2. Определи, какие данные у тебя должны жить в URL, какие в `localStorage`, а какие вообще не должны храниться в браузере.
3. Добавь реакцию на `visibilitychange`: например, сохраняй черновик формы при уходе вкладки в background.
4. Объясни, почему `beforeunload` не является надёжной стратегией автосохранения.

Признак освоения: ты можешь объяснить, почему конкретное состояние живёт именно в URL, `storage` или памяти, а не просто «так получилось».
