# 4.2 DOM-события: bubbling, delegation и управление поведением

## Уровень

Junior

## Кратко о теме

События — это способ, которым браузер сообщает приложению: пользователь кликнул, ввёл текст, отправил форму, зажал клавишу, тапнул по экрану, элемент получил фокус.

Большая часть проблем здесь не в том, что разработчик «не знает `addEventListener`». Проблема в другом: обработчики вешаются хаотично, `preventDefault` ставится по привычке, `stopPropagation` ломает соседние сценарии, а события перестают быть предсказуемым контрактом.

## Зачем это нужно на практике

- Строить UI без лишней связности между элементами.
- Управлять поведением списков, форм, dropdown, modal, autocomplete, drag-and-drop.
- Писать код, который переживает перерисовку и динамическое добавление DOM-узлов.
- Не убивать accessibility кастомным JS поверх нативного HTML-поведения.

## Что ты освоишь

- Что такое `target`, `currentTarget`, capture phase и bubble phase.
- Когда использовать event delegation и почему это обычно лучше сотни локальных listener-ов.
- Как работать с `click`, `input`, `change`, `submit`, `keydown`, pointer-событиями.
- Как проектировать `CustomEvent`, чтобы он был понятным контрактом, а не секретным каналом между модулями.

## Термины урока

- `Event target` — это узел, на котором событие фактически произошло.
- `CurrentTarget` — это узел, чей обработчик сейчас выполняется.
- `Bubbling` — это всплытие события от внутреннего элемента вверх по дереву.
- `Delegation` — это обработка событий на общем контейнере вместо индивидуальных listener-ов на каждом потомке.

## Как устроена нормальная модель событий

- Большинство UI-сценариев живёт на bubbling. Это значит, что тебе часто достаточно одного обработчика на контейнере.
- Capture phase нужна реже. Обычно она полезна, когда тебе нужно перехватить событие раньше других обработчиков или отследить поведение на верхнем уровне.
- `preventDefault()` отменяет встроенное действие браузера. Это нормально для кастомной обработки формы, но не должно становиться рефлексом.
- `stopPropagation()` и особенно `stopImmediatePropagation()` нужны редко. Если ты ставишь их регулярно, у тебя обычно проблема в архитектуре взаимодействий, а не в DOM.

## Практический паттерн: delegation

Когда список карточек или меню рендерится динамически, listener на каждом элементе быстро становится дорогим и хрупким. Чаще лучше так:

```js
const list = document.querySelector("[data-todo-list]");

list.addEventListener("click", (event) => {
  const actionButton = event.target.closest("[data-action]");

  if (!actionButton || !list.contains(actionButton)) {
    return;
  }

  const item = actionButton.closest("[data-todo-id]");
  const { action } = actionButton.dataset;
  const { todoId } = item.dataset;

  if (action === "remove") {
    removeTodo(todoId);
  }

  if (action === "toggle") {
    toggleTodo(todoId);
  }
});
```

Почему это хороший вариант:

- новый DOM продолжает работать без перевешивания обработчиков;
- логика завязана на контракт `data-action`, а не на визуальный класс;
- один контейнер отвечает за один тип взаимодействия.

## Формы и клавиатура: частые места потерь

- Для формы почти всегда нужен `submit`, а не только `click` по кнопке. Иначе ломаются `Enter`, screen reader и нативные сценарии отправки.
- Для полей ввода различай `input` и `change`. Первый нужен для мгновенной реакции, второй для фиксации завершённого изменения.
- Если строишь keyboard interaction, не своди всё к `keydown`. Сначала проверь, не закрывает ли сценарий нативный `button`, `input`, `details`, `dialog`.
- Pointer-события полезны, когда нужно единое поведение для мыши, тача и стилуса. Но если задача обычная, `click` и `submit` всё ещё нормальный baseline.

## CustomEvent без мистики

`CustomEvent` полезен, когда у тебя есть изолированный виджет и тебе нужно наружу отдать понятное доменное событие.

```js
modal.dispatchEvent(
  new CustomEvent("modal:confirmed", {
    bubbles: true,
    detail: { orderId },
  })
);
```

Нормальный контракт здесь такой:

- имя события объясняет смысл, а не реализацию;
- `detail` содержит нужные данные, но без мусора;
- слушатель не зависит от внутренней структуры виджета.

## Типичные ошибки

- Вешают listener на каждый элемент списка, хотя можно было обойтись delegation.
- Ставят `preventDefault()` на каждое событие «на всякий случай».
- Используют `stopPropagation()` так часто, что ломают аналитические слушатели, модалки, вложенные меню и accessibility-слой.
- Реагируют только на `click`, забывая про `submit`, клавиатуру и фокус.
- Генерируют `CustomEvent` с невнятным именем вроде `update`, после чего никто не понимает, что именно обновилось.

## Как проверить понимание

1. Возьми список с динамически добавляемыми элементами и перепиши обработку кликов на delegation.
2. Проверь форму: работает ли отправка по `Enter`, не ломается ли нативная валидация.
3. Найди место, где стоит лишний `stopPropagation()` или `preventDefault()`, и объясни, зачем он там вообще нужен.
4. Придумай один `CustomEvent` для реального виджета и опиши его контракт в одном предложении.

Признак освоения: ты можешь объяснить путь события по DOM и не строишь интерфейс из случайного набора listener-ов.
