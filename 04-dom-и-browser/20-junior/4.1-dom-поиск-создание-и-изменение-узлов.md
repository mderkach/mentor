# 4.1 DOM в практике: поиск, создание и изменение узлов

## Уровень

Junior

## Кратко о теме

`DOM` — это runtime-представление документа в браузере. Именно с ним ты работаешь, когда ищешь элементы, меняешь текст, вставляешь карточки, переводишь фокус и измеряешь геометрию интерфейса.

На junior-уровне здесь важно не количество API, а дисциплина: какие селекторы считаются устойчивыми, как не сломать доступность, когда достаточно `classList`, а когда уже нужен `template` или `DocumentFragment`.

## Зачем это нужно на практике

- Подключать поведение к уже существующей разметке, а не переписывать страницу с нуля.
- Делать виджеты, фильтры, dropdown, tabs, modals и простые SSR-enhancement сценарии.
- Уметь читать и чинить код вне фреймворка, в legacy, в CMS, в аналитических скриптах и интеграциях.
- Не устраивать `innerHTML`-хаос там, где достаточно пары безопасных операций с узлами.

## Что ты освоишь

- Когда брать `querySelector`, `querySelectorAll`, `getElementById`, `closest`, `matches`.
- Как создавать и вставлять узлы через `createElement`, `append`, `prepend`, `replaceWith`, `remove`.
- Как использовать `classList`, `dataset`, `textContent`, `style` и `template.content` без лишней магии.
- Когда измерять элемент через `getBoundingClientRect`, `getComputedStyle`, `scrollIntoView`, `focus`.

## Термины урока

- `DOM-узел` — это объект в памяти браузера, представляющий часть документа.
- `Селектор-якорь` — это стабильный способ найти элемент, который не ломается от косметических правок в разметке.
- `Mutation` — это изменение DOM-дерева: вставка, удаление, замена, изменение атрибута или текста.
- `Layout measurement` — это чтение геометрии элемента (`width`, `height`, координаты) после того, как браузер уже собрал layout.

## Как думать о DOM без хаоса

- `DOM` не является источником истины для бизнес-состояния. Он показывает текущее состояние интерфейса, но не должен хранить всю логику приложения.
- Хороший DOM-код начинается не с `querySelector`, а с вопроса: какой у элемента устойчивый контракт? `id`, `data-*`, роль, контейнер?
- Сначала старайся выразить смысл через HTML, а потом подключай JS. Если элемент уже должен быть кнопкой, делай `button`, а не `div` с хрупким поиском и ручным `keydown`.

## Практический шаблон выбора и обновления

- Для одного понятного элемента обычно достаточно `getElementById` или `querySelector`.
- Для поведения по месту полезнее `closest('[data-role=\"...\"]')`, чем длинный селектор через несколько `.parent .child .thing`.
- Для пакетной вставки коллекции лучше собрать `DocumentFragment`, а потом сделать одну вставку.
- Для текста почти всегда нужен `textContent`, а не `innerHTML`. Если вдруг тебе реально нужно вставлять пользовательский HTML, это уже security-тема: [1.4.2 Безопасный HTML-контракт](../../01-html/40-senior/1.4.2-XSS-в-HTML-шаблонах-и-пользовательском-контенте.md).

## Пример

```html
<ul data-todo-list></ul>

<template id="todo-item-template">
  <li class="todo-item">
    <button type="button" data-action="toggle"></button>
    <span data-role="title"></span>
  </li>
</template>
```

```js
const list = document.querySelector("[data-todo-list]");
const template = document.getElementById("todo-item-template");

function renderTodos(todos) {
  const fragment = document.createDocumentFragment();

  for (const todo of todos) {
    const item = template.content.firstElementChild.cloneNode(true);
    const title = item.querySelector("[data-role='title']");
    const toggleButton = item.querySelector("[data-action='toggle']");

    item.dataset.todoId = todo.id;
    item.classList.toggle("todo-item_done", todo.done);
    title.textContent = todo.title;
    toggleButton.textContent = todo.done ? "Вернуть" : "Готово";

    fragment.append(item);
  }

  list.replaceChildren(fragment);
}
```

Почему это хороший вариант:

- разметка остаётся читаемой;
- текст вставляется безопасно через `textContent`;
- элементы ищутся по `data-*`, а не по случайной визуальной структуре;
- вставка идёт одним пакетом, а не по одному `append` в живой DOM на каждый шаг.

## Где чаще всего ошибаются

- Цепляют логику к селекторам вроде `.sidebar > div:nth-child(2) .item`, а потом любой рефакторинг HTML всё ломает.
- Используют `innerHTML` для всего подряд, включая пользовательские данные.
- Читают `getBoundingClientRect()` внутри цикла после каждой записи в DOM и сами создают лишние пересчёты layout.
- Хранят «состояние» компонента только в классах и `dataset`, а потом не могут объяснить, что реально считается источником истины.
- Переводят фокус и скролл как побочный эффект без проверки, что элемент уже существует и действительно должен получить `focus`.

## Как проверить понимание

1. Возьми список карточек и добавь к нему рендер через `template` и `DocumentFragment`.
2. Убери все хрупкие селекторы и замени их на `id` или `data-*` якоря.
3. Проверь, где тебе действительно нужен `innerHTML`, а где хватит `textContent`.
4. Объясни, в каких местах код читает layout, а в каких только меняет DOM.

Признак освоения: ты не просто «умеешь найти элемент», а умеешь строить DOM-код, который не ломается от первой же правки разметки.
