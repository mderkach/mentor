# 5.6 Модули, декларации и расширение типов

Этот урок начинается там, где обычной типизации компонентов и функций уже мало. Проект подключает внешние библиотеки, глобальные переменные сборки, SVG-модули, `env`-файлы и плагинные расширения. Именно здесь TypeScript сталкивается с реальной файловой и модульной системой проекта.

## 🎯 Что ты освоишь

- как устроены `ambient declarations` и зачем нужны файлы `*.d.ts`;
- когда нужен `module augmentation`, а когда лучше отдельный adapter-слой;
- где уместны `enum`, `const enum`, `object as const`, `namespace` и почему ими легко злоупотребить;
- как не превращать глобальные объявления в источник скрытой связности.

---

## Термины урока

- `Declaration file` — это файл `*.d.ts`, который описывает типовой контракт модуля или глобального окружения без runtime-реализации.
- `Ambient declaration` — это объявление "эта сущность существует во внешнем мире, поверь системе типов".
- `Module augmentation` — это расширение типов уже существующего модуля.

---

## Зачем это нужно на практике

Рано или поздно фронтенд-проект сталкивается с такими задачами:

- библиотека без корректных типов;
- импорт `*.svg`, `*.css`, `*.md`;
- переменные `import.meta.env`;
- глобальные API в `window`;
- расширение типа роутера, query-библиотеки или UI-пакета.

Если не понимать декларации и модули, команда начинает:

- складывать костыли в случайные `global.d.ts`;
- расширять типы библиотек без правил;
- маскировать проблемы через `declare const something: any`.

---

## `d.ts` нужен для контракта, а не для прятания ошибки

Простой прикладной пример:

```ts
declare module '*.svg' {
  const src: string;
  export default src;
}
```

Это честный use case: runtime уже умеет импортировать SVG через сборщик, а TypeScript просто не знает тип такого импорта.

Нечестный use case:

```ts
declare module 'legacy-widget';
```

Такой код говорит "модуль существует, но мы ничего о нём не знаем". Иногда это временно допустимо, но как постоянное решение он выключает типовую пользу на всей границе интеграции.

---

## `module augmentation`: инструмент сильный, но связность растит быстро

```ts
declare module 'my-ui-kit' {
  interface ButtonProps {
    analyticsId?: string;
  }
}
```

Это может быть полезно, если библиотека действительно задумана как расширяемая.

Но есть риски:

- расширение часто плохо видно из точки использования;
- разные команды могут расширять один и тот же тип по-разному;
- поведение становится менее прозрачным, чем у явного wrapper-компонента или adapter-а.

Практическое правило:

- если расширение локальное и проектное, сначала подумай о wrapper-layer;
- augmentation используй там, где он действительно уменьшает шум, а не скрывает архитектурный долг.

---

## `enum`, `const enum`, `object as const` и string unions: что выбирать в современном коде

```ts
type ThemeMode = 'light' | 'dark';

const ThemeModeMap = {
  light: 'light',
  dark: 'dark',
} as const;

type ThemeModeFromMap = typeof ThemeModeMap[keyof typeof ThemeModeMap];

enum LegacyThemeMode {
  Light = 'light',
  Dark = 'dark',
}
```

Во многих frontend-сценариях уже хватает `string union`. Если нужен и runtime-объект, и тип из одного источника, чаще удобнее `object as const`.

Практичный default по современному фронтенду такой:

- `string union` бери, когда нужен только типовой контракт набора значений;
- `object as const` бери, когда те же значения нужны ещё и в runtime для мап, списков, селектов и сериализации;
- `enum` бери осознанно, когда нужен именно runtime-объект как часть контракта или есть совместимость с legacy/generated code;
- `const enum` оставляй для полностью контролируемых application-сборок, где ты уверен в пайплайне компиляции end-to-end.

### Когда хватает `string union`

```ts
type RequestStatus = 'idle' | 'loading' | 'success' | 'error';
```

Это лучший default, если:

- значения нужны только для типизации;
- runtime-список можно держать отдельно или он вообще не нужен;
- важна простота без дополнительного output-кода.

### Когда лучше `object as const`

```ts
export const RequestStatus = {
  idle: 'idle',
  loading: 'loading',
  success: 'success',
  error: 'error',
} as const;

export type RequestStatus =
  typeof RequestStatus[keyof typeof RequestStatus];
```

Это хороший вариант, когда:

- нужен и runtime-объект, и тип;
- значения нужно итерировать, показывать в UI или мапить на лейблы;
- хочется избежать особенностей `enum` в output и tooling.

Во фронтенд-коде это часто самый практичный компромисс между выразительностью и предсказуемостью.

### Когда `enum` всё ещё оправдан

`enum` создаёт реальный объект в runtime. Это полезно, когда:

- ты работаешь с legacy-кодом, где `enum` уже часть публичного API;
- есть generated-контракты или библиотечный интерфейс, который ожидает именно `enum`;
- нужен один экспортируемый runtime-артефакт без ручной сборки объекта и union-типа.

Но есть и цена:

- `enum` добавляет runtime-output;
- numeric `enum` даёт reverse mapping, который легко удивляет в дебаге и сериализации;
- как default для UI-состояний и API-статусов он чаще тяжелее, чем нужно.

### Почему с `const enum` нужен холодный расчёт

`const enum` инлайнит значения на этапе компиляции:

```ts
const enum Priority {
  Low = 1,
  High = 2,
}
```

Плюс здесь один: минимальный output.

Минусы важнее в большинстве командных проектов:

- поведение зависит от того, кто и чем именно транспилирует код;
- смешанные пайплайны с `tsc`, Babel, SWC, esbuild и `isolatedModules` делают такой контракт более хрупким;
- экспортировать `const enum` из библиотек и shared-пакетов особенно рискованно.

Практическое правило:

- внутри закрытого приложения с полностью контролируемым build-процессом `const enum` иногда допустим;
- в библиотеках, shared packages и mixed-tooling окружениях лучше почти всегда обойтись без него.

## `namespace` и triple-slash directives: в современном фронтенде редко нужны

Эти конструкции стоит знать, потому что они встречаются в legacy и в старых типовых пакетах. Но делать их default-выбором в современном ESM-проекте не стоит.

Если ты видишь `namespace` в новом приложении, это повод спросить:

- зачем тут не сработали обычные модули;
- не тянем ли мы старую архитектурную модель в современный стек;
- нельзя ли сделать контракт явно через export/import.

## Что нужно знать

- роль `*.d.ts` и ambient declarations;
- как объявлять типы для нестандартных импортов и глобального окружения;
- когда использовать `module augmentation`;
- границы `enum`, `const enum`, `object as const`, `namespace`, triple-slash directives в современном проекте.

---

## Что нужно уметь

- создать понятный `d.ts` для внешнего модуля или asset import;
- расширить внешний тип так, чтобы это было видно и поддерживаемо;
- избегать лишней глобальности и скрытых зависимостей;
- выбирать между `string union`, `object as const`, `enum` и wrapper-layer без магии.

---

## Типичные ошибки

1. Складывают все объявления в один огромный `global.d.ts`.
2. Расширяют тип библиотеки, хотя проще и честнее было сделать собственный adapter.
3. Используют `enum` там, где хватило бы `union` или `object as const`.
4. Экспортируют `const enum` из shared-пакета и удивляются, почему tooling начинает спорить.
5. Заводят ambient-объявление с `any`, а потом считают проблему решённой.
6. Не понимают, что runtime и типы должны сходиться: `d.ts` не создаёт поведение сам по себе.

---

## Anti-pattern

"Если библиотека плохо типизирована, объявим её через `any`, а потом разберёмся".

Это допустимо только как временная мера под горячую интеграцию. В нормальном состоянии проекта такие дыры надо либо локализовать adapter-слоем, либо закрыть корректным объявлением.

---

## Как проверить понимание

1. Добавь типы для импорта `*.svg` или `*.md`.
2. Опиши `window.__APP_CONFIG__` через декларацию так, чтобы было видно его форму.
3. Объясни, почему в одном кейсе нужен `module augmentation`, а в другом лучше wrapper-компонент или adapter.

Мини-задача:

- проект импортирует `virtual:icons/home`;
- runtime-сборщик это поддерживает;
- TypeScript ругается.

Что нужно сделать:

- добавить `declare module 'virtual:icons/*'`;
- описать контракт импорта;
- положить объявление туда, где его реально найдут и поддержат.
