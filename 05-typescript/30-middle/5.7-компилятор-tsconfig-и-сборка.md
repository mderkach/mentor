# 5.7 Компилятор, tsconfig и сборка

Если предыдущие уроки были про язык и модели данных, то этот про эксплуатацию TypeScript как инженерного инструмента. Здесь решается, помогает ли типовая система команде, или каждый разработчик живёт со своим набором локальных исключений и случайных `tsconfig`.

---

## Что уже должно быть знакомо

- базовая типизация и работа с контрактами: [5.1 Базовые типы и базовые конструкции](../10-intern/5.1-базовые-типы-и-базовые-конструкции.md);
- модули и декларации: [5.6 Модули, декларации и расширение типов](./5.6-модули-декларации-и-расширение-типов.md);
- соседний слой по сборке и delivery: [7.4 Vite, env и базовая сборка frontend-приложения](../../07-infrastructure/20-junior/7.4-Vite-env-и-базовая-сборка-frontend-приложения.md).

---

## 🎯 Что ты освоишь

- какие флаги `tsconfig` действительно влияют на надёжность и DX;
- как связаны `strict`, `strictNullChecks`, `noImplicitAny`, `isolatedModules`, `verbatimModuleSyntax`;
- зачем нужны `paths`, `baseUrl`, `moduleResolution`, `project references`, `declaration emit`;
- почему TypeScript-конфиг нельзя настраивать в отрыве от bundler-а и монорепы.

---

## Термины урока

- `tsconfig` — это контракт компилятора: как TypeScript читает проект, проверяет типы и эмитит артефакты.
- `Project references` — это способ связать несколько TS-проектов в предсказуемую графовую сборку.
- `Declaration emit` — это генерация `*.d.ts` для библиотек и shared packages.
- `Module resolution` — это правило, по которому TypeScript ищет модули по импортам.

---

## Зачем это нужно на практике

Плохой `tsconfig` создаёт очень узнаваемые симптомы:

- локально всё зелёное, а в CI другой результат;
- импорт через alias работает в IDE, но падает в runtime;
- библиотека публикуется без нормальных `d.ts`;
- часть кода живёт в `strict`, часть в бесконечных исключениях;
- монорепа тормозит, потому что проект не разделён на осмысленные TS-графы.

Хороший `tsconfig` делает другое:

- правила едины для команды;
- ошибки ловятся раньше;
- типы для внутренних пакетов предсказуемо публикуются и читаются;
- bundler и TypeScript смотрят на проект одинаково.

---

## `strict` — это не "режим для перфекционистов"

Практичный baseline для взрослого проекта:

- `strict: true`;
- `noImplicitAny: true`;
- `strictNullChecks: true`.

Почему это важно:

- `noImplicitAny` не даёт дыркам появляться молча;
- `strictNullChecks` заставляет честно работать с отсутствием данных;
- `strict` в целом поднимает нижнюю границу качества типовой системы.

Выключать эти правила имеет смысл только как временный этап миграции legacy, а не как архитектурную норму.

---

## `paths` и `baseUrl`: удобно, но только если bundler согласован

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@shared/*": ["src/shared/*"]
    }
  }
}
```

Это работает только тогда, когда:

- тот же alias понимает bundler;
- тестовый раннер тоже знает про него;
- команда не плодит локальные alias без системы.

Если alias есть только в `tsconfig`, IDE будет счастлива, а runtime может упасть. Это не мелочь, а типичный источник ложного чувства безопасности.

---

## `moduleResolution`, `isolatedModules`, `verbatimModuleSyntax`

Эти опции важны, когда TypeScript живёт рядом с современным bundler-ом и ESM.

Что важно понять:

- `moduleResolution` должен соответствовать реальной среде импорта;
- `isolatedModules` полезен, если код трансформируется пофайлово;
- `verbatimModuleSyntax` помогает не маскировать расхождение между тем, что импорт написан в коде, и тем, что реально должно быть сэмитировано.

Это уже не "синтаксис языка", а граница между TypeScript и build-pipeline.

---

## `project references`: когда один tsconfig на всё уже перестаёт работать

```json
{
  "compilerOptions": {
    "composite": true,
    "declaration": true,
    "incremental": true
  },
  "references": [{ "path": "../shared-ui" }]
}
```

Это полезно для:

- монорепы;
- shared packages;
- крупных приложений, где нужна предсказуемая графовая сборка;
- ускорения typecheck и повторных сборок.

Но `project references` не нужны "просто чтобы было современно". Если проект маленький и линейный, они могут дать больше сложности, чем пользы.

---

## `declaration emit`: критично для библиотек и shared packages

Если пакет экспортирует типы наружу, `d.ts` должен быть частью нормального контракта публикации.

Проверь:

- публикуются ли `declaration`-файлы;
- совпадает ли публичный API типов с runtime API;
- не утекли ли во внешний контракт внутренние типы, alias и пути.

На уровне команды это уже влияет на DX соседних приложений и библиотек.

---

## Что нужно знать

- роль `strict`, `noImplicitAny`, `strictNullChecks`;
- зачем нужны `baseUrl`, `paths`, `moduleResolution`;
- где применимы `composite`, `incremental`, `project references`;
- когда нужен `declaration emit`;
- почему `tsconfig` должен быть согласован с bundler, тестами и файловой структурой.

---

## Что нужно уметь

- собирать осмысленный `tsconfig` под тип проекта;
- видеть опасные расхождения между TypeScript и build-tooling;
- постепенно ужесточать legacy-конфиг без паралича команды;
- принимать решение, нужен ли проекту отдельный TS-граф или достаточно простого конфига.

---

## Пример

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "noImplicitAny": true,
    "strictNullChecks": true,
    "jsx": "react-jsx",
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    },
    "isolatedModules": true,
    "verbatimModuleSyntax": true
  },
  "include": ["src"]
}
```

Это не универсальный шаблон на все случаи, но в нём уже видно несколько здоровых решений:

- строгий режим включён;
- конфиг ориентирован на современный bundler;
- alias описан явно;
- нет выключенных правил "чтобы не мешали жить".

---

## Типичные ошибки

1. Копируют `tsconfig` из статьи, не понимая половину опций.
2. Включают alias только в TypeScript, забывая про bundler и тесты.
3. Выключают `strictNullChecks`, потому что "слишком много ошибок".
4. Тащат `project references` в маленький проект без реальной причины.
5. Публикуют пакет без `d.ts`, а потом чинят DX downstream-команд.

---

## Anti-pattern

"TypeScript-конфиг можно настроить один раз и больше не думать".

Нет. Конфиг — это часть инженерной архитектуры. Он должен меняться вместе с ростом проекта, bundler-стратегии и правилами команды.

---

## Как проверить понимание

1. Объясни, зачем проекту `strictNullChecks` и почему это не просто "больше шума от компилятора".
2. Проверь любой alias из `tsconfig` и скажи, знает ли о нём bundler и тестовый раннер.
3. Посмотри на shared package и ответь, генерирует ли он корректные `d.ts`.

Мини-задача:

- есть `src/shared/*`, `src/features/*`, alias `@/*`;
- IDE резолвит импорт;
- dev-сервер падает на runtime.

Найди причину:

- alias настроен только в `tsconfig`;
- bundler не знает про этот mapping;
- типовой контракт и runtime-контракт разошлись.
