# 5.3 Generics, conditional и mapped types

Этот урок про момент, когда TypeScript начинает окупаться в больших кодовых базах: ты перестаёшь копировать похожие типы руками и начинаешь собирать их из базовых правил.

Главная мысль: сила тут не в хитром синтаксисе, а в том, чтобы выразить инвариант один раз и не размазывать его по пяти файлам.

- моделирование структур данных через `type`, `interface`, `keyof`, `typeof`: [5.2 Моделирование структур данных](./5.2-моделирование-структур-данных.md).

## 🎯 Что ты освоишь

- зачем нужны `generic functions`, `generic interfaces` и `constraints`;
- когда использовать `conditional types`, а когда остановиться;
- как работают `mapped types`, `template literal types` и `infer` на прикладном уровне;
- почему "умный тип" полезен только тогда, когда им ещё может пользоваться команда.

---

## Термины урока

- `Generic` — это параметризованный тип или функция, которая работает с семейством значений, а не с одним конкретным типом.
- `Constraint` — это ограничение на generic-параметр, например `T extends { id: string }`.
- `Conditional type` — это типовая ветка "если условие верно, получи один тип, иначе другой".
- `Mapped type` — это тип, который проходит по ключам другого типа и строит новый контракт.

---

## Зачем это нужно на практике

В обычном приложении быстро появляются повторяющиеся конструкции:

- обёртки для ответов API;
- таблицы и списки, работающие с разными сущностями;
- формы редактирования;
- преобразование одного контракта в другой;
- извлечение типов из функций и конфигов.

Если писать всё руками, возникают:

- копипаст;
- рассинхрон между похожими типами;
- ломкие изменения при рефакторинге.

`Generics` и type composition закрывают именно это.

---

## Начни с простого generic, а не с магии

```ts
type ApiSuccess<TData> = {
  status: 'success';
  data: TData;
};

type ApiError = {
  status: 'error';
  message: string;
};

type ApiResponse<TData> = ApiSuccess<TData> | ApiError;
```

Польза здесь не в том, что появился `<TData>`, а в том, что инвариант ответа описан один раз.

Теперь один и тот же контракт можно использовать и для `User`, и для `Order`, и для `FeatureFlag`.

---

## Generic constraints: не пускай любой тип туда, где нужен конкретный контракт

```ts
function getEntityId<T extends { id: string }>(entity: T): string {
  return entity.id;
}
```

Без ограничения `T extends { id: string }` функция была бы слишком широкой и обещала бы больше, чем реально умеет.

Практическое правило:

- если внутри generic-функции ты обращаешься к свойствам, это почти всегда знак, что нужен `constraint`;
- чем точнее ограничение, тем понятнее ошибка компилятора и поведение API.

---

## `mapped types`: когда нужно пройтись по ключам объекта

```ts
type ValidationErrors<T> = {
  [K in keyof T]?: string[];
};

type ProfileForm = {
  email: string;
  displayName: string;
};

type ProfileFormErrors = ValidationErrors<ProfileForm>;
```

Такой тип даёт понятную связь:

- есть форма;
- есть словарь ошибок по тем же полям;
- если поле изменилось, ошибки обновятся вместе с ним.

Вот ради этого `mapped types` и нужны. Не ради впечатления коллег на ревью.

---

## `conditional types`: полезны там, где есть реальная ветка поведения

```ts
type ExtractArrayItem<T> = T extends readonly (infer U)[] ? U : T;

type A = ExtractArrayItem<string[]>; // string
type B = ExtractArrayItem<number>; // number
```

Это прикладной пример: тип работает по-разному в зависимости от структуры входа.

Но здесь есть важная граница:

- если `conditional type` реально убирает дубли или фиксирует повторяющееся правило, он полезен;
- если его невозможно объяснить в двух-трёх фразах, ты, скорее всего, уже переписал бизнес-логику на типовом языке.

---

## `template literal types`: хорошо для строгих строковых контрактов

```ts
type ThemeMode = 'light' | 'dark';
type ThemeAttribute = `theme:${ThemeMode}`;
```

Это полезно, когда строка должна подчиняться формату:

- event names;
- cache keys;
- feature-flag names;
- CSS variable namespaces;
- route params contract.

Если формат не несёт ценности и живёт только ради типовой красоты, лучше не усложнять.

---

## Что нужно знать

- как писать generic-функции и generic-типы;
- когда нужен `constraint`;
- как устроены `mapped types` через `[K in keyof T]`;
- как работает `conditional type` на уровне "если форма типа такая, верни один результат, иначе другой";
- зачем нужен `infer` для вытаскивания вложенного типа.

---

## Что нужно уметь

- выносить повторяющийся типовой паттерн в generic;
- не делать generic шире, чем реально нужно;
- строить прикладные `mapped types` для форм, фильтров, словарей ошибок и преобразований DTO;
- писать `conditional types`, которые сокращают код, а не усложняют чтение.

---

## Типичные ошибки

1. Пишут generic "на вырост", хотя реального повторения ещё нет.
2. Не ставят ограничения и получают слишком широкие, бесполезные типы.
3. Делают conditional types настолько сложными, что читать их уже невозможно.
4. Прячут смысл типов за абстрактными именами вроде `T`, `U`, `X`, `Y` на пять уровней вглубь.
5. Пытаются решить типами проблему, которую проще решить на уровне API или runtime-кода.

---

## Anti-pattern

"Чем сложнее тип, тем сильнее инженер".

Нет. Сильный инженер умеет остановиться там, где тип уже помогает, и не тащит весь проект в type-level квест.

---

## Как проверить понимание

1. Возьми два почти одинаковых типа ответа API и собери общий generic-контракт.
2. Построй `mapped type` для ошибок формы или флагов загрузки по ключам модели.
3. Объясни, где `conditional type` действительно нужен, а где это просто красивый синтаксис без пользы.

Мини-задача:

```ts
type AsyncState<TData> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: TData }
  | { status: 'error'; message: string };
```

Что стоит сделать дальше:

- добавить generic в компонент или хук, который работает с разными `TData`;
- типизировать словарь ошибок для `success`-модели через `mapped type`;
- проверить, не слишком ли рано ты полез в сложную типовую абстракцию.
