# 5.4 Utility types

`Utility types` нужны не для зубрёжки списка из документации. Они нужны, чтобы не переписывать один и тот же типовой паттерн руками и не плодить полурабочие клоны контрактов.

---

## Что уже должно быть знакомо

- моделирование данных через `type`, `interface`, `keyof`, `indexed access`: [5.2 Моделирование структур данных](./5.2-моделирование-структур-данных.md);
- generics, `mapped types` и базовая типовая композиция: [5.3 Generics, conditional и mapped types](./5.3-generics-conditional-и-mapped-types.md).

## 🎯 Что ты освоишь

- когда нужны `Partial`, `Required`, `Pick`, `Omit`, `Record`, `Exclude`, `Extract`, `NonNullable`;
- как использовать `Parameters`, `ReturnType`, `Awaited`, `InstanceType` без лишней магии;
- почему utility types ускоряют разработку только тогда, когда не скрывают смысл контракта.

---

## Термины урока

- `Utility type` — это готовая типовая утилита из TypeScript, которая преобразует существующий тип по стандартному правилу.
- `Patch contract` — это тип для частичного обновления сущности.
- `Projection` — это "вырезка" части исходного типа, например через `Pick`.

---

## Зачем это нужно на практике

В реальном проекте одна и та же сущность живёт в нескольких контекстах:

- полная доменная модель;
- форма создания;
- форма редактирования;
- кусок данных для карточки списка;
- ответ API;
- тип результата функции или хука.

Если каждый раз описывать всё руками, ты получаешь:

- копипаст;
- рассинхрон при изменениях;
- "забытые поля", из-за которых ревью становится лотереей.

Utility types уменьшают этот шум.

---

## База, которую реально используют чаще всего

```ts
type User = {
  id: string;
  email: string;
  displayName: string;
  avatarUrl?: string;
};

type UserPreview = Pick<User, 'id' | 'displayName' | 'avatarUrl'>;
type UserPatch = Partial<Pick<User, 'displayName' | 'avatarUrl'>>;
type UserWithoutAvatar = Omit<User, 'avatarUrl'>;
type RequiredUser = Required<User>;
```

Что здесь важно:

- `Pick` берёт только нужные поля;
- `Omit` исключает поля;
- `Partial` превращает поля в необязательные;
- `Required` делает все поля обязательными.

Это нормальная прикладная работа с контрактами, а не "синтаксический фокус".

---

## `Record` полезен, когда ключи реально образуют множество

```ts
type RequestStatus = 'idle' | 'loading' | 'success' | 'error';

const statusLabels: Record<RequestStatus, string> = {
  idle: 'Ещё не запускали',
  loading: 'Загружаем',
  success: 'Готово',
  error: 'Ошибка',
};
```

Почему это хорошо:

- набор ключей ограничен;
- если забыть одно состояние, компилятор это поймает.

Почему `Record<string, string>` часто плох:

- он обещает, что любой строковый ключ допустим;
- реальный контракт теряется.

---

## `Exclude`, `Extract`, `NonNullable`: чистка union-типов

```ts
type RequestStatus = 'idle' | 'loading' | 'success' | 'error';

type FinishedStatus = Exclude<RequestStatus, 'idle' | 'loading'>;
type ActiveStatus = Extract<RequestStatus, 'loading' | 'success'>;
type SafeName = NonNullable<string | null | undefined>;
```

Такие утилиты полезны, когда нужно вырезать часть состояния или убрать `null/undefined` из уже существующего типа.

Но если ты начинаешь строить из `Exclude` и `Extract` многослойную башню, возможно, пора пересмотреть исходную модель.

---

## Utility types для функций

```ts
async function fetchUser(id: string) {
  return { id, displayName: 'Alex' };
}

type FetchUserParams = Parameters<typeof fetchUser>;
type FetchUserResult = Awaited<ReturnType<typeof fetchUser>>;
```

Это удобно, когда:

- тип функции уже существует;
- ты не хочешь дублировать её сигнатуру в другом месте;
- нужно связать хук, тестовый double или adapter с существующим API.

Практическое правило:

- сначала проверь, можно ли просто экспортировать явный тип;
- если нет, извлечение через `Parameters` и `ReturnType` вполне нормально.

---

## Что нужно знать

- основные object-утилиты: `Partial`, `Required`, `Pick`, `Omit`, `Record`;
- union-утилиты: `Exclude`, `Extract`, `NonNullable`;
- function-утилиты: `Parameters`, `ReturnType`, `Awaited`, `InstanceType`;
- границы применимости: utility types помогают переиспользовать контракт, но не обязаны заменять нормальное именование модели.

---

## Что нужно уметь

- собирать create/edit/view-контракты из одной доменной модели;
- использовать `Record` только там, где ключи реально ограничены;
- вытаскивать тип параметров и результата из функции без дублирования;
- видеть, когда utility type делает модель удобнее, а когда прячет её смысл.

---

## Типичные ошибки

1. Делают `Partial<Entity>` для patch-API, хотя часть полей нельзя менять вообще.
2. Используют `Omit` как способ "спрятать неудобное поле", вместо явной новой модели.
3. Заводят `Record<string, ...>` там, где нужен чёткий список ключей.
4. Пишут цепочки из четырёх utility types подряд, а потом никто не понимает итоговый контракт.
5. Не замечают, что derived-тип уже потерял бизнес-смысл и просит отдельного имени.

---

## Anti-pattern

"Если utility type существует, его надо применить".

Нет. Иногда честный отдельный тип `UserCardViewModel` читается лучше, чем `Pick<Omit<Partial<User>, 'email'>, 'id' | 'displayName'>`.

---

## Как проверить понимание

1. Возьми доменную модель и собери из неё:
   - тип для карточки;
   - тип для patch-запроса;
   - тип для словаря по статусам.
2. Объясни, почему в одном случае `Pick` лучше, а в другом лучше написать новый именованный тип.
3. Посмотри на `Record<string, unknown>` и скажи, не потерялся ли там реальный набор ключей.

Мини-задача:

```ts
type Order = {
  id: string;
  total: number;
  currency: 'RUB' | 'USD';
  customerEmail: string;
  createdAt: string;
};
```

Собери:

- `OrderPreview`;
- `OrderPatch`;
- `OrderFieldMap` с понятными лейблами для разрешённых ключей;
- тип результата `Awaited<ReturnType<typeof fetchOrder>>`, если у тебя уже есть функция `fetchOrder`.
