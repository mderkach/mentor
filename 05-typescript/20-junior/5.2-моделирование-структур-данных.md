# 5.2 Моделирование структур данных

На junior-уровне TypeScript перестаёт быть набором отдельных типов. Он становится способом описать данные так, чтобы код не врал о форме объекта, его обязательности и границах использования.

---

## Что уже должно быть знакомо

- базовые типы, `union`, `tuple`, `readonly`: [5.1 Базовые типы и базовые конструкции](../10-intern/5.1-базовые-типы-и-базовые-конструкции.md);
- объекты и массивы в JavaScript: [3.5 Объекты и массивы: базовый уровень](../../03-javascript/10-intern/3.5-объекты-и-массивы-базовый-уровень.md).

---

## 🎯 Что ты освоишь

- когда выбирать `type alias`, а когда `interface`;
- как описывать DTO, доменные модели и словари ключ-значение;
- как работают `optional properties`, `index signatures`, `keyof`, `typeof` и `indexed access types`;
- почему слабая модель данных ломает и UI, и API-слой, и ревью.

---

## Термины урока

- `DTO` — это тип данных на границе обмена, обычно максимально близкий к формату API.
- `Domain model` — это тип внутри приложения, где данные уже приведены к удобной для бизнеса форме.
- `Indexed access type` — это способ взять тип свойства по ключу: `User['email']`.

---

## Зачем это нужно на практике

Большая часть прикладного TypeScript живёт не в `infer` и не в метапрограммировании, а в хорошем ответе на вопрос:

"Какую форму данных мы обещаем друг другу между слоями?"

Если модель слабая:

- optional-поля плодят `undefined` в неожиданных местах;
- API-ответ тащится в UI без нормализации;
- словари превращаются в мешок `Record<string, any>`;
- команда перестаёт понимать, какие поля обязательны, а какие просто "иногда бывают".

---

## `type` и `interface`: практичное правило выбора

Оба инструмента рабочие. Не нужно строить вокруг них секту.

Практичное правило:

- `type` удобно для `union`, пересечений, alias-ов и композиции;
- `interface` удобно для объектного контракта, который ты хочешь расширять или имплементировать классом.

Если вопрос уже упирается в `implements`, общую реализацию и `abstract class`, это отдельная тема: [5.2.1 Интерфейсы, классы и abstract class](./5.2.1-интерфейсы-классы-и-abstract-class.md).

```ts
type RequestStatus = 'idle' | 'loading' | 'success' | 'error';

interface UserDto {
  id: string;
  fullName: string;
  avatarUrl?: string;
}
```

Если проект не требует `declaration merging`, можно использовать и `type` почти везде. Важнее единый стиль и понятные инварианты, чем спор "что каноничнее".

---

## DTO не равно доменная модель

Плохой сценарий:

- API присылает `snake_case`, nullable-поля и вложенные структуры;
- UI ест это напрямую;
- через месяц код весь завязан на форму ответа бэкенда.

Нормальный сценарий:

```ts
type UserDto = {
  id: string;
  full_name: string;
  avatar_url: string | null;
};

type User = {
  id: string;
  fullName: string;
  avatarUrl?: string;
};

function mapUser(dto: UserDto): User {
  return {
    id: dto.id,
    fullName: dto.full_name,
    avatarUrl: dto.avatar_url ?? undefined,
  };
}
```

Идея простая:

- `DTO` описывает чужой контракт;
- доменная модель описывает удобный для приложения контракт.

---

## Optional-поля: это не "можно забыть"

```ts
type UserProfile = {
  id: string;
  nickname?: string;
};
```

`nickname?: string` означает, что свойства может вообще не быть. Это не то же самое, что `nickname: string | null`.

Практический вывод:

- `?` используй, когда поле реально может отсутствовать;
- `null` используй, когда поле приходит всегда, но может быть пустым значением;
- не смешивай эти две модели без причины, иначе в коде появляется лишняя путаница.

---

## `keyof`, `typeof` и `indexed access`: как избежать копипаста типов

```ts
const filterConfig = {
  sortBy: 'createdAt',
  order: 'desc',
};

type FilterConfig = typeof filterConfig;
type FilterKey = keyof FilterConfig;
type SortValue = FilterConfig['sortBy'];
```

Это полезно, когда тип должен вытекать из уже существующего значения или структуры.

Что это даёт:

- меньше ручного дублирования;
- меньше риска, что объект поменяли, а тип рядом забыли обновить;
- проще переиспользовать ключи и значения в UI-компонентах и утилитах.

---

## Index signatures: осторожно, это быстрый путь к размытым контрактам

```ts
type ValidationErrors = {
  [fieldName: string]: string[];
};
```

Такой контракт допустим для словаря ошибок, feature-flags или кэша по ключу. Но у него есть цена: ты говоришь системе типов, что строковый ключ может быть любым.

Неудачный сценарий:

```ts
type User = {
  [key: string]: string;
};
```

Здесь модель уже лжёт. У реального `User` поля не "любая строка", а вполне конкретный набор.

---

## Что нужно знать

- как выбрать между `type` и `interface`;
- где нужен `optional property`, а где `null` или отдельный `union`;
- как брать тип ключей через `keyof`;
- как строить тип из объекта через `typeof`;
- как вытаскивать тип отдельного поля через `T[K]`;
- когда словарь уместен, а когда он разрушает точность модели.

---

## Что нужно уметь

- разделять внешний и внутренний контракт данных;
- строить типы так, чтобы инварианты были видны без чтения реализации;
- переиспользовать существующие типы через `keyof`, `typeof`, `indexed access`, а не копировать их руками;
- замечать места, где модель стала слишком широкой и скрывает баги.

---

## Типичные ошибки

1. Тащат `DTO` прямо в компонент и называют это "типизация уже есть".
2. Делают всё optional "на всякий случай", а потом ловят `undefined` по всему проекту.
3. Используют словарь с `string`-ключом там, где нужен явный объектный контракт.
4. Дублируют типы вручную вместо переиспользования через `typeof` и `keyof`.
5. Выбирают `interface` или `type` из идеологии, а не по удобству модели.

---

## Anti-pattern

"Если API иногда шлёт мусор, затипизируем всё как optional".

Это не делает систему устойчивее. Это только размазывает хаос по всей кодовой базе. Мусор на входе нужно либо валидировать, либо нормализовать, либо явно выделять как отдельное состояние.

---

## Как проверить понимание

1. Возьми JSON-ответ API и раздели его на `DTO` и доменную модель.
2. Объясни, где в твоём примере поле должно быть `optional`, а где `null`.
3. Покажи, как получить тип ключа и тип конкретного свойства без ручного копипаста.

Мини-задача:

```ts
type ProductDto = {
  id: string;
  title: string;
  price_cents: number;
  image_url: string | null;
};
```

Сделай из этого:

- доменную модель `Product`;
- mapper `mapProduct`;
- тип ключа `ProductField`;
- тип поля `Product['title']`.
