# 5.5 Narrowing, predicates и assertions

Большая часть прикладной безопасности в TypeScript живёт не в типах "на бумаге", а в корректном сужении типа в коде ветвлений. Именно здесь решается, доверяешь ли ты данным обоснованно или просто заставляешь компилятор замолчать.

---

## Что уже должно быть знакомо

- базовые типы, `unknown`, `never`, `union`: [5.1 Базовые типы и базовые конструкции](../10-intern/5.1-базовые-типы-и-базовые-конструкции.md);
- моделирование данных и объектных контрактов: [5.2 Моделирование структур данных](./5.2-моделирование-структур-данных.md).

## 🎯 Что ты освоишь

- как работают `typeof`, `in`, `instanceof` и discriminant-поля для сужения типа;
- когда нужны `user-defined type guards` и `assertion functions`;
- чем `type assertion` отличается от реальной проверки;
- как использовать `satisfies`, `as const` и `type predicates` без ложной безопасности.

---

## Термины урока

- `Narrowing` — это сужение широкого типа до более точного на основе доказательства в коде.
- `Discriminated union` — это union, где у каждого варианта есть общий ключ-маркер, например `status` или `kind`.
- `Type predicate` — это сигнатура функции вида `value is User`, которая сообщает TypeScript, какой тип доказан после проверки.
- `Assertion function` — это функция, которая либо бросает ошибку, либо гарантирует более узкий тип после своего вызова.
- `Const assertion` — это `as const`, которое фиксирует литеральные значения, делает свойства `readonly` и не даёт типу расшириться до более широкого.

---

## Зачем это нужно на практике

На фронтенде типы почти всегда становятся шире на границах:

- ответ API;
- роут-параметры;
- `localStorage`;
- `FormData`;
- `postMessage`;
- состояние UI до завершения загрузки.

Если сужение сделано плохо:

- код либо завален `as`;
- либо runtime падает там, где типы обещали безопасность;
- либо команда начинает не доверять TypeScript вообще.

---

## Лучший прикладной паттерн: discriminated unions

```ts
type UserState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: { id: string; name: string } }
  | { status: 'error'; message: string };

function renderUserState(state: UserState) {
  switch (state.status) {
    case 'idle':
      return 'Ничего не загружали';
    case 'loading':
      return 'Загружаем';
    case 'success':
      return state.data.name;
    case 'error':
      return state.message;
  }
}
```

Почему это хорошо:

- состояние выражено явно;
- доступ к `data` и `message` открыт только там, где он реально допустим;
- UI-код получается предсказуемым и проверяемым.

---

## `type assertion` не проверяет ничего

```ts
const payload = JSON.parse(raw) as { user: { name: string } };
```

Это не валидация. Это команда компилятору "сделай вид, что я уже всё проверил".

Такой код допустим только там, где ты действительно контролируешь источник данных и уже доказал форму раньше. На границе с внешними данными это почти всегда слишком смело.

---

## `user-defined type guard`: когда проверка повторяется

```ts
type UserDto = {
  id: string;
  name: string;
};

function isUserDto(value: unknown): value is UserDto {
  return (
    typeof value === 'object' &&
    value !== null &&
    'id' in value &&
    'name' in value
  );
}
```

Такой guard полезен, если одна и та же проверка нужна в нескольких местах.

Но есть важная граница: guard должен проверять реальную форму данных, а не просто возвращать `true`, потому что "мы так договорились". Иначе типовая безопасность будет фиктивной.

---

## `assertion functions`: когда после проверки код должен продолжаться только в валидном состоянии

```ts
function assertDefined<T>(
  value: T,
  message = 'Value is required'
): asserts value is NonNullable<T> {
  if (value === null || value === undefined) {
    throw new Error(message);
  }
}
```

После вызова:

```ts
assertDefined(user);
console.log(user.id);
```

TypeScript уже знает, что `user` не `null` и не `undefined`.

Это удобно для fail-fast сценариев, где приложение не должно ехать дальше с битым состоянием.

---

## `satisfies` и `as const`: полезные, но легко перепутать

```ts
const routes = {
  home: '/',
  profile: '/profile',
} as const;

const labels = {
  idle: 'Ждём',
  loading: 'Загрузка',
  success: 'Готово',
  error: 'Ошибка',
} satisfies Record<'idle' | 'loading' | 'success' | 'error', string>;
```

Что здесь важно понять по `as const`:

- без `as const` объектные и массивные литералы часто расширяются до более широких типов;
- `as const` фиксирует литеральные значения, делает свойства `readonly` и сохраняет максимально узкий контракт;
- `satisfies` проверяет соответствие контракту, но не размывает тип объекта до этого контракта.

Это особенно полезно для конфигов, route maps, feature flags и списков статусов.

Пример, где разница заметна сразу:

```ts
const requestState = {
  status: 'idle',
};

const fixedRequestState = {
  status: 'idle',
} as const;
```

В первом случае `status` обычно расширится до `string`.

Во втором случае `status` останется литералом `'idle'`.

Это важно, когда ты:

- строишь discriminated union по полю `status` или `kind`;
- описываешь route map, status map или feature-flag config;
- хочешь получить union-тип из объекта без ручного дублирования значений.

Практическое правило:

- `as const` полезен для конфигов и фиксированных словарей;
- `as const` плох как рефлекс "прикрутить везде, чтобы компилятор отстал";
- если объект должен потом меняться, чрезмерное сужение только создаст лишнее трение.

Важно не перепутать роли:

- `as const` не про `public`, `private` или `protected`;
- его ближайшая связь с class API в том, что он тоже делает данные `readonly`, но делает это для литерального значения, а не через модификаторы класса;
- если тебе нужно разобраться именно с `public` / `private` / `protected` / `readonly` в классах, смотри [5.2.1 Интерфейсы, классы и abstract class](./5.2.1-интерфейсы-классы-и-abstract-class.md).

## Что нужно знать

- стандартные способы narrowing через `typeof`, `in`, `instanceof`, сравнение по discriminant-полю;
- разницу между `type assertion`, `type predicate` и `assertion function`;
- как строить `discriminated union` для состояния UI;
- что именно делает `as const` и чем оно отличается от обычного `const`;
- в каких случаях `satisfies` полезнее, чем явная аннотация.

---

## Что нужно уметь

- проектировать state-модели через discriminated union;
- писать честные `type guards`, которые проверяют данные по-настоящему;
- использовать `assertion functions` для fail-fast путей;
- использовать `as const` там, где нужен узкий литеральный контракт, а не "заморозка всего подряд";
- избегать `as`, если проблема решается нормальной проверкой.

---

## Типичные ошибки

1. Подменяют проверку типов `as SomeType`.
2. Делают guard, который проверяет только одно поле и объявляет весь объект валидным.
3. Строят состояние UI через пачку `boolean`-флагов вместо одного `union`.
4. Используют truthy-check там, где важно отличать `0`, `''`, `false` и `null`.
5. Применяют `as const` ко всему подряд, не понимая, зачем им нужен настолько узкий тип.

---

## Anti-pattern

"Если TypeScript ругается, значит нужно привести тип руками".

Чаще наоборот: если TypeScript ругается, значит тебе не хватает доказательства в коде. Нужно или проверить данные, или изменить модель, или честно признать границу unsafe-кода.

---

## Как проверить понимание

1. Возьми ответ API с несколькими вариантами состояния и опиши его через discriminated union.
2. Напиши `type guard` для объекта из `unknown`.
3. Покажи место, где `assertion function` удобнее, чем постоянные `if`-проверки.

Мини-задача:

```ts
type UploadState =
  | { kind: 'idle' }
  | { kind: 'uploading'; progress: number }
  | { kind: 'done'; url: string }
  | { kind: 'failed'; reason: string };
```

Проверь, что:

- доступ к `progress` возможен только в ветке `uploading`;
- ветки обработки исчерпывают все случаи;
- нет слепых `as UploadState`.
