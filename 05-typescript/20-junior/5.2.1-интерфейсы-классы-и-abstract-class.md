# 5.2.1 Интерфейсы, классы и abstract class

На этом уровне разработчик часто уже умеет писать `type` и `interface`, но начинает путаться в более неприятном вопросе: где у нас просто контракт формы, где реальный runtime-объект, а где общая базовая реализация.

Именно здесь появляются странные решения вроде:

- "сделаем `abstract class`, потому что интерфейс слишком простой";
- "проверим `instanceof Interface`";
- "засунем общую логику в базовый класс, а потом унаследуемся ещё три раза".

Этот урок нужен, чтобы развести `interface`, `class` и `abstract class` по ролям, а не по вкусовщине.

## 🎯 Что ты освоишь

- чем `interface` отличается от `class` и почему интерфейсы не существуют в runtime;
- когда использовать `implements`, `extends` и `abstract` без лишней иерархии;
- в каких сценариях `abstract class` полезен как общий каркас, а когда достаточно интерфейса или композиции;
- как не перепутать типовой контракт с реальным поведением объекта.

---

## Термины урока

- `Interface` — это compile-time контракт формы объекта или поведения, который исчезает после компиляции.
- `Class` — это runtime-конструкция JavaScript: у неё есть конструктор, экземпляры, методы и prototype chain.
- `Abstract class` — это базовый класс, который нельзя создать напрямую, но можно наследовать, чтобы переиспользовать общую реализацию и обязать потомков дописать недостающие части.
- `Implements` — это проверка, что класс соответствует интерфейсу по форме.

---

## Зачем это нужно на практике

На фронтенде эта тема всплывает в очень приземлённых местах:

- adapters над `localStorage`, `sessionStorage`, analytics SDK;
- transport-слои над REST/WebSocket/API-клиентами;
- обвязка библиотек, где нужен единый контракт и несколько реализаций;
- shared-код, где хочется переиспользовать общую логику, но не утащить проект в глубокое наследование.

Если не развести роли конструкций, обычно происходят две неприятные вещи:

- либо всё делают классами, хотя хватило бы интерфейса и пары функций;
- либо пытаются описать runtime-поведение чисто типами и удивляются, почему `instanceof` не работает.

---

## Главное различие: `interface` живёт только в типовой системе

```ts
interface Logger {
  log(message: string): void;
}

class ConsoleLogger implements Logger {
  log(message: string) {
    console.log(message);
  }
}
```

Что здесь важно:

- `Logger` помогает компилятору проверить форму контракта;
- в итоговом JavaScript интерфейса не будет вообще;
- `ConsoleLogger` останется настоящим классом, который можно создать через `new`.

Практический вывод:

- интерфейс подходит для соглашения между модулями;
- класс подходит, когда тебе нужен настоящий объект с состоянием и поведением;
- проверка через `instanceof` работает только с классом, но не с интерфейсом.

---

## Практичное правило выбора

- `interface` выбирай, когда нужен контракт без общей реализации.
- `class` выбирай, когда тебе реально нужен экземпляр, конструктор, состояние, методы или `instanceof`.
- `abstract class` выбирай, когда кроме контракта у нескольких реализаций есть общая рабочая логика и единый инвариант.
- если общей логики нет, `abstract class` почти всегда хуже интерфейса.
- если общую логику можно вынести в обычные функции или композицию, это часто проще и прозрачнее наследования.

---

## `implements` не добавляет поведение

```ts
interface NotificationChannel {
  send(message: string): Promise<void>;
}

class EmailChannel implements NotificationChannel {
  async send(message: string) {
    console.log('email:', message);
  }
}
```

`implements NotificationChannel` не делает класс "потомком интерфейса" в runtime. Это только проверка формы.

Типичный junior-баг здесь такой: разработчик видит `implements` и думает, что дальше можно как-то получить интерфейс в runtime или проверить его через `instanceof`. Нельзя. Для runtime-проверок нужны класс, явный `tag`, schema-валидация или обычная проверка свойств.

---

## `abstract class`: когда это оправдано

Хороший сценарий для `abstract class` выглядит так:

- есть общий алгоритм или инвариант;
- часть шагов одинаковая для всех реализаций;
- часть шагов должна быть реализована по-разному;
- хочется, чтобы базовый слой не забыли использовать одинаково.

```ts
interface FeatureFlagSource {
  get(name: string): boolean;
}

abstract class BaseFeatureFlagSource implements FeatureFlagSource {
  protected normalizeName(name: string): string {
    return name.trim().toLowerCase();
  }

  protected abstract read(name: string): boolean | undefined;

  get(name: string): boolean {
    return this.read(this.normalizeName(name)) ?? false;
  }
}

class MemoryFeatureFlagSource extends BaseFeatureFlagSource {
  constructor(private readonly flags: Record<string, boolean>) {
    super();
  }

  protected read(name: string): boolean | undefined {
    return this.flags[name];
  }
}
```

Почему здесь `abstract class` уместен:

- общий код нормализации и fallback не дублируется;
- наружу торчит маленький и понятный контракт `FeatureFlagSource`;
- конкретная реализация знает только про свой источник данных.

---

## Когда `abstract class` уже лишний

Плохой сценарий:

- в базовом классе почти нет общей логики;
- он нужен только ради "красивой архитектуры";
- потомки наследуют кучу состояния и скрытых зависимостей;
- изменения в базовом классе неожиданно ломают все реализации.

В таких случаях честнее:

- оставить только `interface`;
- вынести общую логику в функцию или helper;
- собрать поведение через композицию, а не через иерархию.

Короткое правило:

- один общий метод и пять защищённых полей в базе — уже тревожный сигнал;
- абстрактный класс без реальной общей реализации обычно не нужен.

---

## `extends` и `implements`: не одно и то же

```ts
interface Serializable {
  serialize(): string;
}

abstract class BaseWidget {
  constructor(protected readonly id: string) {}
}

class ToastWidget extends BaseWidget implements Serializable {
  serialize(): string {
    return JSON.stringify({ id: this.id });
  }
}
```

Здесь:

- `extends BaseWidget` даёт реальное наследование поведения и состояния;
- `implements Serializable` проверяет, что класс умеет `serialize`.

Практический trade-off:

- интерфейсов можно реализовать несколько;
- наследоваться можно только от одного класса;
- поэтому базовый класс стоит заводить только тогда, когда это реально окупается общей реализацией.

Ещё одна полезная граница: не путай class-модификаторы и `as const`.

- `public`, `private`, `protected` управляют видимостью членов класса;
- `readonly` в классе запрещает перезапись поля после инициализации;
- `as const` не про доступ к полям класса, а про сужение литерального значения и `readonly` для объекта или массива.

Когда речь именно про literal config, route map или status map, смотри также [5.5 Narrowing, predicates и assertions](./5.5-narrowing-predicates-и-assertions.md): там `as const` разобран отдельно.

---

## Что нужно знать

- что `interface` исчезает после компиляции и не существует в runtime;
- разницу между `implements` и `extends`;
- роль `abstract`, `protected`, `public`, `private`, `readonly` в классовом API;
- когда `abstract class` полезен для общего каркаса, а когда достаточно интерфейса и композиции.

---

## Что нужно уметь

- выбрать между интерфейсом, классом и абстрактным классом по реальной задаче, а не по привычке;
- описать контракт через `interface`, а общую реализацию держать отдельно или в `abstract class`, если это оправдано;
- не пытаться решать runtime-проверки чисто типовыми конструкциями;
- замечать момент, когда иерархия классов начала расти быстрее пользы от неё.

---

## Типичные ошибки

1. Проверяют `instanceof` там, где есть только интерфейс.
2. Делают `abstract class`, хотя общей логики почти нет.
3. Прячут бизнес-правила в глубокой цепочке наследования.
4. Думают, что `implements` автоматически даёт общий код.
5. Используют классы там, где хватило бы plain object, функции или composition.

---

## Anti-pattern

"Если есть несколько похожих реализаций, срочно нужен базовый абстрактный класс".

Нет. Сначала спроси:

- есть ли реально общий рабочий код;
- есть ли единый инвариант;
- не проще ли собрать это через функцию-фабрику, adapter или композицию.

Если ответы слабые, наследование только увеличит связность.

---

## Как проверить понимание

1. Возьми два адаптера с похожим контрактом и реши, где достаточно интерфейса, а где уже оправдан `abstract class`.
2. Объясни, почему `instanceof` не работает с интерфейсом.
3. Посмотри на базовый класс в проекте и скажи, что в нём реально общее: поведение, состояние или только название.

Мини-задача:

- есть `ApiTransport`, который должен уметь `request`;
- есть `FetchTransport` и `MockTransport`;
- у всех транспортов одинаковая нормализация URL и обработка базового префикса;
- у `MockTransport` и `FetchTransport` разная реализация отправки запроса.

Что нужно сделать:

- описать публичный контракт;
- решить, нужен ли здесь `abstract class`;
- показать, где хранится общая логика, а где конкретная реализация.
