# 5.8 Практики типобезопасности и runtime-схемы

На senior-уровне вопрос уже не в том, умеешь ли ты писать `union` и `generic`. Вопрос в другом: где проходит граница между тем, что TypeScript может гарантировать на этапе компиляции, и тем, что всё равно нужно проверять в runtime.

Если эту границу не держать, проект получает красивую типовую систему, которая уверенно врёт на входе из сети.

---

- честное narrowing и `assertion functions`: [5.5 Narrowing, predicates и assertions](../20-junior/5.5-narrowing-predicates-и-assertions.md);
- `tsconfig`, модульная архитектура и типовой toolchain: [5.7 Компилятор, tsconfig и сборка](../30-middle/5.7-компилятор-tsconfig-и-сборка.md).

## 🎯 Что ты освоишь

- как строить type-safe API client без ложной уверенности;
- зачем нужны runtime-схемы (`zod`, `valibot`, schema-to-type) и где они действительно окупаются;
- когда branded/opaque/nominal-подход помогает, а когда создаёт шум;
- как `exhaustive checking` и `assertNever` снижают риск незакрытых сценариев.

---

## Термины урока

- `Runtime schema` — это исполняемое описание данных, которое может реально провалидировать вход.
- `Schema-to-type` — это подход, где типы выводятся из runtime-схемы, а не дублируются вручную.
- `Branded type` — это способ добавить типовой смысл значению, которое в runtime остаётся тем же `string` или `number`.
- `Exhaustive checking` — это проверка, что все варианты union-типа действительно обработаны.

---

## Зачем это нужно на практике

Типичная ошибка зрелых команд звучит так:

"У нас же всё затипизировано".

Проблема в том, что внешние данные не читают твой `tsconfig`.

Реальные риски:

- backend изменил форму ответа;
- флаг пришёл строкой вместо boolean;
- идентификаторы разных сущностей перепутали между собой;
- switch по union-состоянию не обновили после добавления нового кейса.

Здесь и начинается senior-практика: типы должны работать вместе с runtime-проверкой и архитектурой границ.

---

## Type-safe API client: типы полезны только вместе с границей данных

Плохой вариант:

```ts
async function getUser(): Promise<User> {
  const response = await fetch('/api/user');
  return response.json();
}
```

Здесь обещание `Promise<User>` ничем не доказано.

Более честный вариант:

```ts
import { z } from 'zod';

const UserSchema = z.object({
  id: z.string(),
  email: z.string().email(),
  role: z.enum(['admin', 'editor', 'viewer']),
});

type User = z.infer<typeof UserSchema>;

async function getUser(): Promise<User> {
  const response = await fetch('/api/user');
  const payload: unknown = await response.json();
  return UserSchema.parse(payload);
}
```

Теперь есть и runtime-проверка, и единый источник истины для типа.

---

## Где TypeScript даёт ложное чувство безопасности

Это одна из самых полезных взрослых тем во всём разделе.

TypeScript создаёт ложную безопасность, когда:

- внешний payload объявили типом без runtime-проверки;
- narrowing держится на хрупкой неявной договорённости;
- `as SomeType` скрывает дыру вместо доказательства;
- generic-конструкция выглядит умно, но ухудшает поддержку;
- типовая строгость есть, а boundary всё равно небезопасен.

Практическое правило:

- типы — это доказательство только внутри той части мира, которую ты реально контролируешь;
- всё, что приходит извне, остаётся подозрительным до runtime-проверки.

---

## Narrowing и assertions: где легко соврать самому себе

Плохой паттерн:

```ts
const payload = (await response.json()) as User;
```

Здесь нет никакого настоящего narrowing. Есть только желание, чтобы проблема исчезла из редактора.

С тем же риском связаны:

- неустойчивые user-defined predicates;
- `non-null assertion` без реального инварианта;
- каскады `as unknown as`.

Если narrowing нельзя объяснить через реальное знание о данных, это не безопасность, а косметика.

---

## Не всякий "умный тип" окупается

Иногда команда доказывает, что типовую задачу можно выразить. Но зрелый вопрос звучит иначе:

"Стоит ли её так выражать?"

Если тип:

- трудно читать в hover;
- тяжело дебажить;
- ломает onboarding;
- требует ритуальных знаний автора,

то его цена может быть выше, чем его польза.

Особенно это важно на boundary-слое и в shared packages: consumer-у нужен надёжный и понятный контракт, а не демонстрация type-level изощрённости.

## `zod` и `valibot`: не религия, а выбор trade-off

Обе библиотеки решают схожую проблему:

- валидируют данные в runtime;
- позволяют вывести тип из схемы;
- снижают дублирование между валидацией и типами.

Что важно на senior-уровне:

- выбрать один основной подход для команды;
- не плодить разные схемные библиотеки без причины;
- договориться, на каких границах schema validation обязательна.

Обычно это:

- входы из API;
- `localStorage` и persisted state;
- query params;
- формы до отправки;
- межпакетные публичные контракты.

---

## Branded / opaque / nominal-паттерны: полезны для смысловых идентификаторов

```ts
type Brand<TValue, TBrand extends string> = TValue & {
  readonly __brand: TBrand;
};

type UserId = Brand<string, 'UserId'>;
type OrderId = Brand<string, 'OrderId'>;
```

Это помогает там, где в runtime всё выглядит как `string`, но по смыслу значения несовместимы:

- `UserId` и `OrderId`;
- `Email` и `PlainText`;
- `Cents` и `Percentage`.

Но если брендить всё подряд, код быстро превращается в шум. Такие паттерны нужны для важных бизнес-инвариантов, а не для каждой второй переменной.

---

## Exhaustive checking: защита от "добавили новый кейс и забыли обработку"

```ts
type PaymentState =
  | { kind: 'created' }
  | { kind: 'processing' }
  | { kind: 'paid' }
  | { kind: 'failed'; reason: string };

function assertNever(value: never): never {
  throw new Error(`Unhandled case: ${JSON.stringify(value)}`);
}

function getPaymentLabel(state: PaymentState): string {
  switch (state.kind) {
    case 'created':
      return 'Создан';
    case 'processing':
      return 'В обработке';
    case 'paid':
      return 'Оплачен';
    case 'failed':
      return state.reason;
    default:
      return assertNever(state);
  }
}
```

Этот паттерн особенно ценен в:

- reducer-ах;
- state machines;
- обработке статусов API;
- feature flag flows;
- orchestration UI-сценариев.

---

## Что нужно знать

- где compile-time типы достаточно сильны, а где нужен runtime-schema layer;
- как строить API-контракт через schema-to-type;
- зачем нужны branded/opaque/nominal-подходы;
- как использовать `assertNever` и exhaustive checking в важных state-flow.

---

## Что нужно уметь

- выделять границы, где runtime validation обязательна;
- не дублировать тип и схему там, где это можно избежать;
- проектировать shared contracts для нескольких команд или пакетов;
- делать переходы по union-состояниям исчерпывающими и проверяемыми.

---

## Типичные ошибки

1. Возвращают `Promise<User>` из API-клиента без валидации внешних данных.
2. Дублируют схему и тип вручную, а потом поддерживают два расходящихся источника истины.
3. Брендят каждую сущность подряд и получают тяжёлый DX без реальной выгоды.
4. Не закрывают все ветки `switch`, потому что "потом добавим".
5. Считают, что строгий TypeScript сам по себе заменяет runtime-контроль.
6. Путают "тип компилируется" с "граница действительно безопасна".

---

## Anti-pattern

"Runtime validation слишком дорогая, у нас же есть TypeScript".

TypeScript исчезает после сборки. Если данные приходят извне, без runtime-проверки это только уверенность без доказательства.

---

## Как проверить понимание

1. Возьми API-клиент, который сейчас просто кастит `response.json()` в `User`, и покажи, как сделать его честным.
2. Назови 2-3 места в проекте, где брендированный тип реально снизит риск перепутать значения.
3. Найди union-состояние в reducer-е или компоненте и проверь, есть ли там exhaustive checking.

Мини-задача:

- есть `OrderStatus = 'draft' | 'submitted' | 'paid' | 'cancelled'`;
- есть экран заказа и серверный ответ;
- в проекте уже были баги, когда новый статус забывали обработать.

Что нужно сделать:

- завести schema validation на входе;
- привести state-модель к discriminated union;
- закрыть рендер и бизнес-ветки через exhaustive checking.
