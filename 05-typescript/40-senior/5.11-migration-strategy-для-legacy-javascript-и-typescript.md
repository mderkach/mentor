# 5.11 Migration strategy для legacy JavaScript и TypeScript

## Уровень

Senior

## Кратко о теме

Миграция в TypeScript почти никогда не проваливается из-за синтаксиса. Она проваливается из-за стратегии:

- пытаются переписать всё сразу;
- меряют успех числом `.ts` файлов;
- заливают проект `any` и `@ts-ignore`;
- не разводят режимы "понять кодовую базу", "получить early wins" и "дожать strictness".

Сильная миграция выглядит скучно, но работает: маленькие шаги, ясные метрики, контроль долга и совместимость с delivery.

## Зачем это нужно на практике

- Чтобы команда получала пользу от TypeScript уже во время миграции, а не "когда-нибудь потом".
- Чтобы не ломать velocity продукта из-за большой типовой кампании.
- Чтобы legacy JS не застрял навечно в серой зоне между `checkJs` и `as any`.
- Чтобы migrations не создавали больше ложной безопасности, чем реальной.

## Что ты освоишь

- Как использовать `allowJs`, `checkJs`, JSDoc и пофайловую миграцию без самообмана.
- Почему migration plan должен быть risk-based, а не "по очереди все файлы".
- Как поднимать strictness постепенно и измеримо.
- Как работать с `ts-ignore` debt, не превращая его в вечное состояние проекта.

## Термины урока

- `Migration ratchet` — правило, по которому качество и strictness можно только повышать или удерживать, но не откатывать молча.
- `allowJs` — режим, в котором TypeScript принимает `.js` как вход.
- `checkJs` — режим, в котором JS-файлы начинают реально получать типовые ошибки.
- `JSDoc typing` — промежуточный слой типовой информации для JS-кода без немедленного перевода в `.ts`.

## Не начинай миграцию с массового rename

Самая популярная ошибка:

- берут сотни `.js` файлов;
- переименовывают их в `.ts`;
- получают avalanche из ошибок;
- начинают забивать их `any`, `as unknown as` и `ts-ignore`.

Это создаёт иллюзию прогресса и почти не даёт инженерной ценности.

Практичнее идти так:

1. Включить `allowJs`.
2. Выстроить `outDir` и безопасный compile flow.
3. Включать `checkJs` локально или точечно.
4. Типизировать JSDoc на hot spots и boundary modules.
5. Переводить в `.ts` сначала устойчивые и полезные участки.
6. Поднимать strictness как ratchet, а не как разовую революцию.

## `allowJs` и `checkJs` дают мягкий вход, если использовать их честно

Официальная документация TypeScript хорошо показывает это назначение:

- `allowJs` позволяет включить существующие JS-файлы в проект;
- `checkJs` добавляет к ним реальную проверку;
- JSDoc помогает получить типовую информацию без runtime-импортов.

Это особенно полезно, когда:

- legacy слишком большой для резкого перевода;
- часть кода почти не меняется, но её границы важны;
- команде нужно сначала выявить проблемные зоны, а не переписать всё.

Но здесь есть ловушка: если включить `allowJs` и на этом остановиться, миграция просто становится видимостью контроля.

## Мигрируй по risk surface, а не по алфавиту файлов

Приоритеты обычно такие:

- boundary-модули: API clients, storage, router, feature flags;
- shared utils с высоким fan-out;
- код с частыми регрессиями;
- публичные контракты между пакетами;
- только потом большие объёмы низкорискового glue-кода.

Причина проста: типизация там даёт наибольший инженерный рычаг.

Плохой план:

- "идём по папке `src/components` сверху вниз".

Хороший план:

- "сначала закрываем API boundary, shared types и hot path modules, где ошибки дорогие".

## Strictness надо поднимать как управляемую шкалу

Рабочий подход:

- базово включить проверки, которые дают много сигнала без массового шума;
- отдельно вести backlog по `noImplicitAny`, `strictNullChecks`, `noImplicitThis`, `noUncheckedIndexedAccess` и другим сильным правилам;
- зафиксировать policy: новый код не хуже старого, migrated file не должен терять strictness.

Если strictness поднимать хаотично, команда будет жить в двух реальностях:

- часть кода реально типобезопасна;
- часть формально "под TS", но дырява настолько же, как и до миграции.

## `@ts-ignore` и `any` должны быть контролируемым долгом

Они не запрещены полностью. Но они обязаны быть:

- локальными;
- объяснёнными;
- ограниченными по времени;
- заметными в ревью и отчётности.

Хороший policy:

- предпочитать `@ts-expect-error`, если ошибка ожидаема и проверяема;
- рядом писать причину;
- вести счётчик suppressions;
- не позволять migration PR-ам массово добавлять `any` без owner-решения.

## Временные компромиссы бывают допустимыми, а бывают токсичными

Допустимые временные компромиссы:

- локальный `@ts-expect-error` с причиной;
- JSDoc typing на переходном участке;
- boundary adapter, который пока ещё не идеален, но явно изолирует грязный внешний мир;
- `unknown` там, где честно ещё нет доказательства типа.

Токсичные компромиссы:

- массовый `any` без плана удаления;
- `as any` в boundary-модулях;
- suppressions без owner-а;
- "потом дожмём strictness", когда метрик и ratchet-механики нет.

Зрелая миграция не запрещает временные уступки. Она делает их явными, измеримыми и ограниченными.

## Кейс: продукт на 80% JS, релизы каждую неделю

Плохой migration plan:

- двухмесячная заморозка фич;
- rename всего `src`;
- гора suppressions;
- после релиза команда ненавидит TS.

Зрелый plan:

- root config с `allowJs`;
- boundary-модули получают `checkJs` и JSDoc;
- новые фичи сразу в `.ts`/`.tsx`;
- старые файлы переводятся рядом с изменениями;
- strictness растёт на control points;
- каждую неделю видно не только количество migrated files, но и сокращение suppressions в risk zones.

## Что нужно знать

- роль `allowJs`, `checkJs`, JSDoc и пофайловой миграции;
- разницу между migration progress и cosmetic progress;
- подход к strictness как ratchet;
- цену suppressions и `any` в долгой миграции.

## Что нужно уметь

- собирать migration plan без остановки продукта;
- выбирать modules для миграции по риску и ценности;
- использовать JS+JSDoc как промежуточный устойчивый этап;
- задавать измеримые критерии успеха миграции.

## Типичные ошибки

1. Считают количество `.ts` файлов главным KPI.
2. Делают массовый rename раньше, чем выстроили compiler flow.
3. Заливают ошибки `any` и suppressions без лимита.
4. Не отделяют новый код от legacy carve-out.
5. Путают "миграция идёт" с "система стала безопаснее".
6. Делают временный компромисс постоянной архитектурой проекта.

## Anti-pattern

"Сначала переведём всё в TypeScript, а потом уже сделаем его строгим".

Чаще всего "потом" не наступает. Если strictness не растёт вместе с миграцией, ты просто получаешь TypeScript-файлы с той же старой неопределённостью.

## Как проверить понимание

1. Возьми legacy-проект и назови первые 5 модулей для миграции по risk/impact, а не по размеру.
2. Объясни, зачем на промежуточном этапе нужны `allowJs` и `checkJs`.
3. Предложи политику для `@ts-ignore` / `@ts-expect-error`.
4. Назови метрики, которые реально показывают пользу миграции.

Признак освоения: ты можешь провести миграцию как инженерную программу изменений, а не как кампанию по переименованию файлов.
