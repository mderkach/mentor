# 5.13 Boundary types в monorepo и microfrontends

## Уровень

Senior

## Кратко о теме

Внутри одной кодовой базы очень легко создать иллюзию безопасности:

- "мы же делим одни и те же типы";
- "значит контракты между пакетами уже безопасны".

Это работает ровно до тех пор, пока:

- пакеты начинают версионироваться отдельно;
- команды имеют разный cadence;
- часть системы деплоится независимо;
- shared package превращается в dumping ground.

На senior-уровне важно понимать: boundary types должны отражать реальные организационные и runtime-границы, а не только структуру монорепы.

## Зачем это нужно на практике

- Чтобы shared packages не связывали систему сильнее, чем нужно.
- Чтобы microfrontend-среды не падали из-за несовместимых типов, скрытых deep imports и version skew.
- Чтобы команда различала "общий контракт" и "общая внутренняя модель".
- Чтобы TS помогал держать границы, а не маскировал их отсутствие.

## Что ты освоишь

- Как делить типы между пакетами, не деля лишние implementation details.
- Почему independently deployed части системы требуют versioned contract, а не просто общий `types` package.
- Как бороться с deep imports, shared `common-types` помойками и фантомной синхронностью монорепы.
- Как выбирать между shared package, generated contract и runtime boundary.

## Термины урока

- `Boundary type` — тип, который описывает договор между двумя частями системы.
- `Version skew` — ситуация, когда producer и consumer живут на разных версиях контракта.
- `Shared contract package` — пакет, который хранит только поддерживаемую наружу поверхность, а не все внутренние типы подряд.
- `Independent deployment` — поставка, при которой две части системы могут обновляться не одновременно.

## Shared type package полезен не всегда

Он окупается, если:

- есть понятный общий контракт;
- есть owner;
- есть semver discipline;
- пакет реально публикуется и потребляется как отдельная поверхность.

Он вреден, если:

- туда складывают все типы "чтобы было удобно";
- команды начинают зависеть от внутренних доменных структур друг друга;
- package превращается в узел глобальной связанности;
- версия обновляется по принципу "в монорепе и так всё одновременно".

Это особенно опасно в microfrontend-средах: реальная система деплоится независимо, а типы притворяются, что всё всегда обновится синхронно.

## Shared types очень легко создают ложное чувство безопасности

Ловушка звучит приятно:

"Раз producer и consumer импортируют один и тот же тип, значит они совместимы".

На практике это правда только в очень узком окне:

- одна и та же версия;
- один и тот же build graph;
- одна и та же дисциплина поставки;
- отсутствие скрытых runtime differences.

Как только части системы начинают жить отдельно, shared type package перестаёт быть доказательством совместимости и становится только одной из опор.

## Monorepo не отменяет контрактов

Монорепа технически упрощает обновление зависимостей, но не делает границы менее важными.

Наоборот, в большой монорепе легко потерять дисциплину:

- deep imports через соседние пакеты;
- общие internal types;
- зависимость на `src/*` вместо опубликованного entrypoint;
- случайное использование transitive exports.

TS может даже помогать этому хаосу, потому что локально всё типизируется и IDE счастлива.

Senior-правило:

- import должен идти только через поддерживаемый public entrypoint;
- boundary type package должен содержать только контракт;
- внутренние доменные модели не шарятся без причины.

## Microfrontends требуют думать о version skew

Если части системы деплоятся независимо, shared type package сам по себе не гарантирует совместимость.

Нужно явно решить:

- кто producer контракта;
- кто consumer;
- какой compatibility window поддерживается;
- как обрабатываются deprecated поля;
- что происходит, если consumer старее producer-а.

Именно поэтому для independently deployed boundaries часто полезнее:

- schema/codegen вокруг API;
- versioned event contracts;
- runtime validation на границе;
- более узкий shared contract package вместо общих внутренних типов.

## Не дели внутреннюю модель продукта между командами без причины

Очень частый smell:

- backend-contract package начинает содержать internal UI enums;
- domain package экспортирует всё подряд;
- microfrontend A импортирует внутренний reducer type microfrontend B;
- потом любое локальное изменение становится межкомандным событием.

Правильнее делиться:

- сообщениями;
- DTO/contract types;
- shared primitives;
- versioned public interfaces.

И не делиться:

- внутренними state-моделями;
- helper types без owner-а;
- файлами из `src/internal/*`.

## Цена типовой сложности на границе особенно высока

Boundary type должен быть:

- читаемым;
- версионируемым;
- понятным consumer-у;
- устойчивым к эволюции.

Если граница описана через тяжёлую generic-механику, которую понимают только авторы пакета, это плохая граница даже при формальной точности.

На boundary-слое лучше:

- немного проще модель;
- немного меньше "магической точности";
- но больше ясности и стабильности для consumer-а.

## Кейс: host app и два independently deployed microfrontend

Плохой вариант:

- все используют один `@shared/types`, куда сложено почти всё;
- host знает внутреннюю модель фильтра MF-A;
- MF-B импортирует internal events MF-A;
- локально всё работает, в проде после независимого релиза появляются несовместимости.

Зрелый вариант:

- host и MFE делят только versioned contracts;
- shared package минимален и owner-ed;
- runtime boundary валидируется;
- deprecated fields живут по правилам;
- deep imports запрещены policy и review.

## Что нужно знать

- разницу между shared contract и shared internals;
- риск version skew в independently deployed системах;
- роль semver, owner и runtime validation на границах;
- почему monorepo не освобождает от contract discipline.

## Что нужно уметь

- выделять boundary types отдельно от внутренних моделей;
- проектировать compatibility window для producer/consumer;
- запрещать deep imports и hidden coupling;
- выбирать между shared package, schema/codegen и runtime adapter по типу границы.

## Типичные ошибки

1. Делят одну и ту же внутреннюю модель между всеми пакетами.
2. Считают монорепу доказательством вечной синхронности версий.
3. Экспортируют всё из `common-types`, потому что "иначе неудобно".
4. Не учитывают independently deployed reality у microfrontends.
5. Используют типы как substitute for runtime compatibility.
6. Делают boundary contract типово слишком сложным для тех, кто его потребляет.

## Anti-pattern

"Раз producer и consumer написаны на TypeScript, runtime-совместимость уже обеспечена".

Нет. TypeScript проверяет то, что видит в момент сборки конкретной версии кода. Он не гарантирует совместимость двух separately released артефактов без дисциплины контрактов и версий.

## Как проверить понимание

1. Для одной границы в системе назови producer, consumer, contract owner и compatibility window.
2. Покажи, какие типы должны остаться внутренними и не попадать в shared package.
3. Найди один deep import между пакетами и объясни, почему он опасен.
4. Для microfrontend boundary реши, что лучше: shared types, schema/codegen или runtime adapter, и почему.

Признак освоения: ты строишь boundary types вокруг реальной поставки и ownership, а не вокруг удобства локальной IDE.
