# 5.9 Large-scale TypeScript: project references, composite builds и package boundaries

## Уровень

Senior

## Кратко о теме

`Project references` часто звучат как "ещё одна опция в `tsconfig`". На деле это уже язык проектирования большой кодовой базы.

Когда проект растёт, одна конфигурация на всё начинает ломаться:

- typecheck становится тяжёлым;
- редактор ест память;
- package boundaries не выражены;
- внутренние зависимости проекта видны только по импорту и договорённости;
- downstream части живут на неявных source-level связях.

Тогда TypeScript-граф нужно проектировать осознанно, как и любой другой граф зависимостей.

## Зачем это нужно на практике

- Чтобы ускорять typecheck и incremental builds без хаотичной разбивки репозитория.
- Чтобы редактор и CI видели одинаковую архитектуру проекта.
- Чтобы shared packages публиковали осмысленные `d.ts`, а не случайный набор исходников.
- Чтобы ownership и layering были выражены не только в папках, но и в build-границах.

## Что ты освоишь

- Когда стоит разбивать кодовую базу на TS-проекты, а когда это лишняя сложность.
- Как устроены solution `tsconfig`, leaf projects и build mode `tsc -b`.
- Почему `composite`, `declaration` и `declarationMap` важны для межпроектной навигации и публикации типов.
- Как проводить package boundaries по ownership и runtime reality, а не по моде на монорепы.

## Термины урока

- `Project reference` — это ссылка одного TS-проекта на другой через `tsconfig`, а не через случайный импорт исходников.
- `Composite project` — проект, который TypeScript может быстро проверить и встроить в общий build graph.
- `Solution tsconfig` — верхнеуровневый конфиг, который сам не компилирует код, а описывает граф зависимых проектов.
- `Package boundary` — явная граница ответственности и API между частями кодовой базы.

## Разбивать на проекты нужно по ownership, а не по вкусу

Плохой split:

- "папка большая, давайте нарежем по 10 подпапок";
- "у нас монорепа, значит каждый directory должен быть package";
- "references включим потом как-нибудь".

Здоровый split:

- shared UI как отдельный library-like project;
- domain modules с устойчивым API;
- приложения/endpoint-пакеты как thin consumers;
- test/tooling/server/browser targets разведены там, где у них разные среды выполнения.

Контрольный вопрос:

"Если завтра эта часть кода начнёт жить своей жизнью, будет ли у неё понятный публичный контракт?"

Если нет, скорее всего это ещё не отдельный TS-проект, а просто папка.

## Solution tsconfig нужен для графа, а не для компиляции всего подряд

Официальный паттерн TypeScript для solution-конфига очень показателен:

```json
{
  "files": [],
  "references": [
    { "path": "./packages/shared-ui" },
    { "path": "./packages/api-client" },
    { "path": "./apps/web" }
  ]
}
```

Почему это важно:

- `files: []` не даёт solution-проекту начать компилировать всё ещё раз;
- граф зависимостей становится явным;
- `tsc -b` может работать как build orchestrator;
- редактор видит структуру проекта ближе к реальности.

Если же root `tsconfig` одновременно и solution, и реальный compile target, и место для всех исключений, он быстро превращается в свалку настроек.

## `composite`, `declaration`, `declarationMap` — это не случайный boilerplate

У `project references` есть важные предпосылки:

- referenced projects должны быть `composite`;
- обычно им нужен `declaration: true`;
- `declarationMap` полезен, если команда хочет нормальную навигацию между пакетами.

Практический смысл:

- редактор и `tsc -b` опираются на `.d.ts`, а не на прямое повторное чтение всех исходников;
- build graph становится детерминированнее;
- package boundary начинает жить как реальный контракт.

Но это создаёт и цену:

- надо следить за outDir/rootDir;
- build outputs и editor-navigation требуют дисциплины;
- часть проблем всплывает только после реального build mode, а не при локальном typecheck одного пакета.

## `paths` не заменяют project boundaries

Один из самых частых анти-паттернов:

- в root `tsconfig` настраивают десятки alias;
- весь репозиторий импортирует всех через `@shared/*`, `@entities/*`, `@whatever/*`;
- формально "архитектура есть", но build graph и ownership отсутствуют.

Alias решает удобство импорта. `Project references` решают другое:

- build ordering;
- отдельные compile targets;
- публикацию деклараций;
- более жёсткие межпроектные границы.

Если кодовая база большая, одних alias почти всегда недостаточно.

## Кейс: монорепа с `shared-ui`, `api-client` и `web-app`

Хорошая структура:

- `shared-ui` — library-like project, экспортирует компоненты и их публичные типы;
- `api-client` — отдельный boundary project, где живут transport contract и runtime schema layer;
- `web-app` — endpoint project, который ссылается на оба пакета;
- root solution `tsconfig` только описывает граф;
- CI умеет `tsc -b` по затронутым или всем проектам.

Плохая структура:

- `web-app` импортирует исходники `shared-ui/src/*`;
- `api-client` и `web-app` делят типы через случайный `src/types.ts`;
- root config разруливает всё через aliases и hidden exceptions.

Во втором случае типы вроде работают. Но цена поддержки, навигации и публикации со временем резко растёт.

## Что нужно знать

- роль solution `tsconfig`, `references`, `composite`, `declaration`, `declarationMap`;
- trade-off между простым проектом и large-scale graph setup;
- разницу между alias-удобством и настоящими package boundaries;
- почему `tsc -b` важен для больших кодовых баз.

## Что нужно уметь

- определить, где разбивка на TS-проекты действительно окупается;
- проводить границы по ownership и runtime surface;
- собирать solution config без двойной компиляции и хаоса;
- видеть, где `project references` усиливают архитектуру, а где только усложняют проект.

## Типичные ошибки

1. Делят репозиторий на десятки проектов без устойчивых API.
2. Считают alias и project references взаимозаменяемыми вещами.
3. Держат root `tsconfig` как универсальный конфиг на все среды и все роли.
4. Не включают `declaration` и потом удивляются плохой межпакетной навигации.
5. Не используют `tsc -b`, хотя уже завели реальный project graph.

## Anti-pattern

"Раз проект большой, давайте просто нарежем его на packages, а TypeScript дальше сам разберётся".

Не разберётся. Без явных boundary contracts и build discipline монорепа становится просто большим репозиторием с дорогим typecheck.

## Как проверить понимание

1. Возьми текущую кодовую базу и назови 2-3 зоны, которые реально тянут на отдельные TS-проекты.
2. Объясни, почему root solution `tsconfig` должен иметь `files: []`.
3. Для одного shared package назови его публичный контракт и тип выходных артефактов.
4. Покажи, где alias сейчас скрывает отсутствие настоящей boundary discipline.

Признак освоения: ты можешь нарисовать TS project graph и объяснить каждую границу через owner, API и build reason.
