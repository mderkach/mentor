# 5.12 Schema-driven contracts и типизация внешних интеграций

## Уровень

Senior

## Кратко о теме

Внешние интеграции редко ломаются потому, что команда не умеет писать `type User = ...`. Они ломаются потому, что source of truth не определён:

- тип в фронте один;
- payload из API другой;
- persisted state живёт третьей жизнью;
- query params и feature flags вообще читаются как `string`.

Schema-driven подход нужен не ради моды. Он нужен, чтобы у boundary-слоя был один центр тяжести.

## Зачем это нужно на практике

- Чтобы API, persisted state и runtime payload-ы не расходились с типами.
- Чтобы codegen и runtime validation усиливали друг друга, а не плодили вторую схему рядом.
- Чтобы интеграции между командами были устойчивее к изменениям.
- Чтобы "у нас всё затипизировано" не означало "мы верим в структуру строки из URL".

## Что ты освоишь

- Как выбирать source of truth для контрактов внешних интеграций.
- Где schema-first оправдана, а где проще держать type-first модель с runtime adapters.
- Как кодогенерация, hand-written types и runtime schemas сосуществуют без дублирования.
- Почему query params, local storage и feature flags тоже требуют контрактного мышления.

## Термины урока

- `Source of truth` — место, из которого команда выводит контракт и которому доверяет при изменениях.
- `Schema-driven contract` — контракт, построенный вокруг явной схемы данных, а не только compile-time типа.
- `Codegen governance` — правила, по которым генерируемый код и типы входят в репозиторий, обновляются и ревьюятся.
- `Boundary adapter` — слой, который переводит внешний payload во внутреннюю модель приложения.

## Сначала реши, где живёт истина

Для разных интеграций истина может жить в разных местах:

- OpenAPI/JSON Schema на стороне платформы;
- runtime schema на стороне клиента;
- generated client поверх серверной схемы;
- hand-written adapter, если внешняя система слишком грязная.

Проблемы начинаются, когда команда пытается держать сразу всё:

- вручную написанный TS type;
- отдельно написанную runtime schema;
- отдельно codegen output;
- ещё и swagger-контракт где-то в стороне.

Если эти слои не сведены, расхождение становится вопросом времени.

## Не все внешние данные одинаково опасны, но все требуют boundary

Типичные зоны риска:

- API responses;
- query params;
- feature flags;
- persisted state в `localStorage`/IndexedDB;
- сообщения из iframe/worker;
- payload-ы аналитики и сторонних SDK.

Ошибка многих команд в том, что schema discipline ограничивается только HTTP client-ом, а остальные входы читаются как "ну тут и так строка".

На практике query params и persisted state ломают приложение не хуже кривого бэкенд-ответа.

## Shape в типовой системе и факт валидности в runtime — не одно и то же

Это стоит проговаривать прямо, иначе TypeScript начинает казаться магическим щитом.

`type SearchFilters = { ... }` отвечает на вопрос:

- как код внутри проекта собирается работать с этой структурой.

Runtime validation отвечает на другой вопрос:

- действительно ли пришедшие данные имеют эту структуру прямо сейчас.

То есть:

- тип описывает желаемую форму;
- схема и adapter доказывают допустимость реальных данных;
- без этого внешняя интеграция остаётся актом веры.

## Codegen полезен, но только вместе с ownership

Codegen решает две хорошие задачи:

- убирает ручной копипаст контрактов;
- ускоряет обновление клиентских типов после изменения серверной схемы.

Но codegen не решает автоматически:

- нужен ли runtime validation;
- как маппить серверную модель во внутреннюю доменную;
- как жить с backwards compatibility и deprecated fields;
- кто владеет обновлением generated artifacts.

Без governance generated code быстро превращается в священный лес, который никто не читает, но все боятся трогать.

## Boundary adapter почти всегда нужен, даже если схема уже есть

Причина простая: внешняя модель и внутренняя модель редко совпадают идеально.

Например:

- сервер шлёт DTO;
- UI работает с state-моделью;
- persisted draft хранит частичную форму;
- роутинг использует строковые параметры.

Если всё это бездумно объявить одним типом `User`, проект потеряет ясные границы.

Здоровый подход:

- внешний контракт валидируется или генерируется;
- boundary adapter переводит payload во внутреннюю модель;
- downstream код работает уже с проверенной и нормализованной структурой.

## Schema-driven подход окупается не везде одинаково

Он особенно полезен, если:

- граница внешняя и нестабильная;
- данных много и цена расхождения высокая;
- контракт используется несколькими командами;
- persisted/query/message payload переживает версии и окружения.

Он может быть избыточным, если:

- граница локальная и короткоживущая;
- вход полностью под контролем одного модуля;
- дублирование схемы и сложность governance дороже, чем риск.

То есть schema-driven подход — не религия. Это инженерный выбор под конкретную стоимость расхождения.

## Кейс: search page с API, query params и persisted filters

У страницы есть три источника правды:

- API response;
- query params;
- persisted filters в `localStorage`.

Плохой вариант:

- для каждого слоя свой вручную написанный тип;
- часть свойств optional "на всякий случай";
- данные смешиваются напрямую.

Зрелый вариант:

- API contract приходит из схемы или codegen;
- query params имеют свою parsing/schema boundary;
- persisted filters валидируются при чтении;
- в UI попадает единая normalized search state model.

Это и есть schema-driven thinking: не один "тип на всё", а осознанные контракты по границам.

## Что нужно знать

- разные модели source of truth для контрактов;
- роль runtime schemas, codegen и boundary adapters;
- почему query params, persisted state и external SDK тоже входят в интеграционный слой;
- как избегать многократного ручного дублирования схем и типов.

## Что нужно уметь

- выбирать contract strategy под конкретную интеграцию;
- проводить границу между внешней схемой и внутренней моделью;
- вводить codegen без потери понимания и ownership;
- делать внешние данные проверяемыми ещё до попадания в feature logic.

## Типичные ошибки

1. Держат и schema, и type, и generated types как три независимых источника истины.
2. Считают codegen заменой runtime validation.
3. Тянут внешние DTO напрямую в UI state.
4. Не типизируют query params, storage и flags, потому что "это не API".
5. Не назначают владельца generated contracts.
6. Считают, что раз тип уже есть, данные автоматически валидны в runtime.

## Anti-pattern

"Раз у нас есть OpenAPI/codegen, значит boundary проблема решена".

Нет. Codegen решает только часть механики. Нужны ещё owner, adapter, версия контракта и политика обновления.

## Как проверить понимание

1. Для одной интеграции назови её source of truth и объясни, почему именно он.
2. Покажи, где в системе должен жить boundary adapter.
3. Для query params или persisted state опиши схему чтения, валидации и нормализации.
4. Назови критерии, когда codegen окупается, а когда проще локальный contract layer.

Признак освоения: ты видишь внешние данные как набор явных контрактов и сознательно выбираешь, где схема, где codegen, а где adapter.
