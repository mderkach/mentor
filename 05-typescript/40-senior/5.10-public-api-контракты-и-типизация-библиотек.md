# 5.10 Public API контракты и типизация библиотек

## Уровень

Senior

## Кратко о теме

Как только пакет или модуль начинает использовать кто-то кроме автора, его типы перестают быть "локальным удобством". Они становятся публичным контрактом.

У этого контракта есть свойства обычного API:

- совместимость;
- понятность;
- эволюция;
- цена breaking changes;
- качество tooling experience у потребителя.

Плохая типизация библиотеки почти всегда выдаёт внутреннюю кухню наружу: странные generic-параметры, случайные helper types, deep imports и неустойчивые declaration files.

## Зачем это нужно на практике

- Чтобы shared package не ломал downstream-команды каждой мелкой реорганизацией.
- Чтобы `d.ts` совпадали с реальным runtime поведением.
- Чтобы semver учитывал типовые breaking changes, а не только JS runtime.
- Чтобы библиотека была удобна и понятна без чтения её исходников.

## Что ты освоишь

- Как формировать публичную типовую поверхность пакета.
- Почему структура `d.ts` должна отражать runtime shape библиотеки.
- Как публиковать типы через `declaration`/`types` и не тащить наружу внутренние детали.
- Какие типовые изменения являются semver-breaking, даже если JS-код почти не менялся.

## Термины урока

- `Public surface` — все экспортируемые наружу значения и типы, на которые может опереться потребитель.
- `Declaration publishing` — поставка `d.ts` вместе с пакетом или через `@types`.
- `Library contract` — совокупность runtime- и type-level гарантий, которые пакет обещает потребителям.
- `Breaking type change` — изменение типового контракта, которое ломает корректный consumer code.

## Публичный API не должен совпадать с файловой системой

Частая ошибка library-авторов:

- экспортировать почти всё "на всякий случай";
- разрешать deep imports в любой внутренний модуль;
- считать, что `src/index.ts` автоматически и есть хороший public surface.

Сильный подход другой:

- public entrypoints минимальны и осознанны;
- внутренние helper types не утекают без причины;
- у потребителя есть небольшое число стабильных точек входа;
- runtime и типы ведут себя одинаково.

Контрольный вопрос:

"Если мы завтра переименуем половину внутренних файлов, consumer code должен выжить?"

Если ответ "нет", ваш public API спроектирован плохо.

## `d.ts` должны описывать библиотеку так, как её реально потребляют

Официальная документация TypeScript прямо подводит к этому: структура declaration files зависит от того, как библиотека реально загружается и используется.

Практический вывод:

- module library описывается не так, как global script;
- ESM/CJS interop влияет на contract;
- типовая структура должна быть подчинена модели потребления, а не вкусу автора.

Плохой признак:

- пакет в рантайме экспортирует одно;
- `d.ts` обещают другое;
- потребителю для понимания надо читать bundle output и исходники одновременно.

## Semver для типов существует по-настоящему

Примеры типовых breaking changes:

- сузили union, на который опирались потребители;
- сделали optional property обязательным;
- изменили generic default или constraint;
- скрыли экспортируемый тип, который раньше входил в public surface;
- поменяли модульный entrypoint и сломали `types` resolution.

При этом runtime-код мог почти не измениться.

Senior-уровень здесь — держать в голове две совместимости:

- runtime compatibility;
- type-level compatibility.

## Public API нужно проектировать для потребителя, а не для удобства автора

У автора библиотеки почти всегда есть соблазн:

- экспортировать внутренний generic helper;
- наружу протащить точную внутреннюю модель;
- сделать тип "максимально умным", потому что так красивее внутри репозитория.

Проблема в том, что consumer-у нужен другой контракт:

- устойчивый;
- понятный;
- эволюционируемый;
- не зависящий от внутренней реализации.

Если внешнему потребителю надо понимать внутреннюю кухню библиотеки, чтобы просто использовать её API, public surface спроектирован плохо.

## Слишком точные типы тоже могут быть плохим API

Иногда тип можно сделать очень точным:

- вывести literal union из внутренних деталей;
- связать несколько generic-параметров жёстче, чем это реально нужно;
- наружу вынести shape, который повторяет внутреннее дерево реализации.

Но на публичной поверхности это часто вредно:

- API труднее менять;
- внутренние рефакторинги становятся breaking change;
- consumer code завязывается на детали, которые не должны были стать контрактом.

Хороший публичный тип не обязан быть "максимально точным". Он обязан быть достаточно точным для полезного и устойчивого использования.

## Публикация типов: bundle them, если они происходят из исходников

Официальная рекомендация TypeScript для пакетов с собственным исходным кодом проста:

- если типы генерируются из кода, публикуй их вместе с пакетом;
- если типы пишутся отдельно для чужой библиотеки, тогда уже `@types`/DefinitelyTyped.

Базовый минимум для пакета:

- `declaration: true`;
- корректный `types` entry в `package.json`;
- понятный `outDir`;
- отсутствие зависимости потребителя от `.ts`-исходников.

Если library author игнорирует это, downstream DX платит:

- IDE плохо резолвит типы;
- deep imports становятся нормой;
- type navigation работает случайно.

## Не экспортируй generic-магии больше, чем нужно

Частая проблема "умных" библиотек:

- авторы выносят наружу сложные helper types, потому что они красивые;
- публичный API становится перегруженным conditional/mapped machinery;
- потребителю нужно знать внутреннюю типовую механику, чтобы просто использовать пакет.

Лучший подход:

- внутри можешь иметь сложную type-level систему;
- наружу давай минимальный выразительный контракт;
- complex types должны окупаться реальной полезностью для consumer code.

## Кейс: UI-kit с deep imports и случайными типами

Плохая эволюция:

- `ButtonProps` живёт в одном файле, internal theme helpers в другом;
- потребители начинают импортировать `themeTokens`, `InternalButtonState`, `ButtonContextShape` напрямую;
- потом команда реорганизует файлы и ломает половину приложений.

Зрелое решение:

- есть стабильные публичные entrypoints;
- только поддерживаемые типы экспортируются снаружи;
- внутренние типы остаются внутренними;
- breaking changes по типам проходят через semver-review и migration note.

## Что нужно знать

- разницу между внутренней структурой кода и публичным API;
- роль `declaration`, `types`, `declarationMap` и package publishing;
- какие типовые изменения считаются breaking;
- почему структура declaration files должна повторять структуру потребления.

## Что нужно уметь

- проектировать малую устойчивую публичную поверхность;
- публиковать типы без утечки внутренних деталей;
- делать review типовых изменений с semver-мышлением;
- не позволять consumer code зависеть от deep imports и internal helpers.

## Типичные ошибки

1. Экспортируют внутренние типы "чтобы не мешать потребителям".
2. Ломают типовой контракт без осознания semver-последствий.
3. Публикуют пакет без `types` entry или с кривым `outDir`.
4. Держат библиотеку типобезопасной только в исходниках, но не в поставляемых `d.ts`.
5. Путают удобство автора библиотеки с удобством её потребителей.
6. Делают публичный тип слишком точным и случайно цементируют внутреннюю реализацию.

## Anti-pattern

"Если библиотека компилируется и локальные тесты зелёные, публичный типовой API уже нормальный".

Нет. Для публичного пакета TypeScript contract — часть продукта. Его надо проектировать, тестировать и версионировать отдельно.

## Как проверить понимание

1. Возьми один shared package и перечисли всё, что у него реально является public surface.
2. Назови один type change, который кажется маленьким, но является semver-breaking.
3. Проверь `package.json` пакета: куда указывает `types`, и совпадает ли это с реальным publish layout.
4. Найди хотя бы один deep import у consumer-а и объясни, почему это architectural smell.

Признак освоения: ты смотришь на типы библиотеки глазами потребителя и заранее видишь, какие экспорты станут дорогим долгом.
