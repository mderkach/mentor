# 5.1 Базовые типы и базовые конструкции

`TypeScript` — это надстройка над JavaScript со статической системой типов. Он не меняет поведение браузера, но позволяет поймать часть ошибок до запуска кода.

На этом уровне TypeScript нужен не для трюков, а для дисциплины мышления: что за данные пришли, что функция принимает, что она возвращает и где код врёт сам себе.

## 🎯 Что ты освоишь

- что такое TypeScript, какие инженерные проблемы он решает и где его границы;
- чем статическая типизация TypeScript отличается от динамической типизации JavaScript;
- чем отличаются `any`, `unknown`, `never` и `void`;
- когда нужны `literal types`, `union types`, `intersection types` и `tuple types`;
- как `readonly` помогает не ломать данные случайной мутацией;
- почему типы описывают runtime-модель, но не заменяют runtime-проверки.

---

## Термины урока

- `Static typing` — это проверка типов до запуска программы.
- `Dynamic typing` — это определение и проверка типа значения во время выполнения.
- `Static type system` — это проверка формы данных и контрактов до запуска программы.
- `Type annotation` — это явная подсказка компилятору, какой тип ожидается у значения.
- `Union type` — это тип "одно из нескольких допустимых состояний".
- `Tuple` — это массив с фиксированным порядком и смыслом элементов.
- `Readonly` — это запрет на мутацию через текущий типовой контракт.

---

## Что такое TypeScript и какие проблемы он решает

`TypeScript` не делает JavaScript "другим языком". Он добавляет слой проверок поверх обычного JS-кода.

Что он реально помогает закрыть:

- рассинхрон между тем, что функция ожидает, и тем, что ей передают;
- небезопасные рефакторинги, когда поле или параметр переименовали не везде;
- путаницу между допустимыми состояниями интерфейса;
- слабую читаемость контрактов между разработчиками и слоями приложения.

Что он не решает сам:

- не валидирует ответ API в runtime;
- не чинит плохую бизнес-логику;
- не заменяет понимание JavaScript и реального поведения объектов, массивов и функций.

Поэтому полезная ментальная модель такая:

- JavaScript отвечает за поведение кода в браузере;
- TypeScript помогает раньше заметить, что модель данных и код расходятся с этим поведением.

## JavaScript и TypeScript: динамическая и статическая типизация

Если говорить грубо и полезно для практики:

- `JavaScript` — язык с динамической типизацией: тип значения определяется и всплывает в момент выполнения;
- `TypeScript` — это JavaScript с дополнительной статической проверкой до запуска.

Важно не путать это с магией "TypeScript всё запретит". Он не делает runtime другим. Он просто заранее подсвечивает часть проблем.

Пример:

```ts
function formatPrice(price: number) {
  return price.toFixed(2);
}

formatPrice('100');
```

В чистом JavaScript такая ошибка обычно всплывает уже во время выполнения, когда строка доедет до `toFixed`.

В TypeScript проблема ловится раньше: компилятор видит, что функция ждёт `number`, а ей передали `string`.

Практический вывод:

- динамическая типизация хороша гибкостью, но ошибки часто доезжают дальше по коду;
- статическая типизация даёт раннюю обратную связь и делает рефакторинг безопаснее;
- при этом внешние данные всё равно остаются внешними: ответ API может быть битым и в TypeScript-проекте.

---

## Зачем это нужно на практике

Без этой базы TypeScript быстро превращается в плохой JavaScript с надписью `: any`.

На практике тема нужна для трёх вещей:

- ловить тупые ошибки до запуска приложения;
- делать код читаемым для команды без гадания по названиям переменных;
- не путать "данные точно такого вида" и "мы надеемся, что они такие".

---

## Что нужно знать

- что TypeScript проверяет код до запуска, но не подменяет runtime-проверки;
- в чём разница между динамической типизацией JavaScript и статической проверкой в TypeScript;
- `number`, `string`, `boolean`, `null`, `undefined`, `object` и массивы;
- `any` как аварийный выход, а не значение по умолчанию;
- `unknown` как безопасный тип на границе данных;
- `never` как сигнал невозможного состояния или исчерпанной ветки;
- `void` как "функция ничего полезного не возвращает";
- `literal types` для фиксированных значений вроде `'idle' | 'loading' | 'success'`;
- `union` и `intersection` как способы собирать контракт из нескольких частей;
- `tuple` и `readonly` для структур, где важны порядок и защита от случайной записи.

---

## Что нужно уметь

- типизировать параметры и результат простой функции;
- описывать состояние интерфейса через `union`, а не через строку "что угодно";
- выбирать между `any` и `unknown` осознанно, почти всегда в пользу `unknown`;
- замечать места, где `tuple` читается лучше обычного массива;
- ограничивать мутацию через `readonly`, если данные должны быть только для чтения.

---

## Базовое правило: сначала модель, потом аннотация

Плохой старт в TypeScript выглядит так:

```ts
let status: string = 'ok';
```

Здесь тип слишком широкий. Из него не видно, какие состояния вообще допустимы.

Нормальный старт:

```ts
type RequestStatus = 'idle' | 'loading' | 'success' | 'error';

let status: RequestStatus = 'idle';
```

Теперь контракт наблюдаемый. Если кто-то попытается присвоить `status = 'finished'`, компилятор скажет, что такой режим не предусмотрен.

---

## `any` против `unknown`: это не одно и то же

```ts
function parsePayloadUnsafe(payload: any) {
  return payload.user.name.toUpperCase();
}

function parsePayloadSafe(payload: unknown) {
  if (
    typeof payload === 'object' &&
    payload !== null &&
    'user' in payload
  ) {
    return payload;
  }

  throw new Error('Unexpected payload shape');
}
```

Что важно:

- `any` выключает проверку типов и позволяет делать почти всё;
- `unknown` заставляет сначала доказать, что данные действительно нужной формы.

На входе из `fetch`, `localStorage`, `postMessage` и внешнего API обычно нужен именно `unknown`, а не `any`.

---

## `tuple` и `readonly`: когда массив уже недостаточно выразителен

```ts
type HttpResult = readonly [statusCode: number, message: string];

const result: HttpResult = [200, 'ok'];

// result[0] = 500;
// Ошибка: readonly-кортеж нельзя мутировать
```

Такой тип полезен, когда у массива есть фиксированный смысл:

- координаты `[x, y]`;
- пара `[value, cleanup]`;
- результат `[code, message]`.

Если элементы массива однотипные и порядок не несёт отдельного смысла, обычный `string[]` или `number[]` обычно читается лучше.

---

## `never` и невозможные состояния

```ts
type Shape =
  | { kind: 'circle'; radius: number }
  | { kind: 'square'; side: number };

function area(shape: Shape): number {
  if (shape.kind === 'circle') {
    return Math.PI * shape.radius ** 2;
  }

  if (shape.kind === 'square') {
    return shape.side ** 2;
  }

  const unreachable: never = shape;
  return unreachable;
}
```

`never` здесь не нужен ради красоты. Он помогает поймать ситуацию, когда в `Shape` добавили новый вариант, а код обработки забыли обновить.

---

## Типичные ошибки

1. Ставят `any`, как только компилятор начал спорить.
2. Думают, что тип сам по себе проверяет данные в runtime.
3. Используют `tuple` для сущностей, где лучше работал бы объект с именованными полями.
4. Размазывают аннотации по всему коду, даже там, где вывод типов и так очевиден.
5. Путают `void` и `undefined`: первое про контракт функции, второе про конкретное значение.

---

## Anti-pattern

"Сначала всё затипизируем через `any`, потом ужесточим потом".

Проблема в том, что `потом` обычно не наступает. `any` распространяется по коду как масло по столу: одна дырка быстро портит десяток соседних типов.

---

## Как проверить понимание

1. Объясни своими словами разницу между `any` и `unknown`.
2. Опиши состояние загрузки данных через `union`, а не через `string`.
3. Реши, что лучше в каждом случае: объект, массив или `tuple`.
4. Посмотри на функцию с типом `any` на входе и предложи, как безопасно сузить контракт.

Мини-задача:

```ts
type ToastState = 'hidden' | 'visible';

function showToast(state: ToastState, message: string): readonly [ToastState, string] {
  return [state, message];
}
```

Что здесь хорошо:

- состояние ограничено `literal union`;
- результат читается предсказуемо;
- `readonly` не даёт случайно переписать возвращённую пару.
