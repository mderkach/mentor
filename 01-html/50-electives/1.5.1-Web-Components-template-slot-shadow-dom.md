# 1.5.1 Web Components: template, slot, shadow DOM

Это elective-урок, а не новая "обязательная база по HTML".
Он нужен, чтобы понимать платформу шире: где заканчивается обычный HTML authoring и начинается component-model с другими trade-offs по изоляции, API и поддержке.

---

## 🎯 Что ты освоишь

- что делают `template`, `slot`, `shadow DOM` и `custom elements`;
- в каких задачах Web Components реально оправданы;
- почему эта тема полезна для platform literacy, но не обязательна для core HTML-middle;
- как оценивать цену такого выбора, а не влюбляться в механику ради механики.

## Базовый пример

```html
<template id="card-template">
  <article class="card">
    <slot name="title"></slot>
    <slot></slot>
  </article>
</template>
```

---

## Что важно понять

- `template` хранит фрагмент разметки, пока ты его не клонируешь.
- `slot` задает точки вставки внешнего контента.
- `shadow DOM` изолирует внутреннюю структуру и часть стилевого контракта.
- `custom elements` и `shadow DOM` уже живут не в базовом HTML authoring, а в слое компонентной модели поверх платформы.

---

## Когда Web Components оправданы

- строишь дизайн-систему для нескольких продуктов или команд;
- делаешь виджеты, которые нужно встраивать в разные приложения;
- хочешь платформенный контракт, а не зависимость от конкретного UI-фреймворка;
- готов поддерживать API компонентов, события, стили и документацию как продукт.

---

## Когда лучше не усложнять

- это короткий MVP;
- команда живет внутри одного фреймворка и не испытывает реальной боли от его компонентной модели;
- нет ресурсов поддерживать кросс-проектный API компонентов;
- задача решается обычным HTML authoring и не требует platform-level abstraction.

---

## Anti-pattern

"Раз это часть платформы, значит надо учить всем и тащить в core HTML".

Нет. Это важная тема для кругозора и отдельных архитектурных сценариев, но не обязательный навык для большинства frontend-путей на middle-уровне.

---

## 🔧 Практика

1. Выдели один reusable UI-блок и опиши его API: что приходит снаружи, что рендерится внутри, какие есть события.
1. Сделай прототип на `template + slot`.
1. Сравни стоимость поддержки с эквивалентом во фреймворке проекта.
1. Сформулируй, почему этот путь уместен именно здесь, а не просто "потому что можно".
