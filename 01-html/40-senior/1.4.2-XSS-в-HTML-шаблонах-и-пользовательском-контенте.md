# 1.4.2 Безопасный HTML-контракт: user content, links, embeds, XSS-границы

## Уровень

Senior / Senior+

## Кратко о теме

Этот урок про безопасную работу с пользовательским HTML, ссылками и встраиваемыми ресурсами. На senior-уровне важно уметь задать понятные правила рендера и не пускать опасный HTML в продукт.

## Зачем это нужно на практике

- Снижает риск угона сессий, токенов и аккаунтов.
- Убирает класс критичных инцидентов из релизного цикла.
- Даёт команде единые правила рендера контента из внешних источников.
- Помогает договориться о правилах рендера внешнего и пользовательского контента.

## Что уже должно быть знакомо

- quality bar для HTML-ревью: [1.4.1 HTML Quality Bar для команды](./1.4.1-HTML-Quality-Bar-для-команды.md)
- формы, ссылки и safe defaults на уровне authoring: [1.2.3 Формы: атрибуты и валидация](../20-junior/1.2.3-Формы-атрибуты-и-валидация.md)
- полезно дополнительно пройти обзор по безопасности: [11. Безопасность](../../11-безопасность/README.md)

## Что нужно знать

- Типы XSS: reflected, stored, DOM-based.
- Контекст важнее всего: HTML text, attribute, URL, inline JS/CSS - это разные правила экранирования.
- Автоэкранирование фреймворков помогает, но не спасает при `innerHTML`/`dangerouslySetInnerHTML`/`v-html`.
- Санитайзер нужен только когда действительно нужен HTML от пользователя; в остальных случаях - `textContent`.
- `rel="noopener noreferrer"`, `sandbox`, `referrerpolicy` и sane defaults для внешнего контента - часть HTML-контракта, а не "потом разберутся безопасники".

## Что нужно уметь

- Делать threat model для мест, где рендерится внешний/пользовательский контент.
- Запрещать опасные API на уровне линтинга/ревью.
- Выделять один «безопасный шлюз» для HTML-рендера через санитайзер.
- Понимать, когда достаточно безопасных HTML-практик, а когда уже нужны CSP и Trusted Types.

## Плохой и хороший пример

### Плохо

```ts
// Пользовательский ввод напрямую вставляется как HTML
commentNode.innerHTML = commentFromApi;
```

### Хорошо (если HTML не нужен)

```ts
commentNode.textContent = commentFromApi;
```

### Хорошо (если HTML действительно нужен)

```ts
import DOMPurify from 'dompurify';

const safeHtml = DOMPurify.sanitize(commentFromApi, {
  ALLOWED_TAGS: ['b', 'i', 'a', 'code', 'p', 'ul', 'li'],
  ALLOWED_ATTR: ['href', 'title'],
});

commentNode.innerHTML = safeHtml;
```

Ключевая мысль: не «санитайзим везде», а задаем единый контракт рендера.

## Senior vs Senior+ в этой теме

- `Senior`: задает safe defaults, ревьюит user content, ссылки и embeds, не дает unsafe HTML пройти незамеченным.
- `Senior+`: проектирует единый безопасный путь рендера HTML и внедряет CSP/Trusted Types там, где это действительно нужно.

## Где чаще всего ошибаются

- Санитайзят на клиенте, но не валидируют/чистят данные на сервере.
- Разрешают слишком широкий набор тегов/атрибутов.
- Вставляют ссылки без проверки схемы (`javascript:`/`data:`).
- Считают, что «у нас React/Vue, значит XSS невозможен».

## Anti-pattern

«Нужно быстро показать rich-text, давай через `innerHTML` и потом починим».

Обычно не почините: такой код быстро размножается и превращается в системную дыру.

## 🔧 Практика

1. Найди в проекте все места с `innerHTML`/`outerHTML`/`insertAdjacentHTML`.
1. Для каждого места определи источник данных и контекст рендера.
1. Удали прямую вставку там, где достаточно plain text.
1. Для оставшихся кейсов внедри единый sanitizer-адаптер.

Признак освоения: команда знает единственный approved-способ рендера пользовательского HTML и отклоняет обходные пути на ревью.

## ✅ Чеклист

- [ ] Я отделяю plain text от случаев, где действительно нужен HTML
- [ ] В проекте есть один approved-шлюз для пользовательского HTML
- [ ] На ревью я проверяю не только код, но и источник данных с контекстом рендера
- [ ] Я понимаю, когда достаточно безопасных HTML-практик, а когда уже нужны отдельные защитные механизмы
