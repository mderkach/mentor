# 1.3.9 Progressive Enhancement, app shells и no-JS fallback

Progressive Enhancement - подход, где базовый сценарий работает на HTML, а JavaScript улучшает опыт, но не становится единственной точкой отказа.
На middle-уровне это уже связано не только с no-JS fallback, но и с тем, как ты проектируешь app shell и SSR-safe markup.
Если термины `app shell`, `SSR` и `hydration` пока плавают, см. короткий разбор в [1.3.6 HTML для SSR и hydration-safe markup](./1.3.6-Критический-рендеринг-стили-скрипты-lazy-loading.md#термины-по-человечески).

---

## 🎯 Что ты освоишь

- как проектировать базовый сценарий до JS;
- где нативные элементы реально помогают деградации;
- как отличать живой app shell от пустой коробки под гидрацию;
- как не превратить SPA-подход в "ничего не работает без скриптов".

## Пример подхода

```html
<form action="/search" method="get">
  <label for="q">Поиск</label>
  <input id="q" name="q" type="search" required />
  <button type="submit">Найти</button>
</form>

<noscript>
  <p>Для расширенных фильтров нужен JavaScript. Базовый поиск уже работает.</p>
</noscript>

<script type="module">
  // Улучшаем UX, но не ломаем базовый submit
  const form = document.querySelector('form');
  form?.addEventListener('submit', (event) => {
    // enhancement: быстрый рендер результатов без полной перезагрузки
    // fallback: если код упадет, обычный submit все равно отработает
  });
</script>
```

---

## Что важно понять

- Базовый сценарий должен быть работоспособен без JS.
- Нативные элементы (`button`, `form`, `details`, `dialog`) предпочтительнее кастомных div-виджетов.
- `noscript` - не заглушка для галочки, а честный fallback-сигнал.
- App shell должен оставаться понятным и полезным до client enhancement.

---

## Ошибки

- Делают кнопки и ссылки через `div` и завязывают все на JS-события.
- Перехватывают `submit`, но забывают fallback при ошибке в скрипте.
- Прячут критичную информацию целиком за JS-рендером.
- Делают app shell, который без JS не полезен, а просто мертв.

---

## Anti-pattern

"SPA-first значит HTML можно не проектировать".

Даже в SPA HTML остается контрактом доступности, SEO и базовой устойчивости. Игнорирование этого контракта ведет к хрупкому UI.

---

## 🔧 Практика

1. Отключи JS в браузере.
1. Пройди 1-2 критичных сценария: поиск, логин, форма или app shell.
1. Зафиксируй, что работает нативно, а что ломается.
1. Верни JS и добавь enhancement без поломки базового пути.

---

## ✅ Чеклист

- [ ] Критичный сценарий жив без JS
- [ ] Я не делаю интерактив через `div`, если платформа уже умеет лучше
- [ ] App shell не мертв без hydration
- [ ] Enhancement улучшает сценарий, а не подменяет базовый путь
