# 1.1.3 Семантика, скринридер и встроенные HTML API

Название «семантические ландмарки» звучит абстрактно. По факту тема простая:

если разметка описывает смысл страницы, то интерфейс корректно читается не только глазами, но и скринридером, и поисковыми системами.

---

## 🎯 Что ты освоишь

- как скринридер реально проходит страницу;
- какие теги дают готовое поведение «из коробки» (без кастомного JS);
- почему `button`, `dialog`, `details`, `input` почти всегда лучше, чем `div + onclick`;
- какие семантические теги нужны в каркасе каждой страницы.

---

## Как скринридер «видит» страницу

Скринридер читает не пиксели, а **дерево доступности** (accessibility tree), которое строится из HTML-структуры.

Что это значит на практике:

1. Скринридер получает роли и названия элементов из семантики (`nav`, `main`, `button`, `label` и т.д.).
2. Пользователь двигается по странице не только стрелками, но и «быстрой навигацией»: по заголовкам, ссылкам, формам, landmark-регионам.
3. Если у тебя везде `div`, дерево доступности бедное, и навигация становится болезненной.

Простой вывод:

- хорошая семантика = предсказуемая навигация;
- плохая семантика = «хаос» для ассистивных технологий.

---

## Семантический каркас: минимальный стандарт

```html
<body>
  <header>...</header>
  <nav>...</nav>

  <main>
    <article>
      <h1>Заголовок страницы</h1>
      <section>
        <h2>Раздел</h2>
      </section>
    </article>

    <aside>Дополнительный контент</aside>
  </main>

  <footer>...</footer>
</body>
```

Что даёт этот каркас:

- landmarks для быстрой навигации;
- понятную иерархию контента;
- удобную поддержку и ревью в команде.

---

## Встроенные API компонентов (раньше это часто писали на JS)

### 1) `button` вместо `div onclick`

Что получаешь сразу:

- фокус с клавиатуры;
- активацию Enter/Space;
- состояние `disabled`;
- корректную роль для скринридера.

### 2) `details + summary` вместо кастомного аккордеона

Что получаешь:

- нативное раскрытие/сворачивание;
- базовую доступность и понятную семантику;
- меньше JS-кода и edge-case багов.

### 3) `dialog` вместо «модалки на div»

Что получаешь:

- нативный диалоговый контекст;
- более предсказуемый сценарий с фокусом;
- меньше ручного кода для modal-логики.

### 4) `input` с корректным `type`

Примеры:

- `type="email"` — встроенная проверка формата;
- `type="number"` — числовой ввод;
- `type="tel"` — удобная клавиатура на мобильных.

### 5) Нативные элементы состояния

- `progress` — прогресс операции;
- `meter` — значение в диапазоне (оценка, нагрузка и т.д.).

Главная идея:

- сначала проверяем, есть ли нативный HTML-элемент/атрибут;
- только потом пишем кастом на JS.

---

## Антипаттерны, которые надо выжигать

- Кликабельный `div` вместо `button`.
- «Модалка» без `dialog` и без контроля фокуса.
- Меню/навигация без `nav`.
- `section` без заголовка.
- Заголовки в неправильной иерархии (`h2 -> h4`).

---

## 🔧 Практика

1. Возьми одну страницу проекта и выпиши все интерактивные элементы (`div onclick`, кастомные аккордеоны, модалки).
2. Замени минимум 3 кастомных паттерна на нативные HTML-элементы.
3. Проверь страницу в трех режимах: только с клавиатуры, переход по заголовкам, переход по landmarks.

---

## ✅ Чеклист

- [ ] Основные регионы страницы размечены семантически (`header/nav/main/footer`)
- [ ] Кнопки и ссылки не подменены `div`
- [ ] Для раскрывающихся блоков и диалогов используются нативные элементы, где это возможно
- [ ] Иерархия заголовков логична и последовательна
- [ ] Страница понятна скринридеру на уровне структуры
